# TOML 是什么

一门为**人类手写配置**而生的格式：

<v-clicks>

- **Obvious**：语法极简、语义显而易见，不留二义空间
- **无歧义映射哈希表**：任何合法 TOML 解析结果**唯一确定**
- **缩进无语义**：层级由 `[表头]`/点分键显式表达，不像 YAML 靠缩进
- 原生支持 `#` 注释与日期时间——比 JSON 更适合配置

</v-clicks>

<div v-click class="mt-4 text-sm">

> 它不是编程语言（无函数/变量/循环），也不是机器数据交换协议（那是 JSON 的主场）。

</div>

<!--
TOML 专注做一件事：把配置写得清晰、可读、可批注，同时保证机器能唯一确定地解析。

两个核心设计目标：一是 Obvious，语法刻意收窄、极简，不给一份文档两种解读留余地；二是无歧义映射到哈希表，也就是字典或对象结构，任何合法 TOML 都能被明确唯一地解析成键值嵌套结构。

一个重要边界：缩进在 TOML 里没有语义，会被忽略，层级完全由方括号表头和点分键显式表达，这是它相对 YAML 的关键区别。它也不是编程语言，没有函数变量循环，也不是数据交换协议。
-->

# 基本语法：键值对 + 注释

```toml
# 这是一整行注释
title = "TOML 示例"          # 行尾注释

name = "Tom"                 # 键 = 值，必须在同一行
port = 8080
enabled = true

[server]                    # 表头：之后的键值对归属 server
host = "localhost"
port = 5432
```

<v-clicks>

- **键值对** `键 = 值` 是最小积木，`=` 两侧空白忽略
- 注释用 `#`（字符串内的 `#` 除外）；没有 `//` 或块注释
- **大小写敏感**：`Name` 与 `name` 是不同的键

</v-clicks>

<!--
一份 TOML 由键值对、注释、表三类要素构成。

键值对是最小积木，形式是键等号值，键、等号、值必须在同一行，等号两侧空白被忽略。注释用井号，从它开始到行尾都被忽略，但字符串内部的井号不算注释。TOML 没有双斜杠或块注释。

大小写敏感这点要记住，Name 和 name 是两个不同的键，可以共存。另外注意这里 server 表下面的 port，和顶层的 port 不冲突，因为它们属于不同的表。
-->

# 键的三种形态

```toml
bare_key = "value"          # 裸键：A-Za-z0-9_-
1234 = "value"              # 纯数字也是合法裸键

"127.0.0.1" = "value"       # 引号键：可含任意字符
"character encoding" = true

physical.color = "orange"   # 点分键：拆成多层表
physical.shape = "round"
```

<v-clicks>

- **裸键**：只允许 ASCII 字母/数字/`_`/`-`
- **引号键**：含点/空格/中文时必须用引号（否则被拆层或非法）
- **点分键** `a.b.c`：为每层**自动建中间表**，等价 `{ a = { b = ... } }`
- ⚠️ 键**不可重复定义**：`x` 与 `"x"` 视为同名，定义两次即报错

</v-clicks>

<!--
键有三种形态。裸键不加引号，字符集只限 ASCII 字母、数字、下划线、短横线，纯数字比如 1234 也是合法裸键。

引号键用双引号或单引号包裹，从而能包含任意字符，比如带点的 IP 地址、带空格的名字、中文。如果直接裸写 127.0.0.1，会被当成点分键拆成 127、0、0、1 四层，语义完全不同。

点分键用点号表达层级，会为最后一段之前的每一层自动创建中间表，也叫超级表，等价于嵌套的表结构。点号两侧空白会被忽略。

一个高频坑：键不可重复定义，而且裸键 x 和引号键 x 被视为同一个键，同时定义两次就报错，引号只是书写手段，不会变成另一个键。
-->

# 字符串①：基本 + 多行基本

```toml
basic = "支持转义：\n \t \" \\ \uXXXX"

multi = """
第一行紧跟定界符——这个首换行会被删除
The quick brown \
   fox jumps over \
   the lazy dog."""
# 行尾 \ = 续行：吃掉其后空白与换行，拼成一整句
```

<v-clicks>

- **基本字符串**（双引号）：支持反斜杠转义；`\d` 等非法转义会报错
- **多行基本**（三双引号）：紧跟开头定界符的**首个换行被删除**
- **行尾 `\`**：删除其后所有空白（含换行/前导空白）直到非空白字符

</v-clicks>

<!--
TOML 有四种字符串，先看双引号系。

基本字符串用双引号，支持反斜杠转义，合法转义只有退格、制表、换行、换页、回车、双引号、反斜杠，以及 uXXXX 和大写 U 的 Unicode 码点。其它组合比如反斜杠 d 是非法转义，多数解析器直接报错。

多行基本字符串用三个双引号，可以跨行。两条关键规则：第一，紧跟开头定界符的首个换行会被删除，所以可以整齐地另起一行写正文而不多出空行；第二，行尾反斜杠是行连接符，会删除其后的所有空白，包括换行和下一行的前导空白，直到第一个非空白字符，用来把长句拆成多行书写但拼接后不带换行。
-->

# 字符串②：字面 + 多行字面

```toml
winpath = 'C:\Users\nodejs\templates'   # 不转义，所见即所得
regex   = '<\i\c*\s*>'

regex2  = '''I [dw]on't need \d{2} apples'''
lines   = '''
不转义、可跨行；内部最多连续 2 个单引号
'''
```

<v-clicks>

- **字面字符串**（单引号）：**不转义**，最适合 Windows 路径 / 正则
- **多行字面**（三单引号）：不转义 + 跨行；首换行删除
- ⚠️ 含 `\` 的正则/路径优先用**单引号**——双引号里 `\d` 会报错

</v-clicks>

<!--
再看单引号系，也就是字面字符串。

字面字符串用单引号，不做任何转义，所见即所得，反斜杠和双引号都是普通字符。这让它成为写 Windows 路径、正则表达式的理想选择，因为这些内容里有大量反斜杠，用双引号基本字符串写反而容易因非法转义报错。字面字符串不能跨行。

多行字面字符串用三个单引号，既不转义又能跨行，首个换行同样会被删除，内部可以直接出现 1 到 2 个连续单引号，但不能连续 3 个，否则被当结束定界符。

选串口诀：含大量反斜杠的正则路径用单引号字面，需要转义用双引号基本，带换行的长文本用三引号多行版本。
-->

# 整数：进制、下划线、禁前导零

```toml
int1 = +99
int2 = -17
big  = 1_000_000          # 下划线分隔，解析时忽略

hex  = 0xDEADBEEF         # 十六进制
oct  = 0o755              # 八进制（Unix 权限）
bin  = 0b11010110         # 二进制
```

<v-clicks>

- 64 位有符号；下划线**两侧须紧邻数字**（`_1` `1_` `1__0` 非法）
- 进制前缀 `0x`/`0o`/`0b`，这些形式**不允许** `+`/`-`
- ⚠️ **禁前导零**：`0123` 非法（消除「前导零=八进制」歧义）
- ⚠️ 超 `2^63−1` 会溢出，超大 ID 用字符串承载

</v-clicks>

<!--
整数是 64 位有符号数，可带正负号。

下划线可作可读性分隔，解析时忽略，但每个下划线两侧都必须紧邻数字，所以下划线在开头、结尾或连续两个都非法。

支持三种非十进制前缀：十六进制 0x、八进制 0o、二进制 0b，八进制常用于 Unix 权限位。这些进制形式不允许带正负号，前缀和数字之间也不能加下划线。

两个坑：一是禁止前导零，0123 非法，这是为了消除前导零是否代表八进制的歧义，要八进制必须显式写 0o；二是整数只保证 64 位有符号范围，超过 2 的 63 次方减 1 会溢出，雪花 ID 这类超大数字要用字符串承载。
-->

# 浮点与布尔

```toml
flt1 = 3.1415
flt2 = 6.022e23           # 指数
flt3 = 224_617.445_991    # 下划线
sf1  = inf                # 正无穷（必须小写）
sf2  = -inf
sf3  = nan                # 非数
ok   = true               # 布尔只有小写 true/false
```

<v-clicks>

- 浮点遵循 **IEEE 754 双精度**；`inf`/`nan` **必须全小写**
- ⚠️ 小数点**两侧都要有数字**：`.5`、`5.`、`3.e+20` 全非法
- 布尔只有 `true`/`false`（小写）；`yes`/`on`/`1` **不是**布尔

</v-clicks>

<!--
浮点遵循 IEEE 754 双精度，由整数部分加可选小数部分加可选指数构成，支持指数写法和下划线分隔。

特殊值有正无穷 inf、负无穷 -inf、非数 nan，注意必须全小写，写成 Inf、NaN、Infinity 都非法，这和布尔一样，TOML 刻意只保留小写字面量。

一个高频坑：小数点两侧都必须至少有一位数字，所以点五、五点、3 点 e20 全都非法，必须写成 0.5、5.0、3.0e+20。

布尔只有小写的 true 和 false，yes、on、1 都不是布尔，这点和 YAML 不同，YAML 会把 yes、on 当布尔，是著名的挪威问题的一部分。
-->

# 日期时间：四种原生类型

```toml
odt = 1979-05-27T07:32:00Z          # 带偏移：绝对时刻
odt2 = 1979-05-27 07:32:00-07:00    # T 可用空格替代
ldt = 1979-05-27T07:32:00           # 本地日期时间：无时区
ld  = 1979-05-27                     # 本地日期
lt  = 07:32:00                       # 本地时间
```

<v-clicks>

- 遵循 **RFC 3339**，都是**原生类型**、无需引号（相对 JSON 的核心优势）
- **带偏移**（`Z`/`±hh:mm`）= 绝对时刻；**本地**无时区、非唯一时刻
- 小数秒至少支持毫秒，超精度**截断**（非四舍五入）

</v-clicks>

<!--
TOML 内置四种日期时间类型，都遵循 RFC 3339，而且都是原生类型无需引号，这是它相对 JSON 的核心优势，JSON 只能拿字符串充当日期。

第一种带偏移日期时间，末尾带 Z 或正负时分偏移，锚定一个全球绝对时刻。注意大写 T 可以用空格替代。第二种本地日期时间，没有时区，只是墙上时钟读数，不对应唯一绝对时刻。第三种只有日期，第四种只有时间。

带偏移和本地的区别很重要，别混用。另外小数秒规范要求至少支持到毫秒，超出实现可表示精度时多余小数位应当截断而不是四舍五入。
-->

# 数组

```toml
integers = [1, 2, 3]
colors   = ["red", "yellow", "green"]
mixed    = [1, "two", 3.0, true]     # 1.0.0 允许混合类型

numbers = [
  1,
  2,   # 中间可写注释
  3,   # 尾随逗号允许
]
```

<v-clicks>

- 方括号 `[ ]`、逗号分隔、有序；**1.0.0 允许异构（混合类型）**
- 可**跨行**、可插注释、**允许尾随逗号**（利于 diff）
- ⚠️ 尾随逗号：**数组允许，内联表禁止**

</v-clicks>

<!--
数组用方括号包裹、逗号分隔，元素有序。

一个版本相关的点：1.0.0 允许数组混合不同类型的元素，也就是异构数组，早期草案曾要求同构。规范甚至给出字符串加内联表混合的例子。当然从可维护性角度，很多团队仍会自我约束为同构。

数组可以跨行书写，中间插注释，并允许尾随逗号，这对版本控制的 diff 很友好，加一行只改一行。

要特别记住的对比：尾随逗号数组允许，但内联表禁止，这是两者一个高频混淆点。
-->

# 表 [table] 与子表

```toml
[server]
host = "localhost"
port = 5432

[fruit.physical]          # 子表：点号表达嵌套
color = "red"

# 点分键与表头语义相通，下面等价于 [fruit.apple] color="red"
fruit.apple.color = "red"
```

<v-clicks>

- `[表头]` 声明一张表，之后键值对都归属它；首个表头前是**根表**
- `[a.b]` 是 `[a]` 的子表；点分键 `a.b.c = 1` 与表头**语义相通**
- 乱序定义表**合法但不鼓励**（不影响结构，只是可读性差）

</v-clicks>

<!--
方括号里写一个名字就是表头，server 之后到下一个表头或文件末尾之前的所有键值对都归属 server 这张表。在首个表头之前直接写的键值对属于根表。

用点号写出嵌套层级，fruit.physical 就是 fruit 的子表 physical。带引号的表名段可以包含点等特殊字符而不被拆层。

点分键和表头语义相通，fruit.apple.color 等号 red 这条点分键，和写 fruit.apple 表头再写 color 等号 red，结构完全一样。

顺序方面，乱序定义表，比如先 fruit.apple 再 animal 又 fruit.orange，是合法但官方不鼓励的，它不影响最终结构，只是可读性差，要和重复定义同一张表这种非法情况区分开。
-->

---

# 点分键 vs 表头：经典冲突

```toml
fruit.apple.color = "red"
fruit.apple.taste.sweet = true

# [fruit.apple]          # ❌ 已被点分键定义，重定义报错
# [fruit.apple.taste]    # ❌ 同理

[fruit.apple.texture]    # ✅ 追加一张此前不存在的子表
smooth = true
```

<v-clicks>

- 点分键**已定义**的表，不能再用 `[table]` 表头重定义 → 报错
- 但可以**追加一张全新子表**（`[fruit.apple.texture]`）→ 合法
- 别把「不能重定义」误读成「之后完全不能出现相关表头」

</v-clicks>

<!--
这是 TOML 最经典的坑之一，务必分清边界。

用点分键定义了 fruit.apple.color 和 fruit.apple.taste.sweet 之后，fruit.apple 和 fruit.apple.taste 这两张表都已经被隐式定义了。此时再用方括号 fruit.apple 或 fruit.apple.taste 表头去定义它们，等于重复定义已存在的表，报错。

但是，可以用 fruit.apple.texture 追加一张此前不存在的全新子表，这是合法的。

关键区别：不能重定义已存在的表，和可以追加一张全新子表，是两回事。别把前者误读成点分键之后完全不能再出现相关表头。
-->

---

# 内联表 { }

```toml
name  = { first = "Tom", last = "Preston-Werner" }
point = { x = 1, y = 2 }
animal = { type.name = "pug" }     # 内部可用点分键建子表

# 依赖配置的典型用法（Cargo.toml）
serde = { version = "1.0", features = ["derive"] }
```

<v-clicks>

- 花括号 `{ }` 把一张表写在**一行**内，语义上就是一张表
- **完全自包含**：定义后**不能**从外部追加键、**禁尾随逗号**、不能跨行
- 内部点分键（`type.name`）合法；外部再 `animal.type.x=1` 追加则非法

</v-clicks>

<!--
内联表用花括号把一张表写在一行内，适合简短的嵌套对象，比如坐标点，或者 Cargo.toml 里带 version 和 features 的依赖配置，这是内联表最典型的用途。

它等价于对应的方括号表写法，但有一组严格限制：完全自包含，一旦定义完就不能再从花括号外部往里追加键或子表；不能跨行；禁止尾随逗号，花括号内最后一个键值对后多一个逗号就非法。

一个容易混的点：花括号内部用点分键建子表，比如 type.name 等号 pug，是允许的；非法的是定义完之后在花括号外部再写 animal.type.edible 去追加。一内一外，区别关键。
-->

---

# 表数组 [[products]]

```toml
[[products]]
name = "Hammer"
sku = 738594937

[[products]]                 # 每出现一次追加一个表元素
name = "Nail"
color = "gray"

[products.meta]              # 可挂子表 / 子表数组
weight = 20
```

<v-clicks>

- 双方括号 `[[name]]` = **表数组**，每次**追加**一个表元素
- 用于表达「对象列表」：一组商品、一组依赖、一组重定向
- 元素下可挂子表 `[products.meta]` 与子表数组 `[[products.tags]]`

</v-clicks>

<!--
双方括号声明表数组，每出现一次 products，就往名为 products 的数组里追加一个新的表元素。这是表达对象列表的标准方式，比如一组商品、一组依赖、Netlify 里的一组重定向规则。

它和单方括号表不同：单方括号是定义一张表，双方括号是往表的数组里追加元素；也和内联的静态数组不同。

表数组元素下面还能挂子表和子表数组，用带前缀的普通表头 products.meta 挂子表，用双方括号 products.tags 挂子表数组，前提是父级必须先存在。
-->

# 经典陷阱清单

```toml
[fruit]
apple = "red"

[fruit]              # ❌ 重复定义表
[fruit.apple]        # ❌ apple 已是字符串值，类型冲突

fruits = []
[[fruits]]           # ❌ 不能用 [[ ]] 追加到静态数组
```

<v-clicks>

- **重复定义表** → 报错（TOML 无覆盖语义）
- **键类型冲突**：键已是值，不能再当表
- **静态数组 vs 表数组**：`x=[]` 后 `[[x]]` 报错；同名普通表与表数组互斥

</v-clicks>

<!--
把几个非法场景集中过一遍。

第一，重复定义表，同一张 fruit 表用方括号定义两次，报错。TOML 没有后者覆盖的语义，重复即报错。

第二，键类型冲突，apple 已经被定义成字符串值 red，之后又用 fruit.apple 表头想把它当表，同一个键既是字符串又是表，冲突报错。

第三，静态数组不能被表数组追加，fruits 用普通数组语法定义了，即使是空数组，也不能再用双方括号 fruits 往里追加。反过来，同名的普通表和表数组也互斥，一律报错。
-->

# TOML vs YAML

| 维度 | TOML | YAML |
| --- | --- | --- |
| 层级表达 | `[表头]`/点分键（显式） | 缩进（空白敏感） |
| 缩进语义 | 无（忽略） | 有（错一格就变） |
| 隐式类型 | 无 | 有（挪威问题） |
| 锚点/引用 | ❌ | ✅ `&`/`*` |
| 深层嵌套 | 易冗长 | 紧凑 |

<div v-click class="mt-3 text-sm">

> 深层嵌套 + 锚点复用（k8s / CI）→ YAML；层级不深的应用配置 → TOML。TOML 用「稍冗长 + 显式 + 无隐式」换「强确定性」。

</div>

<!--
TOML 和 YAML 都面向手写配置，常被拿来比。

层级表达上，TOML 用方括号表头和点分键显式表达，YAML 靠缩进；缩进语义上，TOML 完全忽略缩进，YAML 缩进敏感，错一格结构就变。

隐式类型上，TOML 没有隐式转换，YAML 有很多坑，比如 no 被当布尔 false 的挪威问题、未加引号的版本号被当浮点。锚点引用上，YAML 有锚点和别名可以复用重复内容，TOML 没有。深层嵌套上，YAML 更紧凑，TOML 写一连串方括号表头会显得冗长。

选型：深层嵌套加锚点复用，比如 Kubernetes、CI 流水线，用 YAML 更合适；层级不深的应用配置用 TOML 更省心。TOML 的哲学是用稍微冗长、显式层级、无隐式转换，换取强确定性。
-->

# TOML vs JSON

| 维度 | TOML | JSON |
| --- | --- | --- |
| 定位 | 手写配置 | 机器数据交换 |
| 注释 | ✅ `#` | ❌ 标准不支持 |
| 原生日期 | ✅ 四型 | ❌（只能用字符串） |
| 键 | 可裸写 | 必双引号 |
| 生态 | Rust/Python 配置 | REST API / `package.json` |

<div v-click class="mt-3 text-sm">

> 人手维护为主 → TOML（注释 + 日期 + 可读性）；程序生成/消费、对接 Web API → JSON（更通用、生态更直接）。

</div>

<!--
TOML 和 JSON 的对比更多是分工不同。

TOML 定位手写配置，有 # 注释、四种原生日期时间、键可裸写，可读性好；JSON 定位机器数据交换，标准不支持注释、没有原生日期只能用字符串充当、所有键都要加双引号。

生态上，TOML 是 Rust、Python 工具链配置的主力，比如 Cargo.toml、pyproject.toml；JSON 是 REST API、package.json 的事实标准。

选型很清晰：人手维护为主，需要注释和日期，选 TOML；程序生成消费为主、要和 Web API 无缝对接，选 JSON，它更通用、生态更直接。
-->

# 生态落地①：Rust 与 Python

```toml
# Cargo.toml —— Rust 包清单
[package]
name = "my-crate"
version = "0.1.0"

[dependencies]
serde = { version = "1.0", features = ["derive"] }
```

```toml
# pyproject.toml —— Python 打包
[build-system]              # PEP 518
requires = ["hatchling"]

[project]                   # PEP 621：name/version/dependencies
name = "my-package"
```

<div v-click class="mt-2 text-sm">

> `[build-system]` 由 PEP 518 引入，`[project]` 由 PEP 621；Ruff/Black/mypy 配置放 `[tool.*]`。

</div>

<!--

# 易错点 Top 8

<v-clicks>

- 以为**缩进有意义** → TOML 空白无语义，层级用 `[表头]`/点分键
- **内联表加尾随逗号** → 数组允许、内联表禁止
- **重复定义表/键** → 报错，TOML 无覆盖语义
- **点分键后又 `[a.b]`** → 重定义报错，只能追加全新子表
- **静态数组后 `[[x]]`** → 报错，一开始就用表数组
- **`.5` / `5.`** → 浮点小数点两侧都要有数字
- **`Inf` / `NaN` / `yes`** → 特殊值/布尔必须小写、无 `yes`
- **超大整数** → 超 `2^63−1` 溢出，用字符串承载

</v-clicks>

<!--
过一遍最高频的踩坑点。

第一，从 YAML 来的人常以为缩进有意义，TOML 里缩进被忽略，层级要用方括号表头或点分键。第二，内联表加尾随逗号，数组允许但内联表禁止。第三，重复定义表或键，TOML 没有覆盖语义，直接报错。第四，点分键已定义的表又用方括号表头重定义，报错，只能追加全新子表。

第五，静态数组后用双方括号追加，报错，一开始就该用表数组。第六，浮点点五、五点这种写法非法，小数点两侧都要有数字。第七，特殊浮点 Inf、NaN 和布尔必须小写，而且没有 yes、on 这种布尔。第八，超过 2 的 63 次方减 1 的超大整数会溢出，要用字符串承载。
-->

# 总结

TOML = **面向人类、无歧义映射哈希表的极简配置格式**

- 基本单位键值对；层级靠 `[表头]`/点分键**显式**表达，缩进无语义
- 类型丰富：四种字符串、进制整数、`inf`/`nan` 浮点、四种日期时间
- 复合结构：数组 `[ ]` / 内联表 `{ }` / 表数组 `[[ ]]`
- 坑集中在表与点分键交互：重定义、类型冲突、静态数组不可追加
- 1.0.0（2021-01）事实标准，Cargo/pyproject/Wrangler 广泛落地
