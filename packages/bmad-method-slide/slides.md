传统 AI 替你思考 → 平庸结果

# 规模自适应：从 bug 到企业系统

同一套框架，自动调规划深度

<div class="grid grid-cols-2 gap-6 mt-6">
<div v-click>

**小：bug 修复**

- 轻流程
- 不必完整 PRD + 架构
- 不小题大做

</div>
<div v-click>

**大：企业级系统**

- 重规划
- 深度分析 + 完整 PRD + 正式架构
- 不大题小做

</div>
</div>

# BMM 四阶段 + 域专家

敏捷 SDLC 的角色，落成 agent

| 阶段 | 域专家 | 产出 |
| --- | --- | --- |
| **1 分析** | `agent-analyst` | 需求、研究、产品简报 |
| **2 规划** | `agent-pm` + `agent-ux-designer` | PRD、UX 规格 |
| **3 方案** | `agent-architect` | 架构、技术方案 |
| **4 实现** | `agent-dev` | 敏捷实现 |

<div v-click class="mt-3 text-center text-sm opacity-80">

12+ 域专家逐阶段推进，每阶段有专家主导。`bmad-help` 随时问下一步。

</div>

<style>
h1 { background: linear-gradient(45deg, #b45309 10%, #b91c1c 60%); -webkit-background-clip: text; -webkit-text-fill-color: transparent; }

# Party Mode：多专家圆桌

把多个 persona 拉进一个会话

```text
bmad-party-mode
```

<v-clicks>

- PM、架构师、UX 在**同一场**从各自视角碰撞
- 一个决策同时被产品、设计、工程三视角审视
- 解决「单角色单线程看不全」

</v-clicks>

<div v-click class="mt-4 text-center text-sm opacity-80">

对比 gstack 的「逐个角色命令」——BMAD 是「多角色同场」。

</div>

# 对抗审查 + 前期工具箱

从想法到规格，主动找漏洞

| 技能 | 作用 |
| --- | --- |
| `review-adversarial-general` | 通用对抗审查——主动找漏洞 |
| `review-edge-case-hunter` | 边界情况猎手 |
| `review-verification-gap` | 验证缺口——找「声称做了没验证」 |
| `advanced-elicitation` | 高级需求挖掘 |
| `brainstorming` / `forge-idea` / `spec` | 头脑风暴 / 锻造想法 / 写规格 |
