# 为什么用 Markdown？

轻量、纯文本、跨平台，写作与协作的通用语：

<v-clicks>

- **纯文本可读**：源码不渲染也一目了然，对 Git diff / `grep` / 版本控制天然友好
- **生态无处不在**：README、Issue/PR、聊天、静态站点、笔记应用几乎都以它为默认
- **有权威规范**：CommonMark 消歧 + GFM 补齐表格/任务列表等协作刚需
- **可内嵌 HTML 兜底**：表达不了的排版直接写 HTML 透传

</v-clicks>

<div v-click class="mt-6">

但要认清边界：

- 方言碎片化，扩展语法跨平台支持不一
- 复杂排版（多列、合并单元格）力不从心

</div>

<!--
为什么用 Markdown？它轻量、纯文本、跨平台，是写作与协作的通用语。

优势：纯文本可读，源码不渲染也一目了然，对 Git diff、grep、版本控制天然友好，是「文档即代码」的理想载体；生态无处不在，GitHub 的 README、Issue、PR，聊天工具，静态站点如 VitePress、Hugo、Jekyll、Astro，笔记应用如 Obsidian，几乎都以 Markdown 为默认书写格式；有了权威规范，CommonMark 消歧、GFM 补齐表格任务列表等协作刚需；还能平滑内嵌 HTML 兜底。

但要认清边界：Markdown 是一族方言而非单一实现，脚注、上下标这类扩展语法跨平台支持不一，可移植性差；复杂排版比如多列布局、合并单元格力不从心，往往要退回裸 HTML。
-->

# 由来与 CommonMark 规范化

Gruber 2004 年发布 Markdown 与 Perl 参考实现 `Markdown.pl`，但语法描述留有大量歧义：

<v-clicks>

- 子列表缩进几格？块引用前要不要空行？列表项何时被 `<p>` 包裹？
- 早期实现只能参考「相当有 bug」的 `Markdown.pl` → 同一文档 GitHub 与 Pandoc 渲染不一致
- **CommonMark**（2014 起）用三件套消歧：精确规范 + 参考实现 `cmark` + 测试样例
- 当前版本 **`0.31.2`（2024-01-28）**；**GFM 是它的严格超集**

</v-clicks>

<div v-click class="mt-4 text-sm">

> 「In the absence of a spec, early implementers consulted `Markdown.pl`… But `Markdown.pl` was quite buggy.」—— CommonMark 规范导言

</div>

<!--
Markdown 诞生于 2004 年，作者 John Gruber，Aaron Swartz 参与早期设计，配了一个 Perl 参考实现 Markdown.pl。

问题在于 Gruber 的语法说明是散文式的，留下大量未定义的边界情形：子列表要缩进几格、块引用前要不要空行、列表项何时被 p 标签包裹、行内标记优先级如何。在没有规范的情况下，早期实现只能去参考本身相当有 bug 的 Markdown.pl，于是同一份文档在 GitHub wiki 上和用 Pandoc 转换时渲染结果可能不同。

CommonMark 就是为解决歧义而生，2014 年发起，一度叫 Standard Markdown。它提供三件套：极其精确的规范、参考实现比如 C 语言的 cmark、以及一整套输入 Markdown 到期望 HTML 的测试样例。当前锁定版本 0.31.2，2024 年 1 月 28 日。GFM 则声明自己是 CommonMark 的严格超集。
-->

# 两层心智：块级 vs 行内

Markdown 解析分**两层**，理解它能解释很多行为：

<v-clicks>

- **① 块级元素**先切分文档结构：段落、标题、块引用、列表、代码块、分隔线、HTML 块
- **② 行内元素**再在块内解析：强调、链接、图片、代码跨度、转义、自动链接
- CommonMark 进一步分**叶子块**（段落/标题/代码块…）与**容器块**（仅**块引用/列表/列表项**三种，可含其它块）

</v-clicks>

<div v-click class="mt-4 text-sm">

> 解释力：代码块（叶子块）里的 `*` 不变斜体；块引用（容器块）里能塞列表和标题；标题（叶子块）里仍能有 `**加粗**`。

</div>

<!--
理解 Markdown 解析的关键，是它分两层处理文档。

第一层块级元素，先把文档按行切分成一个个块：段落、标题、块引用、列表、代码块、分隔线、HTML 块。CommonMark 进一步把块分成叶子块，不含其它块，比如段落、标题、代码块、分隔线；和容器块，可含其它块，只有块引用、列表、列表项三种。

第二层行内元素，在每个叶子块的文本内容里再解析强调、链接、图片、代码跨度、转义、自动链接。

这个分层能解释很多行为：为什么代码块里的星号不会变斜体，因为块内不再解析行内标记；为什么块引用里能塞列表和标题，因为它是容器块；为什么标题里可以有加粗，因为标题是叶子块，内部仍解析行内元素。
-->

# 标题：ATX vs Setext

```md
# 一级标题        ← ATX：# 数量表示 1-6 级
## 二级标题       ← 井号后必须有空格！

一级标题
========          ← Setext：= 一级、- 二级，仅两级

二级标题
--------
```

<div v-click class="mt-3 text-sm">

> ⚠️ 两个坑：`#标题`（井号后无空格）在 CommonMark 下**不成标题**、变普通段落；一段文字紧接一行 `-` 会被当 **Setext 二级标题**而非分隔线（分隔线前后要留空行）。

</div>

<!--
Markdown 有两套标题语法。ATX 标题最常用，用行首井号数量表示级别，一到六级。CommonMark 有条硬规则：开头的井号序列后必须跟空格、制表符或行尾，所以井号后紧跟文字没有空格不构成标题，会被当普通段落输出。最佳实践是井号后永远加一个空格。ATX 还允许可选的闭合井号，纯装饰。

Setext 标题用下一行下划线的方式，等号是一级、连字符是二级，下划线长度任意。它的局限很明显，只能表达一二级，三级以上只能用 ATX。还有个交叉陷阱：一段普通文字紧接一行连字符，会被优先解析成 Setext 二级标题而不是分隔线，所以写分隔线时务必前后留空行。
-->

# 段落与换行（隐形坑）

```md
第一行末尾有两个空格 → 硬换行（<br>）  
第二行，与上一行同段但换行

这是新的一段（上面有一个空行）
```

<v-clicks>

- **软换行**：段内单个普通换行 → 多被当**一个空格**，不产生 `<br>`
- **硬换行**：行尾**两个及以上空格**，或行尾一个 `\` → 渲染为 `<br>`
- **分段**：需要一个**空行**

</v-clicks>

<div v-click class="mt-3 text-sm">

> ⚠️ 行尾两空格「肉眼不可见」，还常被编辑器自动裁掉——很多人更爱用行尾 `\` 或空行分段。

</div>

<!--
段落是最基础的叶子块，一行或连续多行文本，段与段之间用空行分隔。这里最容易踩的是换行语义。

软换行是段落内一个普通换行，多数渲染器把它折叠成一个空格，输出仍在同一行文本流内，不产生 br。硬换行要真正的行内换行，有两种标准触发：行尾放两个及以上空格，或行尾放一个反斜杠，二者都渲染为 br。要另起一个段落，需要一个空行。

行尾两空格肉眼不可见是新手最常漏的坑，很多编辑器还会自动裁剪行尾空格，因此不少人更偏爱行尾反斜杠或干脆用空行分段。
-->

# 列表：有序、无序、嵌套

```md
- 无序项（-、*、+ 任选，同列表别混用）
- 第二项
  - 嵌套子项（靠缩进对齐）

1. 有序项（全写 1. 也会自动递增）
1. 第二项
1. 第三项
```

<v-clicks>

- 无序标记 `-` / `*` / `+` 语义相同，但同一列表内混用会被拆成多个列表
- 有序列表：**起始编号看第一项，后续数字被忽略、自动递增**（首项写 `3.` 就从 3 起）
- 嵌套靠缩进对齐子项

</v-clicks>

<!--
无序列表用连字符、星号或加号任一，三者语义相同，但同一列表内不宜混用，否则会被拆成多个列表。

有序列表用数字加点或数字加右括号。一条重要规则：起始编号由第一项决定，后续项的具体数字被忽略、只按顺序自增。所以全写 1. 会渲染成 1、2、3；首项写 3. 则从 3 起数。这带来一个好处：增删条目时不用手动重排数字。

嵌套靠缩进对齐子项，一般两格或四格，与父项标记后的文字起点对齐最稳。
-->

# 松紧列表 tight vs loose（高频坑）

「列表莫名变稀疏」的根因——**项与项之间的空行**：

```md
- A          ← 紧凑(tight)：项内容不被 <p> 包裹
- B

- A          ← 松散(loose)：项间有空行，
             ←   每项被 <p> 包裹、间距变大
- B
```

<div v-click class="mt-3 text-sm">

> 判定：**任意两项之间出现空行**（或项内含空行分隔的多个块）→ 整个列表变 loose → 所有项内容被 `<p>` 包裹。想要紧凑观感却被撑开时，第一反应就查项间空行。

</div>

<!--
这是列表莫名变稀疏的根因。判定规则：只要任意两个列表项之间出现空行，或列表项内部由空行分隔的多个块，整个列表就变成松散列表，此时每一项的内容都会被 p 标签包裹，视觉上行距明显变大；反之项与项紧挨、无空行则是紧凑列表，不加 p。

所以想要紧凑的观感却发现列表被撑开时，第一反应就应该是检查项与项之间是不是多了空行。这个坑几乎每个写 Markdown 的人都踩过。
-->

# 代码：行内 / 围栏 / 缩进

<v-clicks>

- **行内代码**：一对反引号包裹；内容含反引号时用**更多反引号定界**（两侧数量相等）
- **围栏代码块**（首选）：3+ 个反引号或 `~~~`，上下须同种字符；起始行 info string 首词标**语言**
- **缩进代码块**（老写法）：每行缩进 **4 空格**；无法标语言，且**不能中断段落**（前面需空行）

</v-clicks>

```md
行内用 `code`；下面是围栏码（也可用三个反引号）：

~~~js
const x = 1;   // 起始围栏后的 js 指定高亮语言
~~~
```

<!--
Markdown 有三种代码写法。行内代码用一对反引号包裹，内容以等宽字体原样显示，且内部不再解析任何 Markdown 或转义；当代码内容本身包含反引号时，不能用反斜杠转义，而要用更多反引号作定界符，两侧数量相等且与内容里的反引号数不同。

围栏代码块是如今首选，用连续三个及以上反引号或三个波浪号作上下围栏，上下须同种字符不能混用；围栏码不需要每行缩进，还能在起始行紧跟 info string，其第一个词按约定表示语言，驱动语法高亮。

缩进代码块是最古老的写法，每行缩进至少四个空格。它两个短板：无法标注语言不能高亮，且不能中断一个段落，若上一行是段落文字下一行直接缩进四空格，这行会被当段落的软换行延续，要生效前面必须有空行。
-->

# 引用与分隔线

```md
> 一级引用（容器块，可放列表/标题/代码）
>
> > 嵌套引用
> - 引用里的列表项

***          ← 分隔线：3+ 个相同 - * _
```

<v-clicks>

- **块引用**行首 `>`，是**容器块**：内部可嵌套引用、放列表/标题/代码等其它块
- **分隔线**：单独一行、3 个及以上**相同**字符（`-`/`*`/`_`），混用字符不成立
- ⚠️ `-` 型分隔线紧贴上一行文字会被当 **Setext 二级标题**——前后留空行

</v-clicks>

<!--
块引用在每行行首加大于号加一空格。它是容器块，内部可以嵌套其它块级元素：标题、列表、代码块，乃至用两个大于号嵌套的更深引用。

分隔线是单独一行、三个及以上相同字符连字符、星号或下划线组成，字符间可有空格。混用不同字符比如连字符加星号不构成分隔线。

再次提醒那个交叉陷阱：连字符型分隔线紧贴上一行文字会被当成 Setext 二级标题，所以分隔线前后请留空行。用星号或下划线型分隔线可以绕开这个歧义。
-->

# 强调：斜体、加粗与侧翼规则

```md
*斜体* 或 _斜体_
**加粗** 或 __加粗__
***粗斜体***

un*frigging*believable   ← 词内斜体 ✅
un_frigging_believable   ← 不生效 ❌
```

<div v-click class="mt-3 text-sm">

> **侧翼规则**（left/right-flanking）：`_` 在两侧都是字母数字时不作分界符，护住 `snake_case`、`file_name` 不被误强调；`*` 无此限制。**结论：单词内部强调用 `*`。**

</div>

<!--
强调分两级，可用星号或下划线。单个符号是斜体 em，两个是加粗 strong，三个是二者叠加。

关键差异在单词内部：CommonMark 用左右侧翼分界符规则判定一个星号或下划线能否作为开闭标记。简化说，下划线在两侧都是字母数字时不被视为有效分界符，于是 snake_case、file_name 里的下划线不会被误当强调；而星号没有这条内部限制。

所以 un 星号 frigging 星号 believable，词内 frigging 变斜体；但换成下划线就不生效，原样保留。结论：需要在单词内部做强调时，用星号而不是下划线。
-->

# 链接：行内 / 引用式 / 自动

```md
行内： [文字](https://commonmark.org "可选标题")

引用式（URL 与正文分离，可复用）：
参见 [CommonMark][cm]。
[cm]: https://commonmark.org

自动链接： <https://example.com> 、 <a@b.com>
```

<v-clicks>

- **行内链接**：`[文字](url "标题")`，标题渲染为悬停提示
- **引用式**：正文 `[文字][标签]` + 别处 `[标签]: url`；简写 `[文字][]` / `[文字]`
- **自动链接**：CommonMark 需**尖括号**包 URL/邮箱；裸 URL 成链是 **GFM 扩展**

</v-clicks>

<!--
Markdown 有三类链接。行内链接方括号内是显示文字，圆括号内是 URL，URL 后可跟引号包裹的可选标题，渲染为悬停 title。

引用式链接把 URL 定义与正文分离，正文写方括号文字加方括号标签，在文档别处集中定义标签。适合链接多的长文档，URL 可复用、正文更干净。它还有折叠式和快捷式两个简写。

自动链接方面，CommonMark 的自动链接必须用尖括号包住一个绝对 URI 或邮箱。注意标准 CommonMark 下裸写的 URL 不会自动成链，那是 GFM 的扩展自动链接才有的能力，后面会讲。
-->

---

# 图片与转义

```md
![替代文字](/logo.png "可选标题")     ← 比链接多前导 !
[![可点击的图片](/thumb.png)](https://x.com)

\* \# \` \|  ← 反斜杠转义，输出字面标点
```

<v-clicks>

- **图片** = `!` + 链接语法；方括号内是 **alt 替代文字**；套进链接即可点击
- **转义**：`\` + **任意 ASCII 标点**转字面（`\*`/`\#`/`\|`…）
- ⚠️ 转义在**代码块、代码跨度、自动链接、原始 HTML 内不生效**

</v-clicks>

<!--
图片语法就是链接前加一个感叹号，方括号里的内容是 alt 替代文字而非可点击文字，圆括号里是图片地址、可带可选标题。想让图片可点击，就把图片语法整体当作链接的文字部分嵌进去。

转义方面，要显示某个有特殊含义的字符本身，在它前面加反斜杠。CommonMark 规定任意 ASCII 标点字符都可被反斜杠转义，比如字面星号、井号、竖线。

注意转义在代码块、代码跨度、自动链接、原始 HTML 内部不生效，因为这些语境本就原样处理。此外 Markdown 也识别 HTML 实体与数字字符引用，比如 and 符号分号、井号数字分号，可用来输入不便直接打的符号。
-->

# HTML 内嵌与安全

```md
<details>
  <summary>点击展开</summary>
  这里是被折叠的内容。
</details>
```

<v-clicks>

- Markdown 允许直接内嵌**原始 HTML**并**原样透传**（复杂表格、`<details>` 折叠等）
- ⚠️ 传统约定：**块级 HTML 标签内部默认不再解析 Markdown**
- **安全**：透传即风险，渲染不可信来源**必须 sanitize**；GFM **tagfilter** 转义 `script`/`style`/`iframe` 等 9 个风险标签的起始 `<`（仅一层）

</v-clicks>

<!--
Markdown 从设计之初就允许直接内嵌原始 HTML。写 Markdown 表达不了的排版时，比如复杂表格、details 折叠、带 class 的容器，直接写 HTML，它会被原样透传到输出。

一个传统约定要记住：块级 HTML 标签内部，默认不再把内容当 Markdown 解析，CommonMark 的 HTML 块规则亦如此。想在 div 里继续用 Markdown 语法，往往需要解析器额外支持或特定的空行写法。

安全是内嵌 HTML 的另一面：既然 HTML 透传，用户提交的 Markdown 就可能夹带 script、iframe 等危险标签。因此渲染不可信来源的 Markdown 必须做 HTML 清洗。GFM 为此提供 tagfilter 扩展，会把九个风险标签的起始尖括号转义成实体，但它只是一层，GitHub 转 HTML 后还有额外清洗。自建渲染要搭配 DOMPurify、sanitize-html 这类库。
-->

# GFM 扩展①：表格

```md
| 语法 | 含义 | 对齐示例 |
| :--- | :--: | -------: |
| `#`  | 标题 | 左/中/右 |
| `>`  | 引用 | 由冒号定 |
```

<v-clicks>

- 三部分：**表头行 + 分隔行（必需）+ 数据行**，单元格用 `|` 分隔
- **对齐**由分隔行冒号定：`:---` 左、`:---:` 中、`---:` 右
- **表头与分隔行列数必须一致**；数据行列数可不同（不足补空、超出截断）
- 单元格内字面竖线用 `\|`；单元格内**不能真换行**，要换行用 `<br>`

</v-clicks>

<!--
GFM 表格由三部分组成：表头行、紧接其下的分隔行、以及若干数据行，各单元格用竖线分隔，首尾竖线可选。

规则要点：分隔行是必需的，由连字符构成、可加冒号表示对齐，它的单元格数必须与表头一致，否则整体不被识别为表格。对齐由分隔行冒号位置决定，左端冒号左对齐，两端冒号居中，右端冒号右对齐。

数据行的列数可以与表头不同，不足则右侧补空单元格，超出的被丢弃，这与表头分隔行必须匹配是两回事。单元格内的字面竖线要转义成反斜杠竖线，单元格内不能包含真正的换行，需要换行用 br 标签。单元格内可用行内格式如链接、code、强调，但不能放标题、引用等块级元素。
-->

---

# GFM 扩展②：任务列表与删除线

```md
- [x] 已完成的任务
- [ ] 待办任务

~~这段文字被划掉~~   ← 删除线：两个波浪号
```

<v-clicks>

- **任务列表**：列表项标记后加 `[ ]`（未完成）/ `[x]`（已完成），GitHub 上可交互勾选
- **删除线**：`~~文字~~`——**规范要求两个波浪号**
- ⚠️ 单个 `~` 只在 github.com 等部分实现生效，GitLab/VS Code 只认 `~~`；且单 `~` 在 Pandoc 里可能是**下标**——跨平台统一 `~~`

</v-clicks>

<!--
任务列表在普通列表项的标记后面加一个方括号复选框即可，方括号空格表示未完成，方括号 x 表示已完成，在 GitHub 上会渲染成可交互勾选的复选框。

删除线用两个波浪号包裹。一个易混点：GFM 规范定义的是两个波浪号；虽然 github.com 的实现放宽到单个波浪号也能触发，但那是实现层的额外宽松，GitLab、VS Code 等许多解析器只认双波浪号。而且单个波浪号在 Pandoc 等处理器里可能被当作下标语法，语义完全不同。因此为跨平台可移植性，应统一用双波浪号。
-->

# GFM 扩展③：扩展自动链接与 tagfilter

<v-clicks>

- **扩展自动链接**：**裸写**的 `http://`/`https://`/`www.` URL 与邮箱**无需尖括号**即自动成链
  - 智能修剪末尾标点：`访问 https://commonmark.org。` 的句号不会被吞进链接
- **tagfilter（禁用原始 HTML）**：把 9 个风险标签的起始 `<` 转义成 `&lt;`

</v-clicks>

<div v-click class="mt-3 text-sm">

```txt
title  textarea  style  xmp  iframe  noembed  noframes  script  plaintext
```

> 只转义起始尖括号、不删内容；名单外 HTML 照常透传。它只是**一层**，GitHub 另有后处理清洗。

</div>

<!--
CommonMark 的自动链接必须用尖括号包住 URL；GFM 的扩展自动链接更省事，裸写的地址也会被自动识别成链接：http、https 开头的 URL，www 开头的地址，以及形如 user at example.com 的邮箱。它还会做末尾标点智能修剪，URL 后紧跟的句号、逗号、感叹号、问号等不会被算进链接，成对的括号也会被正确匹配。

tagfilter 也就是禁用原始 HTML 扩展，针对一小撮有安全或渲染破坏风险的标签，把它们的起始尖括号转义成实体，使其变成字面文本。名单是固定的九个：title、textarea、style、xmp、iframe、noembed、noframes、script、plaintext。

要点：它只转义起始尖括号，不删除内容、不改语法，名单之外的 HTML 如 div、span、details 仍照常透传；它只是一层防护，GitHub 转 HTML 后还会做额外的后处理与内容清洗。自建渲染管线别只依赖 tagfilter。
-->

# front matter 与方言地图

```yaml
---
title: 我的文章
date: 2026-07-05
layout: doc
---
```

<v-clicks>

- **front matter**：文件**顶部** `---` 包裹的 YAML 元数据；**非 Markdown 规范**，是 SSG（Jekyll/Hugo/VitePress/Astro）约定，**渲染前先被剥离**；另有 TOML（`+++`）、JSON 变体
- 方言地图：传统 Markdown（基线）→ **CommonMark**（消歧）→ **GFM**（超集）；**MMD/Markdown Extra**（早期补表格/脚注）；**Pandoc**（最全、多格式互转）

</v-clicks>

<!--
几乎所有静态站点生成器都约定，Markdown 文件顶部可以放一段用三连字符包裹的元数据块，最常见是 YAML，用来写标题、日期、标签、布局、是否草稿等。

关键认知：它不是 CommonMark 或 GFM 规范的一部分，而是各工具比如 Jekyll、Hugo、VitePress、Astro、Gatsby、11ty 自定义的约定；必须位于文件最顶部，由生成器在把正文交给 Markdown 解析器之前先解析剥离；除 YAML 外还有 TOML front matter，Hugo 用三加号包裹，以及 JSON front matter。

方言地图：传统 Markdown 是基线、有歧义；CommonMark 是消歧规范；GFM 是 CommonMark 严格超集加表格等；MultiMarkdown 和 Markdown Extra 早期补齐表格、脚注、定义列表；Pandoc Markdown 是特性最全的超集之一，脚注、多种表格、数学、引文、属性一应俱全，还支持几十种格式互转。
-->

# 常见坑 Top 8

<v-clicks>

- 行尾**两空格硬换行不可见**，还常被编辑器裁掉 → 用 `\` 或空行
- **列表松紧**：项间空行 → loose → 每项被 `<p>` 包裹、间距变大
- **`#` 后缺空格**不成标题；永远「井号 + 空格」
- **`_` 单词内部不强调**（侧翼规则）→ 词内用 `*`
- **`-` 分隔线紧贴文字**被当 Setext 二级标题 → 前后留空行
- **表格**：单元格 `\|` 转义、换行 `<br>`、表头与分隔行列数一致
- **裸 URL 仅 GFM 成链**；标准 CommonMark 要尖括号
- **删除线统一 `~~`**；**缩进码不能中断段落**；**块级 HTML 内不解析 Markdown**

</v-clicks>

<!--
过一遍最高频的踩坑点。第一，行尾两空格硬换行肉眼不可见，还常被编辑器自动裁掉，改用行尾反斜杠或空行分段更稳。第二，列表松紧，项与项之间多一个空行整列表变松散，每项被 p 包裹、行距变大。第三，井号后缺空格不成标题，永远井号加空格。第四，下划线在单词内部不强调，侧翼规则护住 snake_case，词内强调用星号。

第五，连字符型分隔线紧贴上一行文字会被当 Setext 二级标题，分隔线前后留空行。第六，表格三连坑：单元格字面竖线要转义、单元格内换行要用 br、表头与分隔行列数必须一致。第七，裸 URL 只在 GFM 自动成链，标准 CommonMark 要用尖括号。第八，删除线统一用双波浪号，缩进码不能中断段落，块级 HTML 内部默认不解析 Markdown。
-->

# 与 MDX 的关系 · 选型建议

<v-clicks>

- **Markdown vs MDX**：标准 Markdown 把 HTML 当**静态文本透传**、不执行；MDX 是「**Markdown + JSX**」，能 `import` 并运行真实组件、写 JSX——把文档**编译成组件**（细节见 MDX 叶）
- **跨平台协作文档** → 坚持 CommonMark 核心 + GFM 五扩展，最稳
- **重度文档 / 学术出版**（脚注、公式、引文）→ Pandoc / MultiMarkdown 方言更强，但产物依赖特定工具链
- **用扩展语法前**先确认目标平台是否支持，否则原样露出字面文本

</v-clicks>

<!--
先说和 MDX 的关系：标准 Markdown 里的 HTML 是静态透传的文本，不会执行；MDX 是 Markdown 加 JSX，能 import 并渲染真实的 React、Vue 组件，写 JSX 表达式，它把文档编译成组件而非只转 HTML。一句话，Markdown 透传静态 HTML，MDX 编译并运行组件。MDX 的细节见独立的 MDX 叶。

选型建议：面向 GitHub 或多平台协作的文档，坚持 CommonMark 核心加 GFM 五扩展最稳；做重度文档或学术出版，需要脚注、公式、引文，Pandoc 或 MultiMarkdown 方言更强，但要接受产物依赖特定工具链；用脚注、上下标、高亮这类扩展语法前，务必先确认目标平台是否支持，否则会被原样输出成难看的字面文本。
-->

# 总结

Markdown = **纯文本可读的轻量级标记语言**，CommonMark 消歧 · GFM 补扩展

- 两层结构：块级元素先切分、行内元素后解析（叶子块 vs 容器块）
- 核心语法：标题（ATX/Setext）、列表（含松紧坑）、代码（围栏/缩进/行内）、强调（侧翼规则）、链接/图片/转义
- HTML 可透传但需 sanitize；GFM 加表格/任务列表/删除线/扩展自动链接/tagfilter
- front matter 与脚注等属**非规范约定**，跨平台前先确认支持
- 版本：CommonMark `0.31.2`（2024-01）· GFM 严格超集

核心心智：解析分两层，块级元素先切分文档结构，行内元素后在块内解析，块又分叶子块和容器块。核心语法包括标题的 ATX 和 Setext 两套、列表及其松紧陷阱、代码的围栏缩进行内三态、强调及侧翼规则、链接图片与转义。HTML 可以透传但渲染不可信来源必须 sanitize；GFM 加了表格、任务列表、删除线、扩展自动链接、tagfilter 五类扩展。front matter 与脚注、上下标等属于非规范约定，跨平台使用前先确认支持。
