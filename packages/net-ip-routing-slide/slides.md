# IP 协议与 IPv4 寻址

网络层的「快递地址系统」

---

# IP 协议是什么

工作在**网络层（第 3 层）**，干三件核心工作

<v-click>

- **逻辑寻址（Addressing）**：给每台设备一个 IP 地址作为唯一标识
- **路由（Routing）**：按目的 IP 把数据包一跳一跳转发到目的地
- **尽力而为投递**：尽最大努力送达，**只是「尽力」非「保证」**

</v-click>

<v-click>

它还是**无连接 + 不可靠**的（这是设计取舍）：

- **无连接**：发包前不建连、不握手，每个包独立寻路
- **不可靠**：不保证送达/顺序/不重复 —— 可靠性**下放**给上层 **TCP**，**UDP** 则原样暴露

</v-click>

<!--
IP 三件事：寻址、路由、投递（best-effort）。它无连接、不可靠，可靠性下放给 TCP，UDP 原样暴露。正因 IP「少做事」，海量路由器才能低成本高速转发。
-->

---

# IPv4 地址：32 位与网络/主机号

日常说的「IP 地址」通常仍指 32 位 IPv4

<v-click>

- **长度**：32 位二进制（4 字节）
- **写法**：拆 4 段、每段 8 bit 转十进制、用点连接 —— `a.b.c.d`，每段 `0~255`
- **总量**：2³² ≈ **42.9 亿** 个 —— 早已不够分（私有地址 + NAT 与 IPv6 的根因）

</v-click>

<v-click>

逻辑上分两部分（如 `192.168.1.10/24`）：

- **网络号**（前缀）：标识「在哪个网络」，同网设备**相同**
- **主机号**（后缀）：标识「网络内哪台主机」
- 分界线**不固定**，由子网掩码 / CIDR 前缀（如 `/24`）决定 —— 下一节细讲

</v-click>

<!--
32 位、点分十进制、每段 0~255、约 43 亿个。地址逻辑上分网络号 + 主机号，分界线不固定、由掩码决定，这是子网那一节的地基。
-->

---

# 传统分类 A/B/C/D/E

按首字节高位比特硬切边界（**历史概念**）

<v-click>

| 类别 | 首位比特 | 首字节范围 | 默认划分 |
| --- | --- | --- | --- |
| A | `0` | 1~126 | 8 位网络 + 24 位主机 |
| B | `10` | 128~191 | 16 位网络 + 16 位主机 |
| C | `110` | 192~223 | 24 位网络 + 8 位主机 |
| D | `1110` | 224~239 | 组播 Multicast |
| E | `1111` | 240~255 | 保留 / 实验 |

</v-click>

<v-click>

> **1993 年起被 CIDR 取代**：固定边界太浪费，今天只作历史概念理解。

</v-click>

<!--
分类法固定边界太死板——要 300 个地址，C 类不够、B 类浪费六万。1993 年 CIDR 用任意 /n 取代它。这表只需作历史概念记。
-->

---

# 公网地址 vs 私有地址

不是所有 IPv4 都能在互联网上「裸奔」

<v-click>

| 私有地址段（RFC 1918） | CIDR | 常见场景 |
| --- | --- | --- |
| `10.0.0.0` | `10.0.0.0/8` | 大型企业 / 数据中心 |
| `172.16.0.0` | `172.16.0.0/12` | 中型网络 |
| `192.168.0.0` | `192.168.0.0/16` | 家用路由器、小型办公 |

</v-click>

<v-click>

- **公网**：全球唯一，由 IANA / RIR 分配，能被互联网直接路由
- **私有**：内网自治、**可重复使用**，访问公网时由路由器做 **NAT** 转换出去

</v-click>

<!--
私有地址三段要背熟：10/8、172.16/12、192.168/16。它们不进公网路由表，所以你家和邻居家都能用 192.168.1.1，靠 NAT 出网。
-->

---

# 特殊地址

不是给主机「正常用」的那些，调试常打照面

<v-click>

| 地址 / 网段 | 名称 | 含义 |
| --- | --- | --- |
| `0.0.0.0` | 本机 / 未指定 | 服务端 `bind 0.0.0.0` = 监听所有网卡 |
| `127.0.0.1` (`/8`) | 环回 Loopback | localhost，指向本机，数据不出网卡 |
| `255.255.255.255` | 受限广播 | 发本网段所有主机，路由器不转发 |
| `224.0.0.0/4` | 组播 Multicast | 发给「订阅了某组」的一组主机 |
| `169.254.0.0/16` | 链路本地 | 拿不到 DHCP 时自动生成（APIPA） |

</v-click>

<!--
重点辨析 127.0.0.1 与 0.0.0.0：前者是「连我自己」的访问端，后者是「监听所有 IP」的监听端。前端起 dev server 最常踩这个概念点。
-->

# IPv4 首部要点

共 14 个字段（1 个可选），前端认得这几个就够

<v-click>

| 字段 | 作用 |
| --- | --- |
| 版本 Version | 4 bit，IPv4 值 `4` / IPv6 值 `6` |
| TTL | 跳数上限，每过一路由器减 1，到 0 丢弃（**防打转**） |
| 协议号 Protocol | 上层是谁：**TCP=6 / UDP=17 / ICMP=1** |
| 首部校验和 | 仅校验首部、不含数据 |
| 源 IP / 目的 IP | 32 位收发方地址，**路由器据目的 IP 选路** |

</v-click>

<v-click>

> **协议号 ≠ 端口号**：协议号在 IP 首部分上层协议；端口号在 TCP/UDP 首部分进程。

</v-click>

<!--
首部重点认五个：Version、TTL、协议号、源/目的 IP。协议号 6/17/1 要背。TTL 减到 0 回送 ICMP，是后面 traceroute 的伏笔。
-->

掩码定分界，AND 算网络，借位做子网

# 子网掩码与按位与

掩码标出分界线，AND 算出网络地址

<v-click>

- 掩码也 32 位：**连续的 1 标网络位，紧接连续的 0 标主机位**（RFC 4632 要求连续）
- 掩码**不随包在公网传输**，是路由器/主机本地配置
- **网络地址 = IP `按位与` 掩码**（主机位清零）；**广播地址** = 主机位**全置 1**

</v-click>

<v-click>

```text
       IP   192.168.10.37   11000000.10101000.00001010.00100101
  AND  掩码  255.255.255.0   11111111.11111111.11111111.00000000
  ──────────────────────────────────────────────────────────────
  网络地址  192.168.10.0    11000000.10101000.00001010.00000000
```

</v-click>

<v-click>

> 同子网所有主机 AND 后得**相同网络地址** —— 这是判断「是否同网段」的依据。

</v-click>

<!--
掩码是连续 1 + 连续 0、只在本地用。核心操作 AND：主机位清零得网络地址、全置 1 得广播地址。同网段判断、要不要走网关全靠这个 AND。
-->

---

# CIDR：用 `/前缀` 取代分类

1993 年 RFC 1518/1519 引入，2006 年 RFC 4632 更新

<v-click>

- **CIDR**（无类域间路由）抛弃「类」：分界可落在 `0~32` **任意一位**
- 写法 **`IP/前缀长度`**，前缀 = 掩码中 1 的个数
- `/24` 即掩码前 24 位为 1，等价 `255.255.255.0`

</v-click>

<v-click>

```text
198.51.100.14/24   ←→  网络 198.51.100.0，掩码 255.255.255.0
10.0.0.0/22        ←→  掩码 255.255.252.0，含 1024 个地址
```

</v-click>

<v-click>

> 现代设备、路由协议（OSPF/BGP）一律用 `/前缀`，简洁且不受「类」束缚。

</v-click>

<!--
CIDR 把分界从「A/B/C 三种死尺寸」解放到任意一位。/前缀 = 掩码里 1 的个数，是今天唯一的写法。
-->

---

# CIDR 前缀对照表（常用段）

块大小 = `2^(32-n)`；可用主机 = `2^h − 2`

<v-click>

| 前缀 | 子网掩码 | 地址总数 | 可用主机 |
| --- | --- | --- | --- |
| `/24` | 255.255.255.0 | 256 | 254 |
| `/26` | 255.255.255.192 | 64 | 62 |
| `/28` | 255.255.255.240 | 16 | 14 |
| `/30` | 255.255.255.252 | 4 | 2（点对点，减 2） |
| `/31` | 255.255.255.254 | 2 | **2**（RFC 3021，不减 2） |
| `/32` | 255.255.255.255 | 1 | 1（单主机路由） |

</v-click>

<v-click>

> **减 2** 扣掉「网络地址 + 广播地址」；**`/31` 点对点两个全可用、`/32` 表单台主机**是两个例外。

</v-click>

<!--
手算速查表。块大小 2^(32-n)、可用主机 2^h−2。/24=254、/26=62、/30=2 最常考。/31（点对点不减 2）和 /32（单主机）是减 2 规则的例外。
-->

---

# 子网划分：从主机位「借」位

把一个大网络切成若干小子网

<v-click>

以 `/24` 切 4 个子网为例 —— 借 2 位主机位（`2² = 4`），`/24` → `/26`：

```text
原网络   192.168.10.0/24
子网 0   192.168.10.0/26    .1   – .62    广播 .63
子网 1   192.168.10.64/26   .65  – .126   广播 .127
子网 2   192.168.10.128/26  .129 – .190   广播 .191
子网 3   192.168.10.192/26  .193 – .254   广播 .255
```

</v-click>

<v-click>

> 手算快捷法：**块大小 = 256 − 掩码末段值**（`256 − 192 = 64`），网络地址按块大小递增。

</v-click>

<!--
借位口诀：借 n 位主机位得 2^n 个子网。块大小 = 256 − 掩码末段，这里 256−192=64，所以网络地址按 64 递增：0/64/128/192。
-->

---

# 一个完整子网计算示例

给定 IP `172.16.20.200`、掩码 `/26`，求它在哪个子网

<v-click>

```text
       IP   172.16.20.200  末段 200 = 11001000
  AND  /26  255.255.255.192 末段 192 = 11000000
  ─────────────────────────────────────────────
  网络地址  172.16.20.192   （200 落在 192 这个块）
  可用范围  .193 – .254     广播 .255
  可用主机  2^6 − 2 = 62
```

</v-click>

<v-click>

> 块大小 `256 − 192 = 64` → 边界 `…128 / .192 / .256`，`200` 落在 `[192, 255]` 区间。

</v-click>

<!--
这是手算标准流程：先算块大小 64，列边界 0/64/128/192，看 200 落在 192 块；可用 .193–.254，广播 .255，主机 62。
-->

# VLSM：同网络里掩码可变

按各网段真实规模量体裁衣，杜绝浪费

<v-click>

`192.168.1.0/24` 服务「100 台 / 50 台 / 两条点对点」，从大到小切：

```text
100 台主机  →  192.168.1.0/25    可用 126   .0   – .127
 50 台主机  →  192.168.1.128/26  可用 62    .128 – .191
点对点链路1 →  192.168.1.192/30  可用 2     .192 – .195
点对点链路2 →  192.168.1.196/30  可用 2     .196 – .199
```

</v-click>

<v-click>

> 口诀：按「主机需求从大到小」排序，从起点连续分配，避免碎片无法对齐。

</v-click>

<!--
VLSM 允许同一网络不同子网用不同掩码。诀窍是从大需求往小排、连续切，否则会留下对不齐的碎片。CIDR 就建立在 VLSM 之上。
-->

# 超网与路由聚合

子网划分的反向操作：把成片小块收成一条路由

<v-click>

```text
198.51.96.0/24
198.51.97.0/24      16 个连续 /24
198.51.98.0/24   →  聚合通告   198.51.96.0/20
   …（共 16 个）        （1 条路由替代 16 条）
198.51.111.0/24
```

</v-click>

<v-click>

- 前缀从 `/24` 缩短到 `/20`（覆盖 `16 × 256 = 4096` 个地址）
- 上游路由器只需 **1 条表项**就能匹配这 16 个网络 → **压缩路由表**
- 前提：地址块**连续且可对齐**

</v-click>

<!--
超网聚合方向和子网相反：前缀变短、多网并一。目的是压缩全球路由表。它是 BGP 路由表得以「相对可控」增长的关键机制。
-->

# 为什么要划分子网

不是为了好看，是解决真实工程问题

<v-click>

- **隔离广播域、缩小故障域**：ARP/DHCP 广播只在本子网泛洪，风暴不蔓延全网
- **提升 IP 利用率**：配合 VLSM 按需切，避免「要么不够、要么浪费六万」
- **缩短路由路径**：按子网分拣后能走更直接的路径，不必绕路
- **便于管理与安全**：按部门/楼层分子网，可在边界施加 ACL / 防火墙策略

</v-click>

<v-click>

> 一句话主线：**掩码定分界，AND 算网络，借位做子网，聚合省路由。**

</v-click>

<!--
划子网四大收益：隔离广播域、提升利用率、缩短路径、便于管理安全。每个子网就是一个独立广播域，这点最重要。
-->

# IPv6 与过渡技术

128 位，从根上解决地址耗尽

---

# 为什么需要 IPv6

IPv4 地址早已耗尽

<v-click>

- IPv4 只有 **32 位 ≈ 43 亿**个地址，1983 年投用，随设备爆炸早已不够分
- IANA 顶级地址池 **2011 年**即分配完毕，各区域注册局陆续告罄
- 今天还能上网靠 **NAT**「续命」，但 NAT 破坏端到端连通，治标不治本

</v-click>

<v-click>

> IPv6 把地址拉到 **128 位 ≈ 3.4×10³⁸**（约 340 涧）—— 从根上解决耗尽。

</v-click>

<!--
IPv6 的诞生动机就一句：IPv4 的 43 亿地址 2011 年分完了。NAT 是续命手段但破坏端到端。IPv6 用 128 位彻底解决。
-->

---

# IPv6 地址表示与零压缩

**8 组、每组 16 位**（4 个十六进制数），冒号 `:` 分隔，共 128 位

<v-click>

| 步骤 | 规则 | 示例 |
| --- | --- | --- |
| 原始 | 完整 8 组 | `2001:0db8:0000:0000:0000:ff00:0042:8329` |
| 规则① | 每组**省前导零** | `2001:db8:0:0:0:ff00:42:8329` |
| 规则② | **连续全零组**用 `::` 折叠 | `2001:db8::ff00:42:8329` |

</v-click>

<v-click>

- **`::` 全址只能用一次**：出现两次就无法判断各代表几组零，**不可还原**（最常见错误）
- URL 带端口要用方括号：<code v-pre>http://[2001:db8::1]:8080/</code>

</v-click>

<!--
IPv6 = 8 组 × 16 位、冒号分隔。零压缩两规则：① 省前导零 0042→42；② 连续全零组折叠成 ::。红线：:: 只能用一次。URL 里要用方括号包。
-->

---

# 前缀 `/64` 与特殊地址

IPv6 子网标准**固定**为 `/64`

<v-click>

```text
2001:db8:abcd:0012 : 0000:0000:0000:0001 / 64
└──── 前缀（64 位）────┘ └── 接口标识 IID（64 位）──┘
```

</v-click>

<v-click>

- 前 64 位标识网络（ISP / 路由器下发），后 64 位是**接口标识**（可由 MAC EUI-64 或随机生成）
- **未指定地址** `::`（等价 IPv4 `0.0.0.0`）
- **回环地址** `::1`（等价 IPv4 `127.0.0.1`，即 localhost）

</v-click>

<!--
IPv6 子网固定 /64：前 64 位网络前缀，后 64 位接口标识 IID。后 64 位能自动生成，是 SLAAC 的基础。::1 是回环，:: 是未指定。
-->

# IPv6 核心优势

不只是「地址更多」

<v-click>

| 维度 | IPv4 | IPv6 |
| --- | --- | --- |
| 地址长度 | 32 位（≈43 亿） | **128 位（≈3.4×10³⁸）** |
| 首部 | 可变 20~60 字节 | **固定 40 字节** |
| 首部校验和 | 有（每跳重算） | **取消** |
| 分片 | 路由器可沿途分片 | **仅源主机分片** |
| 地址配置 | 多靠 DHCP / 手动 | **SLAAC 自动配置** |
| NAT | 普遍依赖 | **端到端直连，无需 NAT** |
| 广播 | 有 | **无（改用组播）** |

</v-click>

<!--
IPv6 优势一表收：地址海量、固定 40 字节首部、取消校验和、路由器不分片、SLAAC 自动配置、端到端无 NAT、废弃广播。转发更快是综合结果。
-->

---

# 优势细看：转发更快 + 自动配置

挑前端/运维最有感的两条展开

<v-click>

- **取消首部校验和**：IPv4 校验和需**每跳重算**（TTL 在变），是路由器固定开销；IPv6 认为链路层 + 传输层已保完整性，**整层去掉**
- **路由器不再分片**：分片只由源主机完成，减少逐包处理开销
- **SLAAC 无状态自动配置**：设备先自动生成链路本地地址，再监听路由器**通告（RA）**拿前缀，**自己拼出**全局地址 —— 无需 DHCP 服务器即可上网

</v-click>

<!--
两条最有感：① 取消校验和 + 路由器不分片 → 转发开销小；② SLAAC → 设备自己拼地址，连 DHCP 都不用。这是 IPv6 「简化首部」的典型体现。
-->

# IPv6 地址类型

用前缀区分作用范围

<v-click>

| 类型 | 前缀 | 范围 | 类比 / 说明 |
| --- | --- | --- | --- |
| 全局单播 | `2000::/3` | 全球 | 公网可路由，类比公网地址 |
| **链路本地** | `fe80::/10` | 仅本链路 | 每接口必有、自动生成、不跨路由器 |
| 唯一本地（ULA） | `fc00::/7` | 组织内部 | 类比 IPv4 私有地址 |
| **组播** | `ff00::/8` | 一组接口 | 一对多，**替代广播** |

</v-click>

<v-click>

- `ifconfig` / `ip addr` 里那个 `fe80::` 开头的就是**链路本地地址**，你天天见
- IPv6 **彻底废弃广播**：一对多全用**组播**，`ff02::1` 全节点、`ff02::2` 全路由器，更省资源

</v-click>

<!--
四类地址：全局单播（公网）、链路本地 fe80::（必有、不跨路由器）、ULA fc00::（私有）、组播 ff00::。IPv6 废弃广播改组播，ff02::1 全节点、ff02::2 全路由器。
-->

# 三大过渡技术

IPv4 与 IPv6 不能直接互通，需长期共存

<v-click>

- **双栈（Dual-Stack）**：主机/路由器**同时跑** v4 + v6，对端是谁用谁 —— 最平滑、最推荐
- **隧道（Tunneling）**：把 IPv6 报文**封进 IPv4 报文**穿越只支持 v4 的网络（如 6in4，用协议号 **41**）
- **翻译（NAT64 + DNS64）**：让**纯 IPv6** 客户端访问**纯 IPv4** 服务

</v-click>

<v-click>

> 选哪种看你卡哪步：**两端能升级→双栈**；**中间是 v4→隧道**；**对端是回不去的 v4→翻译**。

</v-click>

<!--
三种过渡：双栈（最平滑）、隧道（6in4 封装穿 v4 网）、NAT64/DNS64（纯 v6 访问纯 v4）。三者常叠加用，支撑数十年渐进迁移。
-->

# 路由原理与路由器/网关

每台路由器只回答一个问题：「下一步交给谁？」

---

# 路由 vs 转发，与转发一跳

**路由**（控制平面/画地图，慢没事）算路；**转发**（数据平面/开车，要极快）搬包

<v-click>

```text
① 收包       网卡从某入接口收到一个 IP 分组
② 读包头     取出目的 IP 地址
③ 查路由表   用目的 IP 匹配，最长前缀匹配选最优条目
④ 改 TTL     TTL 减 1；减到 0 则丢弃（防环兜底）
⑤ 转发       按「下一跳 + 出接口」送出，交给下一台路由器
```

</v-click>

<v-click>

> 路由表（RIB）编译出精简**转发表（FIB）**供硬件高速查表；**没人掌握完整路径**，逐跳接力送达全网。

</v-click>

<!--
路由 = 算路（慢没事，控制平面），转发 = 搬包（必须快，数据平面），RIB 生成 FIB。转发五步：收包→读目的 IP→查表 LPM→TTL−1→送下一跳。逐跳接力，没人知道全程。
-->

---

# 路由表里存了什么

又称路由信息库 **RIB**，最小三要素

<v-click>

| 字段 | 含义 |
| --- | --- |
| 目的网络 / 掩码 | 目的子网 + 掩码，合写即 `192.168.0.0/24` |
| 下一跳 / 网关 | 把包交给路径上下一台设备的地址 |
| 度量 metric | 该路由成本；多条路时**走度量最低的** |

</v-click>

<v-click>

条目按**来源**分三类：

- **直连路由**：接口配上 IP + 掩码后该网段**自动进表** —— 无需配置、最可信、度量最低
- **静态路由**：到远端网络，由**管理员手工配置**
- **动态路由**：到远端网络，由**路由协议自动学习**、随拓扑自动更新

</v-click>

<!--
路由表三要素：目的网络/掩码、下一跳（即网关）、度量。条目来源三类：直连（自动、最可信）、静态（手工）、动态（协议学习）。直连一配就有，远端才需静态/动态。
-->

---

# 最长前缀匹配（LPM）

多条都命中时，选**最具体**的那条

<v-click>

目的地址 `10.1.2.200`，三条都能命中：

```text
候选条目          下一跳     前缀长度   命中
10.0.0.0/8     →  R1         /8        命中（最粗）
10.1.0.0/16    →  R2         /16       命中
10.1.2.0/24    →  R3         /24       命中（最细）★ 选它
0.0.0.0/0      →  默认网关    /0        兜底，永不优先
```

</v-click>

<v-click>

- `0.0.0.0/0` 前缀长 0、最不具体 —— **「默认路由永远最后兜底」的数学原因**
- **默认网关** = 本网段出口，就对应这条默认路由：主机发包先 AND 判断同网段（直接交付）还是异段（全丢给默认网关，如家用 `192.168.0.1`）

</v-click>

<!--
LPM：前缀越长越优先，/24 比 /8 精确选 R3。0.0.0.0/0 前缀长 0、最不具体，永远垫底。默认网关就对应这条默认路由：发包先 AND 判同/异网段，异段全丢给默认网关。
-->

---

# 路由器 ≠ 调制解调器

家用「光猫一体机」往往集成两者，但职责不同

<v-click>

- **路由器（router）**：组建并管理网络内/网络间的数据流；跨**不同网络**转发分组
- **调制解调器（modem）**：把网络**接入互联网** —— 将 ISP 的信号转成数字信号
- **交换机（switch）**：在**同一网络内**的设备间转发（对比：路由器在不同网络间转发）

</v-click>

<v-click>

> 只有路由器没 modem：能组 LAN 互通但上不了网；只有 modem 没路由器：单机能上网但没法分享。

</v-click>

<!--
三者职责：路由器跨网转发、modem 接入互联网、交换机同网内转发。家用光猫一体机把路由器 + modem 揉一起，但概念上是两回事。
-->

---

# 静态路由 vs 动态路由

省算力 vs 自愈，按规模取舍

<v-click>

| 维度 | 静态路由 | 动态路由 |
| --- | --- | --- |
| 来源 | 管理员手工配置 | 路由协议自动学习 |
| 随拓扑 | 不变，需手工更新 | 自动更新、自愈 |
| 算力/带宽 | 几乎为零 | 需持续计算与交换报文 |
| 适用规模 | 小网、单出口 | 中大型、多路径 |
| 故障应对 | 链路断了需人工介入 | 自动绕开拥塞/故障 |

</v-click>

<v-click>

> 动态更费算力，所以**小网常用静态；中大型网络动态高效得多**。

</v-click>

<!--
静态：手工配、零开销、适合小网末端；动态：协议学、能自愈、适合中大型多路径网。取舍核心是「算力 vs 自愈能力」。
-->

---

# 三大动态路由协议

按作用范围分域内 / 域间

<v-click>

- **RIP**（距离矢量 / IGP）：用「跳数」找最短路，简单但跳数上限低、收敛慢
- **OSPF**（链路状态 / IGP）：路由器掌握全域拓扑，收敛快、可扩展，**企业内网主力**
- **BGP**（路径矢量 / EGP）：**自治系统（AS）之间**的域间路由，通告「谁掌管哪些 IP」，是互联网互通的「黏合剂」

</v-click>

<v-click>

> 一句话记：**RIP 数跳数、OSPF 看全图、BGP 连 AS。**

</v-click>

<!--
三协议一句话：RIP 数跳数（距离矢量，域内）、OSPF 看全图（链路状态，域内主力）、BGP 连 AS（域间，撑起整个互联网）。
-->

# ICMP 与 ping/traceroute

给「尽力而为」的 IP 当差错信使

---

# ICMP 是什么

网络层的「差错报告与控制协议」，封装在 IP 内（协议号 = **1**）

<v-click>

- IP 尽力而为、**不报告失败**；包被丢了（不可达/TTL 耗尽/缓冲区满）由 **ICMP** 回送源端
- 两大用途：① **差错报告**（主）② **网络诊断**（次，ping/traceroute 底层）
- **不是传输层协议**：架在 IP 之上、**无连接、无端口**（无法定位到具体进程）

</v-click>

<v-click>

```text
[ IP 首部 (Protocol=1) ][ ICMP: Type | Code | Checksum | 数据 ]
   差错报文的「数据」= 出错的原始 IP 首部 + 前 8 字节（让源端对号入座）
```

</v-click>

<!--
ICMP = IP 的差错信使，封装在 IP 里（协议号 1）。无连接、无端口。Type 定大类、Code 定细因。差错报文回带原始 IP 首部 + 前 8 字节，便于源端对号入座。
-->

---

# 常见 ICMP 报文类型

排障高频就这几类，记「类型号 + 场景」

<v-click>

| 报文 | Type | 触发场景 | 谁在用 |
| --- | --- | --- | --- |
| Echo Reply | 0 | 收到回显请求后的「我在」 | ping |
| Destination Unreachable | 3 | 到不了目标网络/主机/端口 | traceroute 收尾 |
| Redirect | 5 | 网关提示「有更优下一跳」 | 路由优化 |
| Echo Request | 8 | 主动探测「你在不在」 | ping |
| Time Exceeded | 11 | TTL 在途中减到 0 被丢 | traceroute |

</v-click>

<v-click>

> Redirect 今天少见：可被用于流量劫持，现代主机多默认**忽略 ICMP 重定向**。

</v-click>

<!--
四类高频：Echo（8/0，ping）、Destination Unreachable（3）、Time Exceeded（11，traceroute）、Redirect（5）。类型号要记牢。
-->

# ping：测「通不通、快不快、稳不稳」

连发 Echo Request、等 Echo Reply

<v-click>

```bash
$ ping -c 4 example.com
PING example.com (93.184.216.34): 56 data bytes
64 bytes from 93.184.216.34: icmp_seq=0 ttl=56 time=11.6 ms
64 bytes from 93.184.216.34: icmp_seq=1 ttl=56 time=10.9 ms
--- example.com ping statistics ---
4 packets transmitted, 4 packets received, 0.0% packet loss
round-trip min/avg/max/stddev = 10.7/11.1/11.6/0.35 ms
```

</v-click>

<v-click>

> 三件套：**丢包率**（稳定性，>1% 警觉）+ **RTT min/avg/max**（延迟）+ **stddev/mdev**（抖动）。

</v-click>

<!--
ping 连发 Echo 测连通。读三行汇总：丢包率看稳定性、RTT 看延迟、mdev/stddev 看抖动。每行 ttl=56 是回包剩余 TTL，可反推跳数。
-->

# ping 不通 ≠ 对方宕机

一个高频误判

<v-click>

- 出于抗攻击考虑，**大量服务器、云主机、CDN 默认丢弃 ICMP**，于是 `ping` 超时
- 这只能说明「**ICMP 这条路被堵了**」，不能据此判定主机已死

</v-click>

<v-click>

要确认服务存活，改测真实业务端口：

```bash
curl -I https://example.com     # 测 HTTP/HTTPS
nc -vz example.com 443          # 探 TCP 端口
```

</v-click>

<!--
经典误判：ping 不通就说主机挂了。其实很多主机默认丢 ICMP。要判存活得测真实端口：curl 测 HTTP、nc 探 TCP 443。
-->

# traceroute 原理

巧用 TTL：从 1 递增，每跳「自爆」暴露身份

<v-click>

```text
源 ─TTL=1─► R1(减到0,丢弃) ─► 回 Time Exceeded   ← 第1跳现身：R1
源 ─TTL=2────────► R2(减到0,丢弃) ─► 回 Time Exceeded ← 第2跳：R2
源 ─TTL=3──────────────► 目标主机(到达) ─► 回终点应答  ← 抵达，停止
```

</v-click>

<v-click>

- TTL 每经一路由器减 1、到 0 被丢，并回送 **Time Exceeded（type 11）**
- traceroute 故意用**递增 TTL**逐个「逼出」沿途路由器
- 每个 TTL 默认发 **3 个探测包** → 每跳显示 3 个时延

</v-click>

<!--
traceroute 反向利用 TTL：从 1 开始递增，每跳 TTL 归 0 触发 Time Exceeded，源端据此逐个点名沿途路由器。每跳发 3 个包测 3 个时延。
-->

---

# traceroute 平台差异

「TTL 递增」是共同骨架，**首发协议**因平台而异

<v-click>

| 维度 | Linux `traceroute` | Windows `tracert` |
| --- | --- | --- |
| 探测包协议 | **UDP**（端口 33434~33534） | **ICMP Echo Request** |
| 途中每跳 | 收 Time Exceeded | 收 Time Exceeded |
| 到达终点 | 收 **Destination Unreachable** | 收 **Echo Reply** |

</v-click>

<v-click>

```bash
$ traceroute example.com
 1  192.168.1.1    1.2 ms  1.1 ms  1.0 ms   # 家用网关
 2  100.64.0.1     8.5 ms  8.7 ms  8.6 ms   # 运营商接入
 3  * * *                                   # 此跳超时不回 ICMP（非中断）
 4  93.184.216.34  11.6 ms 11.2 ms 11.0 ms  # 抵达终点
```

</v-click>

<v-click>

> `*` = 该跳超时无应答（限速/防火墙），**非链路中断**，后续跳能显示即仍在前进；Linux 可 `-I`(ICMP)/`-T`(TCP) 切换。

</v-click>

<!--
平台差异：Linux 默认 UDP、Windows 默认 ICMP Echo，决定终点靠什么收尾。输出里 * 是某跳不回 ICMP，常见无害。Linux -I 切 ICMP、-T 切 TCP 穿防火墙最强。
-->

# ICMP 的滥用

便利反成 DDoS 温床 —— 解释「为何运维限制 ICMP」

<v-click>

- **Ping 洪水（Ping flood）**：海量 Echo Request 淹没目标，逼其逐个回应耗尽资源
- **Smurf 攻击**：发**伪造源 IP**（伪装受害者）的 ICMP 到广播地址，大量应答涌向受害者（反射放大）
- **死亡之 Ping（Ping of Death）**：发**超大畸形包**，目标重组分片时缓冲区溢出崩溃

</v-click>

<v-click>

> 后两者多已是「历史课」（仅威胁老旧设备），但 Ping 洪水仍现实存在 —— 这正是众多主机**默认丢弃/限速 ICMP** 的根本原因。

</v-click>

<!--
三类滥用：Ping 洪水（耗资源）、Smurf（反射放大）、Ping of Death（畸形包崩溃）。后两者基本是历史。这解释了为何主机默认限 ICMP，也回扣「ping 不通≠宕机」。
-->

# NAT 与 DHCP

私网共享公网 + 自动分配配置

---

# NAT 是什么：转换表是心脏

在路由设备**改写 IP 首部地址**，让整片私网共享少量公网 IP（缓解 IPv4 耗尽）

<v-click>

| 内网（原始源） | → 公网（转换后源） | 目的端 |
| --- | --- | --- |
| `192.168.1.10:5001` | `198.51.100.1:40001` | `203.0.113.80:80` |
| `192.168.1.11:5500` | `198.51.100.1:40002` | `203.0.113.80:80` |

</v-click>

<v-click>

- **出方向**：把源 IP（必要时连端口）换成公网地址，**新建表项**记录映射
- **回方向**：按数据包**目的端口查表**，把目的 IP/端口**替换回**内网值投递

</v-click>

<!--
NAT 改写 IP 首部地址、让整片私网共享少量公网 IP，是 IPv4 续命关键。心脏是转换表：出方向改源 + 记表项，回方向按目的端口查表还原回内网。类比：分机号 vs 总机号，NAT 像前台。
-->

---

# PAT / NAPT：多设备共享一个公网 IP

家用路由器的默认模式（又称 NAT overload）

<v-click>

在基础 NAT 上**同时改写 IP + 端口**，靠**端口号区分**不同内网设备：

| 内网设备 | 转换后（共享同一公网 IP，端口不同） |
| --- | --- |
| `192.168.1.10:5001` | `198.51.100.1:`**`40001`** |
| `192.168.1.11:5001` | `198.51.100.1:`**`40002`** |
| `192.168.1.12:6000` | `198.51.100.1:`**`40003`** |

</v-click>

<v-click>

- 即便两台设备碰巧用相同内网端口（都 5001），NAT 也分配**不同公网端口**，回包不会送错
- 但端口是 **16 位**：一个公网 IP 约 **64000** 并发上限，海量并发会**端口耗尽**（推进 IPv6 的现实动力）

</v-click>

<!--
PAT/NAPT 同时改 IP + 端口，多设备共用一个公网 IP，靠公网端口区分。即便内网端口撞了公网端口也不同。但端口 16 位、约 64000 上限，大出口会端口耗尽。
-->

---

# SNAT 与 DNAT

方向不同，改的地址也不同

<v-click>

| 类型 | 改写对象 | 方向 | 典型用途 |
| --- | --- | --- | --- |
| **SNAT**（源 NAT） | 数据包的**源**地址 | 内网 → 公网（出门） | 内网共享公网 IP 上网 |
| **DNAT**（目的 NAT） | 数据包的**目的**地址 | 公网 → 内网（进门） | **端口转发**、对外发布服务 |

</v-click>

<v-click>

- 刷网页是 **SNAT**（外界只看到公网 IP）
- 让外部访问内网服务器要 **DNAT 端口转发**：把「公网 IP:8080」改写为「`192.168.1.100:80`」转发进内网
- 没有 DNAT，外部无法主动触达内网设备

</v-click>

<!--
SNAT 改源（出门上网），DNAT 改目的（进门访问内网服务，即端口转发）。PAT 是一种 SNAT。对外发布服务必须靠 DNAT。
-->

---

# NAT 对 P2P 的影响

转换表项默认只能**出方向发起**时建立

<v-click>

- 外部主机**无法直接主动连接**内网设备（公网侧没有指向它的表项）
- 两台分处各自 NAT 后的设备（如两浏览器做 WebRTC）彼此都连不上 → 需 **NAT 穿透**

</v-click>

<v-click>

- **STUN**：让客户端**得知自己经 NAT 后的公网 IP:端口**，再尝试「打洞」直连
- **TURN**：打洞失败时（典型是对称型 NAT），用**中继服务器转发**全部流量
- **ICE**：统一收集 STUN/TURN/本地候选，**自动挑出能连通的最佳路径**（WebRTC 用它）

</v-click>

<!--
NAT 副作用：表项只能出方向建，外部主动连不进来，P2P 卡壳。穿透三件套：STUN 探公网地址打洞、TURN 中继兜底、ICE 统一调度。WebRTC 用 ICE。
-->

---

# NAT 类型与穿透难度

不同类型对穿透影响很大

<v-click>

| NAT 类型 | 行为 | 穿透难度 |
| --- | --- | --- |
| 全锥（Full Cone） | 映射后任意外部主机可连入 | 最易 |
| 受限锥（Restricted） | 仅曾发过包的**目标 IP** 可回连 | 中 |
| 端口受限锥（Port-Restricted） | 仅曾通信的 **IP:端口** 可回连 | 较难 |
| **对称（Symmetric）** | 对不同目的用**不同公网端口**映射 | 最难，常须 TURN |

</v-click>

<v-click>

> 对称型 NAT 最难穿透 —— 打洞基本失败，常只能退回 **TURN 中继**。

</v-click>

<!--
四种 NAT 类型，穿透难度递增：全锥最易→对称最难。对称型对不同目的分不同端口，打洞预测不出端口，基本只能靠 TURN 中继。
-->

---

# DHCP 与 DORA 四步流程

**DHCP** 自动下发 **IP / 子网掩码 / 默认网关 / DNS**，免去逐台手配

<v-click>

```text
客户端(UDP 68，暂无 IP)                  DHCP 服务器(UDP 67)
  ① ──DHCPDISCOVER（广播）──────────────▶  「有人在吗？给我个 IP」
  ② ◀─DHCPOFFER（候选 IP/掩码/租期）─────   「这个 IP 给你用？」
  ③ ──DHCPREQUEST（广播，选定该 IP）─────▶  「我要这个，确认下」
  ④ ◀─DHCPACK（确认，配置完成）──────────   「成交，租期内归你用」
```

</v-click>

<v-click>

> 服务器监听 **UDP 67**、客户端用 **UDP 68**；Discover 阶段客户端**还没 IP**，全靠**广播**通信。

</v-click>

<!--
DHCP 自动下发四参数（IP/掩码/网关/DNS）。DORA：Discover（广播找服务器）→ Offer（报候选 IP）→ Request（选定广播请求）→ Ack（确认）。服务器 67、客户端 68，没 IP 全靠广播。
-->

---

# DHCP 租约与续租

分配的 IP 带**租期**，不是永久的

<v-click>

- **T1（约租期 50%）**：客户端向**原服务器单播** `DHCPREQUEST` 续租，正常回 `DHCPACK` 顺延
- **T2（约租期 87.5%）**：T1 没成功（原服务器失联），改为**广播** `DHCPREQUEST` 向任意服务器求租
- 彻底耗尽仍未续上 → 放弃地址，重走完整 DORA

</v-click>

<v-click>

> 跨网段靠 **DHCP 中继代理**：广播不跨网段，中继把客户端广播**单播**转发给跨网段服务器。

</v-click>

<!--
租约两道闸：T1（50%）单播向原服务器续、T2（87.5%）广播向任意服务器续，都失败就重走 DORA。广播不跨网段，跨段靠中继代理转发。
-->

# 小结

一个数据包，六块拼图

<v-click>

- **IP / IPv4**：网络层核心，无连接不可靠；32 位地址 = 网络号 + 主机号，私有地址靠 NAT 出网
- **子网 / CIDR**：掩码定分界、AND 算网络、借位做子网、聚合省路由，可用主机 `2^h − 2`
- **IPv6**：128 位、零压缩（`::` 仅一次）、`/64`、端到端无 NAT、废弃广播改组播
- **路由 / ICMP / NAT-DHCP**：最长前缀匹配 + 默认网关兜底；ping 测连通、traceroute 探路；PAT 共享公网、DORA 自动配

</v-click>

<v-click>

> 跟着一个包走完寻址→划网→选路→诊断→出网，网络层的全貌就在手里了。

</v-click>
