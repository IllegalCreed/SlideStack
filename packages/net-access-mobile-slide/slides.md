# 这一章讲什么

一条主线：**比特如何到达用户，又如何更快到达**

<v-click>

- **接入网 LAN/WAN**：网络按规模分类、接入网=最后一公里、ISP 分层
- **宽带接入**：拨号→DSL→光纤 FTTx/PON，同轴、以太网、卫星
- **蜂窝 2G→5G**：小区/频率复用/切换、四代演进、5G 三场景
- **移动弱网**：高延迟抖动丢包、RRC 状态机、按流量计费
- **CDN 网络原理**：边缘 PoP、缓存命中/回源、Anycast 调度
- **网络性能**：四大指标、延迟拆解、弱网优化清单

</v-click>

<v-click>

> 弄懂"起跑线在哪、路上有多堵"，前端优化才知道该使劲在哪。

</v-click>

# 接入网与 LAN/WAN

网络按规模分层，谁连谁

---

# 按规模分类：人—楼—城—天下

规模是最直观的分类维度

<v-click>

| 类型 | 全称 | 典型范围 | 场景 / 技术 |
| --- | --- | --- | --- |
| PAN | Personal Area Network | ≤ 约 10 米 | 蓝牙耳机、NFC、键鼠 |
| LAN | Local Area Network | 一栋楼 / 校园 | 家庭 Wi-Fi、办公网 |
| MAN | Metropolitan Area Network | 一座城市 | 城市光纤环网 |
| WAN | Wide Area Network | 跨地域 / 跨国 | 企业专线、**互联网** |

</v-click>

<v-click>

> 口诀「人—楼—城—天下」：**互联网是最大的 WAN**；中间还有 CAN（校园网）这一档。

</v-click>

<!--
主干记四档即可，纳米网络、BAN 这些极少用到。关键认知：互联网就是最大的那个 WAN，WAN 本质是"许多 LAN 连起来"。
-->

---

# LAN 与 WAN

一栋楼里的网，和把许多网连起来的网

<v-click>

- **LAN**：限定在小地理范围（通常同一栋楼），**几乎总用以太网 + Wi-Fi**
- LAN 的**汇聚点是路由器**——多数 LAN 在路由器这个中心点接入互联网
- **WAN**：「一堆相连的 LAN 的集合」，大小不限、可达上万公里
- WAN 靠**租用线路、MPLS、分组/电路交换、SD-WAN** 等跨距离手段

</v-click>

<v-click>

> LAN 设备本地自管；WAN 的实现技术与 LAN 截然不同，专为"跨距离"设计。

</v-click>

<!--
LAN 和 WAN 不是规模上的同一种东西放大，连接技术也不同。VLAN 是用软件把一张物理网切成多张逻辑网，和"按规模分类"无关。
-->

---

# 接入网：最后一公里

把终端用户连到其直属服务商的那一段

<v-click>

- **接入网（access network）**：从你家光猫到 ISP 机房的那截网络，俗称**「最后一公里」**
- 可再分两层：**馈线/分布网**（主干段）+ **引入/边缘网**（入户末端）
- **核心网 vs 接入网**：核心网管「服务商↔服务商」（骨干长途），接入网只管「用户↔服务商」

</v-click>

<v-click>

> 前端能感知的弱网、首字节延迟，**绝大多数发生在接入网这一段**——它带宽最受限、最易拥塞。

</v-click>

<!--
打个比方：接入网是从你家到本地邮局的小路，核心网是邮局之间的高速公路。接入段的物理时延无法靠代码消除，只能靠 CDN 缩短后续路径。
-->

---

# 为什么前端要懂接入网

它是前端性能的**物理天花板**

<v-click>

- **带宽不对称**：家用普遍**下行远大于上行**（如 1000M 下 / 30M 上），上传/推流时上行先成瓶颈
- **共享与拥塞**：同楼/同小区共享上游容量，晚高峰拥塞直接抬高 RTT 与丢包
- **首字节延迟**：请求要先穿过最后一公里才进骨干，**这段物理时延砍不掉**

</v-click>

<v-click>

> 前端能优化"请求发出后怎么走得更短"；**接入网决定了起跑线在哪**——知道哪些慢是代码能救的。

</v-click>

<!--
接入网看似离前端很远，实则决定了体验的下限。做上传组件、排查"网页突然变慢"时，要对这段的物理约束有预期。
-->

---

# 互联网 = 网络的网络

LAN 接入 AS，AS 层层互联成全球网

<v-click>

```text
   [ 你家 LAN ]─接入网─[ 本地 ISP ]
                          │ 转接 transit（付费）
                     [ 二级 ISP ]
                          │ 转接 transit（付费）
   [ 一级 ISP ]══对等 peering══[ 一级 ISP ]  ← 全球骨干互通
```

</v-click>

<v-click>

- **AS（自治系统）**：拥有独立路由策略和一批 IP 的大网——**「一个 ISP 就是一个 AS 的例子」**
- 你家小 LAN 接入一个 AS，AS 再与其他大 AS 互联，层层嵌套连通全球

</v-click>

<!--
AS 是路由层面的技术身份，ISP 是它的商业角色。理解这张图，才明白请求要跨多少张网才能到达内容方。
-->

---

# ISP 分层与互联

三层 ISP，两种互联姿势

<v-click>

| 层级 | 角色 | 与其他网的关系 |
| --- | --- | --- |
| 一级（Tier-1） | 全球骨干 | 彼此**对等**，到达全网免付费 |
| 二级（区域） | 区域/国家级 | 向 Tier-1 买**转接**，同级可对等 |
| 本地（接入） | 接到你家那家 | 买转接，经营接入网最后一公里 |

</v-click>

<v-click>

- **对等（peering）**：平等两网**直接互换各自客户流量**，通常不计费
- **转接（transit）**：向上游**付费**，换取「通往整个互联网」的可达性

</v-click>

<!--
每一次跨网（尤其对等/转接的交界点）都可能成为延迟与丢包来源。这正是 CDN 把内容推到"离用户最近的网络边缘"的根本动因。
-->

# 宽带接入技术

从拨号窄带到现代光纤

---

# 从窄带拨号说起

「宽带」是相对早年**窄带拨号**而言的

<v-click>

- **拨号 dial-up**：用 **Modem（调制解调器）** 把数字信号调制进电话语音频段
- 两个致命限制：理论上限仅 **56 kbit/s**、**占电话线**（上网不能打电话）
- **宽带（broadband）** 的本质特征：**永远在线、不占语音、带宽远高于拨号**

</v-click>

<v-click>

> 下文 DSL、光纤、同轴、以太网、卫星，都是为突破拨号窄带瓶颈而生的不同技术路线。

</v-click>

<!--
那段经典的"滋滋"握手声就是调制解调的声音。宽带没有全球统一速率门槛，各国逐年上调，但"在线/高带宽/不占语音"这三点是稳定的。
-->

---

# DSL/ADSL：旧电话线挖高频段

复用电话铜线，但用语音之外的高频段传数据

<v-click>

- **DSL**：复用已入户的普通电话铜线，用**语音用不到的高频段**传数字信号
- 用户端装**分离器**分开语音与数据 → **上网打电话两不误**
- **ADSL** 的 **A = Asymmetric（非对称）**：**下行远大于上行**

</v-click>

<v-click>

> 家用是"重下载、轻上传"——ADSL 顺势把更多频谱给下行（如下行 20M / 上行 1M）。这就是家用宽带"下载快上传慢"的由来。

</v-click>

<!--
更高速的 VDSL2 短距（约 300m 内）可上下行各 100M，但距离越远衰减越快，所以常配"光纤到路边 + 铜线最后几百米"的 FTTC/FTTN。
-->

---

# 光纤接入 FTTx：光铺到哪，x 就是哪

光越靠近用户，速率越高

<v-click>

| 缩写 | 光纤铺到哪 | 最后一段 | 定位 |
| --- | --- | --- | --- |
| **FTTH** | 进到家门口/入户盒 | 光直达，几无铜线 | **当前主流** 1~10G |
| **FTTB** | 楼栋设备间 | 楼内走网线/同轴 | 公寓、写字楼 |
| **FTTC** | 路边机柜 <300m | DSL 铜线最后一段 | 光改过渡 |
| **FTTN** | 街区节点 | 铜线/同轴 | 大范围、速率低 |

</v-click>

<v-click>

> **FTTP（到驻地）** 是 FTTH + FTTB 的上位统称——光纤进到用户产权边界即算。

</v-click>

<!--
FTTH 是当前主流，上下行可对称。FTTC/FTTN 是光纤改造的过渡阶段，光拉到路边后最后几百米仍跑 DSL。
-->

---

# PON：无源光网络

FTTH 的主流承载架构，「无源」是精髓

<v-click>

- **PON（Passive Optical Network）**：**点到多点**，一根光纤经分光器分给多户
- 三个角色：**OLT（局端）↔ 分光器（无需供电）↔ ONU/ONT（家里光猫）**
- 分光器把一路光按 **1:32 / 1:64** 分成多路，**无源 = 不耗电、不维护**
- 多户共享上行：**TDMA + 动态带宽分配**错峰上传；下行 OLT **广播 + 加密**

</v-click>

<v-click>

> 对比 **AON（有源光网络）**：每户独享、距离远，但设备需供电、能耗成本高，非家用主流。

</v-click>

<!--
"无源"二字省掉了局端到用户间所有需供电和维护的有源设备，这是 PON 能大规模、低成本铺开的关键。下行广播+加密，各 ONU 只取属于自己的帧。
-->

---

# PON 速率演进

从 GPON 到对称 10G

<v-click>

| PON 标准 | 下行 | 上行 | 2026 状态 |
| --- | --- | --- | --- |
| **GPON** | 2.5G | 1.25G | **存量最大**的一代 |
| XG-PON | 10G | 2.5G | 与 GPON 共存的 10G 过渡 |
| **XGS-PON** | **10G** | **10G（对称）** | **当前新建主力** |
| NG-PON2 | ~80G | ~80G | 多波长，高端/商用 |
| 50G-PON | 50G+ | 50G+ | 已进入商用试点 |

</v-click>

<v-click>

> 新建普遍转向**对称 10G 的 XGS-PON**，补齐 GPON 上行偏弱的短板——对直播、云盘、远程办公更友好。

</v-click>

<!--
对前端：用户侧最后一公里的上行能力正快速提升，上传瓶颈逐步从接入网转移到其他环节。50G-PON/NG-PON2 已有现网演示但尚未普及。
-->

---

# 同轴 / 以太网 / 卫星

光纤之外的三条接入路线

<v-click>

- **同轴 Cable（HFC）**：有线电视网改造，**光纤到小区 + 同轴入户**，跑 **DOCSIS**；下行可达 1G 但**上行明显偏弱**
- **以太网接入（FTTB + LAN）**：光纤进楼后**双绞线入户**，复用我们熟悉的以太网，国内常见形态
- **卫星宽带**：光纤铜线铺不到的兜底——**轨道高度决定延迟**

</v-click>

<v-click>

| 卫星类型 | 轨道高度 | 单程延迟 | 适用 |
| --- | --- | --- | --- |
| **GEO** 同步轨道 | ~36000 km | 0.5~0.7 秒 | 覆盖广，不适合实时 |
| **LEO** 低轨 | ~300~1200 km | 20~50 ms | Starlink，接近地面 |

</v-click>

<!--
LEO 低轨星座（Starlink/OneWeb）把卫星拉到几百公里，延迟降两个数量级，第一次能跑实时应用，代价是需成千上万颗卫星组网并随过境切换链路。这是"弱网用户版图扩大"的信号。
-->

---

# 上行 vs 下行：理解非对称

搞混会误判瓶颈

<v-click>

- **下行 / 下载**：数据**从服务器流向你**——刷网页、看视频、下文件
- **上行 / 上传**：数据**从你流向服务器**——传图片/视频、直播推流、云备份
- **非对称**（下行 ≫ 上行）：ADSL、Cable、GPON——家用常态
- **对称**（上行 = 下行）：XGS-PON、企业专线——推流/会议/备份友好

</v-click>

<v-click>

> 「100M 宽带」通常**只指下行峰值**。做上传重的应用（直播、网盘、CI 推产物），真正卡的往往是**上行**。

</v-click>

<!--
排查上传慢、推流卡时，先看用户接入的上行能力，而不是只盯下行数字。非对称宽带的上行可能只有下行的 1/10。
-->

---

# 带宽单位：bit 还是 Byte，差 8 倍

前端最容易踩、用户最容易误会的坑

<v-click>

- **网速 / 带宽**用 **bit/s（小写 b）**：`Mbps`、`Gbps`
- **文件大小 / 下载速度**用 **Byte（大写 B）**：`MB`、`MB/s`
- 换算：**1 Byte = 8 bit**

</v-click>

<v-click>

```text
100 Mbps（兆比特/秒）÷ 8 = 12.5 MB/s（兆字节/秒）
```

> 「100 Mbps 下载怎么才 12.5 MB/s」不是缩水，是单位换算。前端展示网速/进度时，**务必标明单位是 b 还是 B**。

</v-click>

<!--
运营商和网络层一律用 bit 标速率，操作系统和下载器显示 Byte。再扣掉协议开销，100 Mbps 实测 10~11 MB/s 也正常。
-->

# 蜂窝移动网络 2G→5G

手机如何随走随连地接入

---

# 为什么叫「蜂窝」

频谱有限、用户多且在移动 → 分而治之

<v-click>

- 难题：若一座大塔覆盖全城，全城用户挤同一份频谱，容量很快撑满
- 解法：把地面划成许多**小区（cell）**，每个小区由一座**基站**用较小功率覆盖
- 理想化下小区排成**正六边形**（蜂窝得名）——能无缝拼满平面、最接近圆形覆盖

</v-click>

<v-click>

> 蜂窝网络 = 「端节点链路为无线、网络分布在称为**小区**的多个地面区域上的电信网络」。

</v-click>

<!--
这是所有移动通信（2G~5G）的共同底层结构。把"一座大塔"换成"许多小基站"，是整个蜂窝思想的起点。
-->

---

# 频率复用：有限频谱反复用

蜂窝最核心的价值

<v-click>

```text
   小区用不同频率 f1~f7 着色，相隔够远即可复用：

      ⬡ f2   ⬡ f3
   ⬡ f7   ⬡ f1   ⬡ f4     ← 中心 f1
      ⬡ f6   ⬡ f5
         ⬡ f1  ← 远处另一个 f1，与中心互不干扰
```

</v-click>

<v-click>

- **相邻小区必须用不同频率**（否则边界信号叠加成干扰）
- **相隔够远的小区可复用同一频率**——同一段频谱在城市里被许多小区同时复用
- 网络总容量随小区数量增长，常见复用因子 1/3、1/4、1/7

</v-click>

<!--
扩容往往不是加大功率，而是把小区拆得更小、加更多基站（宏→微→微微）。小区越密，同频复用次数越多，单位面积承载用户越多——这是 5G 闹市铺小基站的底层逻辑。
-->

---

# 小区分级与切换

越小越密，移动中不掉线

<v-click>

| 小区类型 | 覆盖半径 | 典型场景 |
| --- | --- | --- |
| 宏蜂窝 Macrocell | 城区约 0.8km，乡村达 8km | 广域基础覆盖 |
| 微蜂窝 Microcell | < 2 km | 城区热点、街道 |
| 微微蜂窝 Picocell | < 200 m | 商场、机场、室内 |
| 毫微蜂窝 Femtocell | ~ 10 m | 家庭/小办公室补盲 |

</v-click>

<v-click>

- **切换（handover）**：用户跨小区移动时，**电子化自动换到新基站频率而不中断**
- CDMA 支持**软切换**：先同时连多个基站、确认稳定后再断旧的（避免"先断后连"）

</v-click>

<!--
"边走边打电话不掉线"靠的就是切换。手机持续测量两侧信号，网络在合适时机把连接"交棒"给新基站。
-->

---

# 从 2G 到 5G：四代演进

每代对应一次"能干什么"的跃迁

<v-click>

| 代际 | 代表标准 | 交换 | 典型速率 | 能做什么 |
| --- | --- | --- | --- | --- |
| **2G** | GSM/GPRS | 电路+分组 | 几十~几百 kbps | 数字语音、短信、窄带 |
| **3G** | WCDMA/CDMA2000 | 电路+分组 | 数 Mbps | 移动互联网：网页、邮件 |
| **4G** | LTE | **全 IP** | 数十~数百 Mbps | 流媒体、App、VoLTE |
| **5G** | NR（ITU-2020） | 全 IP+切片 | 峰值 20 Gbps | 三场景，详见下文 |

</v-click>

<v-click>

> **2G** 数字化+短信 → **3G** 移动互联网 → **4G** 全 IP 宽带（连语音都走 VoLTE）→ **5G** 三场景分化。

</v-click>

<!--
2G 把语音从模拟变数字、催生短信；3G 第一次让手机真能上网；4G 彻底全 IP、纯分组，支撑移动视频/支付爆发；5G 不只更快。
-->

# 5G 三大场景

ITU-R 把 5G 能力拆成三类，面向完全不同需求

<v-click>

```text
              ┌──────── 5G ────────┐
        ┌─────────┼─────────────┬───────────┐
        ▼         ▼             ▼
     eMBB      URLLC          mMTC
   增强移动   超可靠低时延    海量机器类
     宽带                      通信
```

</v-click>

<v-click>

| 场景 | 核心诉求 | 典型应用 |
| --- | --- | --- |
| **eMBB** | 极高速率 + 大容量 | 4K/8K、AR/VR、密集人群 |
| **URLLC** | 毫秒级时延 + 极高可靠 | 工业控制、远程手术、车联网 |
| **mMTC** | 海量连接 + 低功耗 | 传感器、智能抄表、IoT |

</v-click>

<!--
URLLC 目标端到端 <1ms 配 99.9999% 可用性，对比 4G LTE 实测中位 33~75ms——这就是远程手术、工业实时控制必须等 5G URLLC 才能落地的原因。
-->

# 5G 频谱与关键技术

速度与覆盖此消彼长

<v-click>

| 频段 | 范围 | 速率 | 特点 |
| --- | --- | --- | --- |
| **FR1（Sub-6 GHz）** | 410M~7.125G | 下行约 5~900 Mbps | **覆盖广、穿透好**，主力 |
| **FR2（毫米波）** | 24~71 GHz | 可达多 Gbps | **极高速但传得近**，易被阻挡 |

</v-click>

<v-click>

- **Massive MIMO**：大型天线阵列同时服务多用户、定向降干扰
- **波束赋形**：把能量定向射向特定用户（毫米波可用的前提）
- **网络切片**：同一物理网切出多个逻辑专网，三场景各享保障
- **SA / NSA**：NSA 复用 4G 核心网过渡；**SA 配 5GC** 才完整释放 URLLC/切片

</v-click>

<!--
毫米波能跑多吉比特但只能传很短距离、易被墙体/雨/植被阻挡，多用于体育场、车站等高密度热点；广域覆盖仍靠 Sub-6 GHz。
-->

# 核心网与 SIM/eSIM

基站之后的"大脑"，与入网身份

<v-click>

- **核心网**：基站背后，传统分**电路交换**（语音/短信）与**分组交换**（数据）
- 4G/5G **全 IP 化**，电路域退场，核心网演进为 **EPC（4G）/ 5GC（5G）**
- 5GC 云原生、服务化，配合网络切片为不同业务分配资源

</v-click>

<v-click>

- **SIM 卡**：存储用户标识（IMSI）与密钥，向运营商**鉴权**「我是合法用户」
- **eSIM（嵌入式 SIM）**：焊在设备里、可**远程写号（OTA）**——免插卡、可多号切换

</v-click>

<!--
手机连上基站只是第一跳。传统经基站接到 MSC，MSC 再连公共电话网。eSIM 广泛用于可穿戴、IoT 与新款手机。
-->

# 移动弱网对前端的挑战

不是网速慢一点，而是另一个世界

# 移动网络 ≠ 缩水版有线

二者的差异是**质**的，不是量的

<v-click>

| 维度 | 有线 / Wi-Fi | 蜂窝移动网络 |
| --- | --- | --- |
| 延迟 RTT | 低且**稳定** | **高且抖动大** |
| 丢包 | 极少 | **明显**，随信号波动 |
| 带宽 | 基本恒定 | **剧烈波动** |
| 连接连续性 | 插着就在 | **会断**：电梯/隧道/切换 |
| IP 地址 | 长期稳定 | **易变**：NAT 共享、移动重分配 |

</v-click>

<v-click>

> 别用「我办公室 Wi-Fi 很快」验收——真实用户可能在地铁、电梯、地库、人挤人的场景打开你的页面。

</v-click>

<!--
有线链路独占、稳定、延迟可预测；移动链路是共享的无线电波，受距离、遮挡、基站负载、用户移动共同影响，瞬时质量剧烈起伏。
-->

---

# 弱网落到前端就是这些 Bug

高 RTT、丢包、断连的直接后果

<v-click>

- **首屏慢**：高 RTT 放大每次 DNS / TLS 握手 / 首字节（TTFB），串行依赖越多越糟
- **请求超时与失败**：丢包触发 TCP 重传，固定短超时会误杀本可成功的慢请求
- **长连接易断**：WebSocket/SSE 在切换、进隧道时被掐断，没心跳+重连就"假死"
- **重连频繁**：跨基站、Wi-Fi↔蜂窝切换让 IP 变化、连接重置

</v-click>

<v-click>

> 真实数值参考（非峰值）：2G 延迟 300–1000 ms、3G 100–500 ms、4G LTE <100 ms。

</v-click>

<!--
表现为"转圈半天后报错"。前端要把"重连"当常态而非异常来设计：心跳探活 + 自动重连 + 退避。
-->

# RRC 状态机：移动端隐藏成本

射频不是"一直开着随时能发"

<v-click>

| RRC 状态 | 射频功耗 | 说明 |
| --- | --- | --- |
| **Idle（空闲）** | **< 15 mW** | 只监听控制信道，不能直接发数据 |
| **Connected（激活）** | **1000–3500 mW** | 高功耗态，正在收发或等待数据 |

</v-click>

<v-click>

- **唤醒延迟**：从 Idle 切到 Connected 要信令握手——LTE <100 ms、HSPA+ 约 2 s
- **尾延迟耗电**：数据发完后射频按**惰性计时器**在高功耗态再赖几秒，**纯烧电**

</v-click>

<!--
你的第一个请求要先等几十到几百毫秒把射频"叫醒"，这段延迟叠加在 DNS/握手/TTFB 之上。尾延迟那几秒不传任何字节，纯粹烧电。
-->

---

# 「频繁小请求」是隐形杀手

每个小请求 = 一次唤醒 + 一段尾延迟

<v-click>

- 每个零散小请求都可能触发一次完整 **Idle→Connected 唤醒 + 尾延迟（数秒高功耗）**
- 极端失衡案例 **Pandora**：心跳信标只占 **0.2% 流量**，却消耗 **46% 电量**

</v-click>

<v-click>

> 启示：**别用 setInterval 每隔几秒打小埋点/轮询/心跳**。能合并就合并、能批量就批量——把请求**聚成几次大的**，远比摊成几十次小的省电省延迟。

</v-click>

<!--
字节数极小但耗电占比极高，这是移动端独有的失衡。能等用户操作再发就别主动发。
-->

---

# IP 易变与流量成本

把"断线"当常态，尊重"省流量"

<v-click>

- **IP 易变**：NAT 后共享公网 IP、移动中重新分配——**别把会话/鉴权绑死在 IP**
- 长连接要**心跳探活 + 指数退避重连 + 请求幂等**，让重写操作能安全重试不重复
- **流量成本**：很多用户按流量计费——`navigator.connection.saveData` 是用户的明确意愿

</v-click>

<v-click>

> 检测到 `saveData` 就**主动降级**：降清晰度、停自动播放、减预取。Twitter Data Saver 移动端省约 50% 图片流量、Web 端约 80%。

</v-click>

<!--
IP 白名单、基于 IP 的限流在移动端都会误判。saveData 应当被当作明确的用户契约。Tinder 在慢网/省流量时关视频自动播放、限制预取。
-->

# 感知网络：navigator.connection

Network Information API（渐进增强）

<v-click>

```js
// 读取当前网络估算，慢网/省流量则不预加载大体积视频
const conn = navigator.connection;
if (conn) {
  let preloadVideo = true;
  if (conn.effectiveType === "2g" || conn.saveData) {
    preloadVideo = false;
  }
  // 网络质量变化时重新决策
  conn.addEventListener("change", () => {
    console.log(`变化：${conn.effectiveType}, rtt=${conn.rtt}ms`);
  });
}
```

</v-click>

<v-click>

> 可读 `effectiveType` / `saveData` / `rtt` / `downlink`。但它处于 **Limited availability**，**Firefox / Safari 不支持**——只能当渐进增强信号。

</v-click>

<!--
effectiveType 是浏览器对近期吞吐/RTT 的估算分级，不是实时测速，也不保证下一秒不变。必须做存在性判断，缺失时走"默认不降级"安全分支。

# CDN 网络原理

把内容推到离用户最近的地方

---

# CDN 是什么

距离即延迟，CDN 缩短传输距离

<v-click>

- **CDN（内容分发网络）**：一组**地理上分散**的服务器，把内容**缓存到离用户更近**的地方
- 核心痛点 **= 距离即延迟**：用户离源站越远，物理 RTT 越大，页面越慢
- 思路：**不让用户连远端源站，而是连一个地理上更近的数据中心**

</v-click>

<v-click>

> **CDN ≠ 虚拟主机**：Web Host 存放原始文件，CDN 只是把它们缓存得离用户更近。CDN 是源站之上的**加速 + 防护层**，不能替代源站。

</v-click>

<!--
今天 Facebook、Netflix、Amazon 等绝大多数大流量站点都经由 CDN。源站依旧不可或缺：登录校验所需的密码哈希库等逻辑通常仍在源站。
-->

---

# 边缘 PoP 与缓存机制

内容的全球前哨

<v-click>

```text
   没有 CDN：所有人都连那一个源站
   用户(亚/欧/美) ───远───▶ [ 源站 ]  距离远=RTT 大=慢

   有 CDN：就近连边缘 PoP，仅未命中才回源
   用户(亚) ─近─▶ [PoP·亚] ┐
   用户(欧) ─近─▶ [PoP·欧] ┼─仅 miss 回源─▶ [ 源站 ]
   用户(美) ─近─▶ [PoP·美] ┘
```

</v-click>

<v-click>

- **PoP / 边缘服务器**：部署在 **IXP（网络交换点）** 等枢纽上、离用户最近的缓存集群
- **边缘 vs 源站**：边缘缓存并就近交付**静态资源**；**源站**保存原始文件、处理**动态请求**

</v-click>

<!--
IXP 是不同网络运营商互联、交换流量的高速汇聚点。一组边缘服务器通常合称一个 PoP。
-->

---

# 缓存命中 / 未命中与回源

一次请求落到边缘只有两种结局

<v-click>

| 结果 | 含义 | 行为 |
| --- | --- | --- |
| **命中 hit** | 资源已在该边缘缓存 | **直接就近返回**，零回源、零源站带宽 |
| **未命中 miss** | 没缓存/已过期/不可缓存 | **回源**取回 → 缓存一份 → 再返回 |

</v-click>

<v-click>

- 登录页例子：HTML/CSS/图片这类**对所有访客相同的静态文件**可被边缘缓存
- 点「登录」提交账密这类**动态请求**则被代理回源站，查库校验后返回专属数据
- **命中率越高**，被边缘挡下、无需回源的请求越多——**省带宽、降成本**的根本机制

</v-click>

<!--
源站每响应一次请求都消耗带宽。带指纹的 JS/CSS、图片天然适合长期缓存，是命中率的主力。
-->

---

# 就近调度：DNS 调度 vs Anycast

让用户连最近节点的两条路线

<v-click>

| 维度 | DNS 调度 | Anycast |
| --- | --- | --- |
| 调度时机 | **解析域名时**返回最近 IP | **路由转发时**自动选最近 |
| IP 模型 | 不同节点**不同 IP** | 多节点**共享同一 IP** |
| 选路依据 | 智能 DNS 按地理位置 | BGP 按拓扑最短路径 |
| 故障切换 | 改 DNS（受 TTL 延迟） | **路由自动绕开**，近实时 |
| 抗 DDoS | 一般 | **强**：摊到全网吸收 |

</v-click>

<v-click>

> 实务中两者常**叠加**：先用 DNS 把用户引向某区域集群，再在集群内用 Anycast 做最后一跳就近与容灾。

</v-click>

<!--
DNS 调度门槛低，但看到的常是递归 DNS（如 8.8.8.8）的位置而非用户本身，TTL 缓存也让切换不实时。
-->

---

# Anycast vs Unicast

抗灾与抗 DDoS 的分水岭

<v-click>

- **Unicast**：每个节点一个**唯一 IP**，流量**直达**单点（家用、办公网都是）
- 用 Unicast 跑 CDN，流量直冲某机房——遇 DDoS 可能被打爆，拒绝服务
- **Anycast**：「**一对多中之一**」，多个数据中心**共享同一 IP**，BGP 送到最短路径节点

</v-click>

<v-click>

> Anycast 让网络极具弹性：**一整个数据中心下线，流量自动流向邻近节点**；攻击流量被**分摊到全网吸收**，没有单点被压垮。CDN 越大越难被有效 DDoS。

</v-click>

<!--
这是 Cloudflare "用 Anycast 做负载均衡、无需负载均衡器"的底层逻辑。家用 Unicast 才会有"IP 地址冲突"。
-->

---

# CDN 对前端的四大价值

加速、抗峰值、抗攻击、省成本

<v-click>

- **加速静态资源**：就近交付 + **压缩/精简 minify** + **TLS 连接复用**，降加载时间
- **抗流量峰值**：负载均衡均摊 + **智能故障转移** + Anycast 路由转移整机房故障
- **抗 DDoS**：在边缘识别拦截恶意流量、Anycast 摊薄、再加 WAF 与新鲜 TLS
- **省带宽**：缓存拦下绝大多数请求，**源站需提供的数据量大幅下降**

</v-click>

<v-click>

> 用对 CDN 的前提：**更换并隐藏源站真实 IP**——否则攻击者绕过 CDN 直击源站，盾牌形同虚设。

</v-click>

<!--
慢站会赶走用户，加速直接降低跳出率。CDN 不能让源站"刀枪不入"，但用对了能让源站"隐身"。
-->

---

# 动态加速与边缘计算

不可缓存的请求也能加速

<v-click>

- **动态请求不可缓存**：登录、下单、个性化接口每次都得回源
- **动态加速**：边缘处在高速 IXP 上，用**更优网间路由 + 连接复用 + 协议优化**压缩回源 RTT
- 即便要回源，也比用户从本地直连源站更快

</v-click>

<v-click>

- **边缘计算（边缘函数）**：把重写、鉴权、A/B 分流、个性化拼装等轻量逻辑**下沉到边缘**执行
- 边缘正从「缓存层」演进为**「可编程的计算层」**——动态响应也能就近生成

</v-click>

<!--
现代 CDN 的能力早已超出"边缘缓存静态、源站处理动态"这一简化模型。边缘函数让部分逻辑前移到离用户最近处。
-->

# 网络性能指标与优化

先量化，再优化

---

# 四个核心指标

脱离指标谈"网络慢"毫无意义

<v-click>

| 指标 | 含义 | 单位 | 前端能做什么 |
| --- | --- | --- | --- |
| 带宽 bandwidth | 链路吞吐上限 | bps | 压缩体积、按需加载 |
| 延迟 latency/RTT | 一来一回往返时间 | ms | 减往返、就近接入 |
| 丢包率 loss | 丢失分组占比 | % | 重试、降级、容错 UI |
| 抖动 jitter | 连续分组延迟波动 | ms | 缓冲、避免强实时依赖 |

</v-click>

<v-click>

> **带宽 ≠ 吞吐量**：带宽是链路理论上限，吞吐量是实际拿到的速率。受丢包、拥塞、TCP 慢启动影响，吞吐量往往远低于带宽。

</v-click>

<!--
慢在带宽？还是慢在延迟？两者解法南辕北辙。测速跑出 100 Mbps 不代表每个请求都能跑满。
-->

---

# 延迟的四个构成

为什么"加钱"治不了延迟

<v-click>

- **传播延迟**：距离 / 传播速度——**只跟物理距离有关，砸钱压不下去**
- **传输延迟**：分组比特数 / 带宽——**这一项才跟带宽相关**，但通常只占一小块
- **处理延迟**：路由器查表/校验，现代硬件极快，可忽略
- **排队延迟**：缓冲区排队，**拥塞时暴涨**，是弱网延迟剧烈抖动的主因

</v-click>

<v-click>

> 光在光纤里只跑约 **200,000 km/s**（折射率约 1.5，是真空光速的 ~2/3）；纽约↔旧金山理论 RTT 约 42 ms。

</v-click>

<!--
传播延迟是物理硬约束，这正是 CDN 把内容推到用户身边的根本原因。加宽链路只能改善传输延迟那一小块。
-->

# 延迟才是真瓶颈

「带宽可以加钱，延迟受光速限制」

<v-click>

- 把带宽从 5 Mbps 升到 10 Mbps，页面加载几乎无感
- 把 RTT 从 100 ms 降到 50 ms，**每一次往返都立竿见影**
- 用户在 **100~200 ms** 就能察觉延迟，超 **300 ms** 体感明显卡顿

</v-click>

<v-click>

| 路径 | 光纤理论 RTT（实际常 2~3 倍） |
| --- | --- |
| 纽约 ↔ 旧金山 | ~42 ms |
| 纽约 ↔ 伦敦 | ~56 ms |
| 纽约 ↔ 悉尼 | ~160 ms |

</v-click>

# 弱网优化清单（下）

自适应 + 离线 + 容错

<v-click>

```js
// 超时 + 指数退避重试：弱网下避免请求卡死与雪崩
async function fetchWithRetry(url, { retries = 2, timeout = 8000 } = {}) {
  for (let i = 0; i <= retries; i++) {
    try {
      return await fetch(url, { signal: AbortSignal.timeout(timeout) });
    } catch (err) {
      if (i === retries) throw err;
      const backoff = 2 ** i * 500 + Math.random() * 300;
      await new Promise((r) => setTimeout(r, backoff));
    }
  }
}
```

</v-click>

<v-click>

> 再加 **自适应加载**（按 `effectiveType`/`saveData` 减质降载）+ **Service Worker** 缓存壳，弱网/断网仍秒开。

</v-click>

# 小结

比特如何到达用户，六块拼图

<v-click>

- **接入/宽带**：接入网=最后一公里决定起跑线；FTTH/PON 是主流，注意上下行非对称与 bit/Byte
- **蜂窝/弱网**：小区+频率复用+切换；移动是另一个世界——RRC 状态机让"频繁小请求"特别糟
- **CDN**：距离即延迟，边缘就近交付、Anycast 抗灾抗 DDoS、隐藏源站
- **性能**：四指标里**延迟才是真瓶颈**——减往返 + 就近 + 减体积 + 容错

</v-click>

<v-click>

> 知道"起跑线在哪、路上有多堵"，前端优化才使得上劲。

</v-click>
