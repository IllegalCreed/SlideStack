# 这一章讲什么

一条主线：**通信的复杂度，靠分层来分而治之**

<v-click>

# 为什么要分层

把大问题切成职责单一的层

---

# 网络通信到底有多复杂

让一条 HTTP 请求跨越地球，要同时操心五类问题

<v-click>

- **物理层面**：0 和 1 怎么变成铜缆电压 / 光脉冲 / Wi-Fi 电磁波
- **链路层面**：同一局域网里，帧发给谁？多设备同时发冲突怎么办
- **网络层面**：对方隔着十几个路由器，怎么找一条路（路由）
- **传输层面**：网络会丢包、乱序、重复，怎么保证完整有序
- **应用层面**：到了对方机器，交给哪个程序？业务语义怎么约定

</v-click>

<v-click>

> 复杂不在「某件事多难」，而在「太多本质不同的事必须同时做对」。

</v-click>

<!--
五类问题一锅烩进单个程序就是灾难：代码极度耦合、换介质要重写、各厂商互不相通。这正是分层要破解的根本困境。
-->

---

# 分层思想：分而治之

把「网络通信」纵向切成若干职责单一的层

<v-click>

- **职责单一**：一层只解决一类问题（如网络层只管跨网找路），不越界
- **各层独立**：实现封装在内部，对外只暴露「能提供什么服务」
- 上层用下层服务时，**完全不必知道下层怎么做到**

</v-click>

<v-click>

> 类比寄跨国信：你只管写信投邮筒，分拣 / 运输 / 海关 / 派送层层分工，你不必懂航空货运怎么调度。

</v-click>

<!--
两个朴素直觉：横看，同层双方「说同一种行话」就是协议；竖看，相邻层「你帮我做这步」就是服务。后面会讲透。
-->

---

# 分层带来的四大好处

不是为了好看，而是解决「复杂 + 异构 + 长期演进」

<v-click>

- **模块化**：每层自包含，写传输层的人不必是无线电专家
- **解耦**：层间只通过接口交互，接口不变则内部随便改
- **标准化与互操作**：同层认同一套协议，思科路由器能和华为对话
- **可独立演进 / 替换**：某层单独换代，上层无感

</v-click>

<v-click>

> Wi-Fi 6 升 Wi-Fi 7、TCP 切 QUIC（HTTP/3），应用层的 HTTP 完全无感。

</v-click>

<!--
解耦是最实用的工程价值。标准化正是 OSI 的初衷——让不同计算机系统用标准协议彼此通信，被称为「网络的通用语言」。
-->

---

# 可独立演进：一张表看清

每一层都能单独升级换代，而不必推倒重来

<v-click>

| 演进 | 变化的层 | 上层是否受影响 |
| --- | --- | --- |
| 铜缆 → 光纤 → 5G | 物理层 | 否 |
| IPv4 → IPv6 | 网络层 | 几乎无感（按域名访问） |
| TCP → QUIC（HTTP/3） | 传输层 | 应用语义不变 |
| HTTP/1.1 → 2 → 3 | 应用层 | 下层照常承载 |

</v-click>

<v-click>

> 底层介质、寻址、传输、应用各自独立换代，互联网才能平滑运行几十年。

</v-click>

<!--
这张表是「解耦 + 可演进」的最佳注脚：每一行只动一层，其余层不受牵连，这正是分层的长期价值所在。
-->

---

# 协议与协议栈

分层的两个基础名词

<v-click>

- **协议（protocol）**：MDN 定义——「一套规定数据如何交换的规则」；双方须就**数据格式**达成一致
- **协议栈（protocol stack）**：各层协议**自上而下叠在一起**协同工作的整体

</v-click>

<v-click>

```text
应用层   HTTP / DNS / …     ← 业务语义
传输层   TCP / UDP          ← 端到端可靠交付
网络层   IP / ICMP          ← 跨网络寻址与路由
链路/物理 以太网 / Wi-Fi     ← 比特在介质上传输
        这一整摞 = 协议栈
```

</v-click>

<!--
别把「模型」和「协议栈」混了：OSI/TCP-IP 是分层模型（抽象骨架），TCP/IP 协议栈是骨架上真实跑着的一组具体协议（血肉）。
-->

---

# 服务竖着走，协议横着走

分层模型里最容易混淆、却最关键的一组概念

<v-click>

- **服务 / 接口（纵向）**：下层为上层提供能力，上层据接口调用
  - 例：传输层向应用层提供「可靠字节流」，HTTP 从不操心丢包
- **协议（横向）**：对等层之间按同一套规矩对话
  - 我方第 N 层与对方第 N 层「逻辑上直接交谈」

</v-click>

<v-click>

> 一句话记牢：**服务问「下层能为我做什么」，协议问「我和对方同层按什么规矩说话」**——两者正交。

</v-click>

<!--
接口是纵向的调用边界，只要接口稳定下层随便重构上层不受影响（解耦的机制落点）。协议是横向的对等约定，数据物理上其实一路下穿再上交。
-->

---

# 分层的代价

不是免费午餐，但代价远小于收益

<v-click>

- **少量冗余 / 开销**：每经一层通常要**加本层首部**（TCP 头、IP 头），层层累加
- **跨层信息隐藏**：上层看不见下层细节——换来解耦，但极致性能优化时偶尔要「打破分层」

</v-click>

<v-click>

> 头部冗余、信息隐藏都是为换取**模块化 / 可演进 / 可互通**主动做的取舍，是工程权衡而非缺陷。

</v-click>

<!--
头部开销正是下一章「封装」要展开的内容。cross-layer design 在追求极致性能时拿下层信息调优，本质是在「干净分层」与「性能」间权衡。
-->

# OSI 七层总表（上半）

ISO 提出的参考模型：七个相互堆叠的抽象层

<v-click>

| 层 | 名称 | 核心职责 | PDU |
| --- | --- | --- | --- |
| 7 | 应用层 Application | 面向用户的应用协议与数据 | 数据 Data |
| 6 | 表示层 Presentation | 编码转换、加密、压缩 | 数据 Data |
| 5 | 会话层 Session | 建立 / 维持 / 关闭会话 | 数据 Data |
| 4 | 传输层 Transport | 端到端、分段重组、流控差错 | 段 Segment |

</v-click>

<v-click>

> 层 4 及以上是「端到端」的主机职责，层 3 及以下才是网络设备真正参与转发的层。

</v-click>

<!--
先看上半四层。记一条规律：层 4 是分水岭——上面四层是主机的事，下面三层才有交换机、路由器参与转发。这条线在排障时极有用。
-->

---

# OSI 七层总表（下半）

越往下越贴近硬件、越贴近比特

<v-click>

| 层 | 名称 | 核心职责 | PDU |
| --- | --- | --- | --- |
| 3 | 网络层 Network | 逻辑寻址（IP）与路由选路 | 包 Packet |
| 2 | 数据链路层 Data Link | 同网相邻节点成帧、MAC 寻址 | 帧 Frame |
| 1 | 物理层 Physical | 比特流的物理传输 | 比特 Bit |

</v-click>

<v-click>

> 助记句 **「Please Do Not Throw Sausage Pizza Away」**对应层 1→7 首字母（P-D-N-T-S-P-A）。

</v-click>

<!--
下三层是网络设备的舞台：物理层管比特、链路层管帧（MAC）、网络层管包（IP）。助记句从层 1 物理到层 7 应用，背下来面试不慌。
-->

---

# 物理层 / 数据链路层

最底两层：从比特到帧

<v-click>

- **层 1 物理层**：在介质上收发**原始比特流**，只管把 1/0 变成信号
  - 关键词：线缆、电平、调制；设备：网卡、集线器 hub、中继器
- **层 2 数据链路层**：同网相邻节点间成**帧**，用 **MAC 地址**寻址、做差错检测
  - 关键词：MAC、成帧、CRC；设备：交换机 switch、网桥

</v-click>

<v-click>

> 二者都做寻址转发，区别在范围：**链路层只管同网（MAC），网络层负责跨网（IP）**——「二层交换、三层路由」由此而来。

</v-click>

<!--
物理层只认信号不懂含义。链路层把包封成帧、在本段链路做 CRC 校验。交换机看 MAC（层 2）、路由器看 IP（层 3），泾渭分明。
-->

---

# 网络层 / 传输层

承上启下的两层：从跨网寻址到端到端

<v-click>

- **层 3 网络层**：跨**不同网络**转发，用 **IP** 逻辑寻址、为每个包**选最优路径（路由）**
  - 协议：IP / ICMP / IGMP / IPsec；设备：路由器 router
- **层 4 传输层**：两台主机间**端到端**通信，切段重组、流控差错，用**端口号**交付到进程
  - TCP（面向连接、可靠有序）；UDP（无连接、快、开销小）

</v-click>

<v-click>

> 网络层只保证「送到对的主机」，**「确保完整、有序、不丢」是传输层（TCP）的事**。

</v-click>

<!--
网络层是腰部承重墙。传输层的关键词是「端到端」——HTTP 能稳定收发整段报文，正是站在 TCP 的可靠传输之上，这也是 TLS 常被画在传输层之上、应用层之下的原因。
-->

---

# 会话层 / 表示层 / 应用层

最上三层：从会话管理到面向用户

<v-click>

- **层 5 会话层**：建立 / 维持 / 关闭会话，可用**检查点**做断点续传
- **层 6 表示层**：数据的「翻译官」——**编码转换 / 加解密 / 压缩**（TLS、MIME）
- **层 7 应用层**：唯一**直接面向用户数据**的层，承载 HTTP / DNS / SMTP / FTP

</v-click>

<v-click>

> 注意：**浏览器、邮件客户端这些软件本身不属于应用层**，应用层指它们依赖的协议。

</v-click>

<!--
表示层让数据「可被应用消费」：加密后呈明文、压缩省流量。应用层的常见误区是把软件当成应用层——它指的是协议（HTTP、SMTP）与数据处理，不是 Chrome 这个程序。
-->

---

# OSI 是参考模型，不是实现

最容易被忽视、却最重要的一点

<v-click>

- **OSI 是 ISO 制定的抽象概念模型**，更像一把「描述与排障的标尺」
- **现代互联网并不严格遵循 OSI**，而是更贴近更简单的 **TCP/IP 协议族**
- OSI 自家协议被认为过于复杂、难落地，最终互联网协议族成了事实标准

</v-click>

<v-click>

> 它的价值在**排障**：把问题「分层定位」——应用故障查层 7，连通性问题查层 3/4。

</v-click>

<!--
学 OSI 的正确姿势：不必纠结会话层、表示层现实里是否独立存在（TCP/IP 里大多并入应用层）。把它当一套通用词汇——精准说出「这是二层还是三层问题」「TLS 大致在表示层附近」，沟通排障立竿见影。
-->

# TCP/IP：真正运行的那套

OSI 结构完美，却几乎没协议严格按它实现

<v-click>

- 真正驱动互联网的是更早诞生、更简洁的 **TCP/IP 模型**（Internet protocol suite）
- Cloudflare 直言：现代互联网**不严格遵循 OSI，更接近更简单的 TCP/IP 协议族**
- 一句话定位：**OSI 用来教学与排障，TCP/IP 用来真正跑业务**

</v-click>

<v-click>

> 模型以两个核心协议命名——**TCP**（传输层）+ **IP**（网际层），实际含 HTTP、DNS、UDP、ICMP 等一大批协议。

</v-click>

<!--
学网络要懂 OSI 的分层思想，但写代码、抓包、查故障，面对的全是 TCP/IP。RFC 1122 是它的权威定义文档。
-->

---

# TCP/IP 四层逐层职责

RFC 1122 把协议族划成四层，自底向上

<v-click>

| 层（自底向上） | 核心职责 | 典型协议 |
| --- | --- | --- |
| 应用层 Application | 为用户应用提供数据交换 | HTTP / DNS / SMTP / SSH |
| 传输层 Transport | **进程到进程**端到端，端口区分应用 | TCP / UDP |
| 网际层 Internet | 跨网**逻辑寻址 + 路由**，尽力而为投递 | IP / ICMP / IGMP |
| 网络接口层 Link | 同一**本地链路**发到相邻节点，无需路由器 | Ethernet / Wi-Fi / ARP |

</v-click>

<v-click>

> 网际层（IP）只管尽力送达——**不保证不丢、不保证有序**，可靠性刻意上移到传输层。

</v-click>

<!--
ARP（IP→MAC）归在网络接口层。ICMP（ping 用它）在网际层。网际层是腰部承重墙，「保持核心简单、把智能放端侧」是 TCP/IP 的设计哲学。
-->

---

# 端到端原则

TCP/IP 最核心的设计哲学

<v-click>

- RFC 1122：端到端的**流控与可靠性，全部实现在两端主机的传输层**
- **网络只管转发，可靠性由两端自己负责**——这就是「端到端原则」
- TCP / UDP 哲学相反：
  - **TCP** 面向连接、确认重传保不丢（网页、邮件、SSH）
  - **UDP** 无连接、首部仅 8 字节、低延迟（DNS、实时音视频、QUIC）

</v-click>

<v-click>

> 把智能放在端侧、核心保持简单，正是互联网能做大、能持续演进的根本设计。

</v-click>

<!--
端到端原则是 TCP/IP 成功的关键：中间节点只转发，不维护连接状态，所以网络核心轻、易扩展。TCP 用端口把「主机到主机」升级为「进程到进程」。
-->

---

# 应用层：吃下 OSI 上三层

TCP/IP 应用层一层承担 OSI 的三层职责

<v-click>

- TCP/IP **不单独设表示层和会话层**
- 数据格式、加密、会话管理若有需要，由**各应用协议自行处理**
- 例：HTTPS 在应用层之下用 **TLS** 做加密，HTTP 自己管会话（Cookie / Keep-Alive）

</v-click>

<v-click>

> Wikipedia：TCP/IP **刻意不管数据表示，也不在应用层和传输层之间多设层次**——一切从简。

</v-click>

<!--
RFC 1122 把应用层分成「用户协议」（FTP、SMTP，今天更多是 HTTP）和「支撑协议」（DNS、SNMP、DHCP）。HTTP 一个协议就同时承担了 OSI 应用 + 表示 + 会话的活。
-->

---

# 四层与 OSI 的映射

中间两层一一对应，头尾各一处合并

<v-click>

| OSI 七层 | TCP/IP 四层 | 对应关系 |
| --- | --- | --- |
| 7 应用 / 6 表示 / 5 会话 | 应用层 | **三合一** |
| 4 传输 | 传输层 | 一一对应 |
| 3 网络 | 网际层 | 一一对应 |
| 2 数据链路 / 1 物理 | 网络接口层 | **二合一** |

</v-click>

<v-click>

> 口诀：**头尾各砍一刀——头部三层并一层，尾部两层并一层，中间原样保留。**

</v-click>

<!--
顶部三合一：会话/表示/应用都交给应用自己看着办。底部二合一：「怎么把比特怼上线路」属实现细节，模型不必操心。这两处最容易记混。
-->

# 五层教学模型

把「胖」的网络接口层拆清楚

<v-click>

- 四层工程够用，但网络接口层把「物理比特」和「成帧 / 链路控制」混在一起
- 教科书（Tanenbaum、Kurose）折中——**把网络接口层拆成物理层 + 数据链路层**
- 得到：**物理 / 数据链路 / 网际 / 传输 / 应用** 五层

</v-click>

<v-click>

```text
TCP/IP 四层            五层教学模型
  应用      ───►        应用
  传输      ───►        传输
  网际      ───►        网际
  网络接口  ───►   数据链路  ← 拆
  (链路)              物理    ← 开
```

</v-click>

<!--
五层模型没有任何 RFC 定义它，本质仍是 TCP/IP——只借 OSI 物理/数据链路的划分把胖层拆清楚。真实系统按四层实现，课堂上按五层讲解，讲的是同一回事。
-->

# 为什么 TCP/IP 成了事实标准

OSI 更完整，为何反被「更糙」的 TCP/IP 取代

<v-click>

- **先有实现，后有模型**：DARPA 从 1960s 末边做边定，在 ARPANET 上**真跑通了**才写进 RFC
- 等 OSI 标准 1980s 敲定，TCP/IP 早已铺遍全球，形成**安装基数与生态惯性**
- **简洁实用**：只设必要的层，把智能放端侧；RFC 1122 坦承「严格分层只是不完美的模型」

</v-click>

<v-click>

> 一句话总结成败：**用 OSI 的语言去理解，用 TCP/IP 的协议去运行。**

</v-click>

<!--
TCP/IP 赢在「能用、好用、已经在用」：跑通在先、简洁务实、生态既成。OSI 赢在「教得清」：分层严谨、概念清晰。所以是双赢的分工，而非谁淘汰谁。
-->

层层套信封 · 层层拆信封

# 为什么需要封装

层与层之间靠什么传递数据

<v-click>

- 上层把整块数据交给下层：「帮我把这坨送到对端的同层」
- 下层不关心含义，只当**不透明的载荷**，在前面加上**自己的首部**，再交给更下层
- 层层向下，数据像被套进一个又一个信封

</v-click>

<v-click>

> 类比：信（应用数据）→ 套「房间号」信封（传输层）→ 套「楼栋地址」信封（网络层）→ 贴「快递面单」（链路层）。每层只看自己那层信封。

</v-click>

<!--
注意别和 OOP 的封装混了：网络的「封装」指给数据加首部，和面向对象「隐藏内部实现」是同名不同义，只是都借「包起来」的意象。
-->

---

# 封装：自顶向下加首部

发送方数据每下沉一层就被当作载荷、加本层首部

<v-click>

```text
应用层   [ HTTP 报文 ]                         ← data
传输层   [ TCP头 | HTTP 报文 ]                  ← segment 段
网络层   [ IP头 | TCP头 | HTTP 报文 ]           ← packet 包
链路层   [ 帧头 | IP头 | TCP头 | 报文 | 帧尾 ]   ← frame 帧
物理层    1011010110011010 ...                 ← bit 比特流
```

</v-click>

<v-click>

- **每层只往最外面加一层首部**，不改动里层——里层对外层是黑盒
- 传输层放端口序号，网络层放源 / 目的 IP，链路层放 MAC + 帧校验 FCS（在尾部）

</v-click>

<!--
物理层不再加首部，它把链路层的帧编码为电/光/电磁信号，以比特为单位发到线路上。首部里装什么由该层职责决定。
-->

---

# 各层 PDU 名称对照

同一份数据在不同层有不同正式称呼

<v-click>

| 层 | PDU 名称 | 主要新增首部信息 |
| --- | --- | --- |
| 应用层 | 数据 / 报文 data / message | 应用协议自身（如 HTTP 头） |
| 传输层 | **段 segment**（TCP）/ 数据报（UDP） | 源 / 目的端口、序号 |
| 网络层 | **包 packet** | 源 / 目的 IP、TTL |
| 数据链路层 | **帧 frame** | 源 / 目的 MAC，尾部 FCS |
| 物理层 | **比特 bit** | 无首部，仅信号编码 |

</v-click>

<v-click>

> 口诀：**数据 → 段 → 包 → 帧 → 比特**。最易混：**段 = TCP 的 PDU，数据报 = UDP 的 PDU**。

</v-click>

<!--
面试常考「网络层的 PDU 叫什么」，答「包/分组 packet」。TCP 用「段」、UDP 用「数据报」是初学者最易混的点，务必记牢。
-->

---

# 解封装：自底向上剥首部

接收方做的事和发送方严格镜像

<v-click>

```text
物理层    1011010110011010 ...        还原成帧
链路层    [ 帧头 | IP包 | 帧尾 ]       校验 FCS、剥帧头尾 → 取 IP 包
网络层    [ IP头 | TCP段 ]            看目的 IP、剥 IP 头 → 取 TCP 段
传输层    [ TCP头 | HTTP 报文 ]       按目的端口分用、剥 TCP 头 → 取报文
应用层    [ HTTP 报文 ]              应用程序拿到完整数据
```

</v-click>

<v-click>

> 每层**先读自己那层首部**再剥：链路层校验 FCS、网络层确认目的 IP、传输层按端口**分用**到进程。

</v-click>

<!--
这正是「每层只认自己的首部」的体现——下层首部对上层已不可见，关注点被层层隔离。封装与解封装严格镜像：HTTP↔HTTP、TCP↔TCP、IP↔IP、帧↔帧。
-->

---

# 首部 / 载荷 / 尾部 与 MTU

每个 PDU 的内部构成，以及一个物理约束

<v-click>

- **首部 header**：控制信息（源 / 目的地址、序号、校验码）
- **载荷 payload**：上一层整体交下来的真正数据
- **尾部 trailer**（部分层才有）：如以太网把 **FCS 校验**放在尾部

</v-click>

<v-click>

- **MTU（最大传输单元）**：链路层一帧能携带的载荷上限，以太网典型 **1500 字节**
- IP 包超过 MTU 就要**分片**；分片代价高（任一片丢失需整体重传），现代靠 **PMTUD** 提前规避

</v-click>

<!--
分片是「不得已」而非常态：放大首部开销、增加重组负担。PMTUD（路径 MTU 发现）让发送方提前探测全路径最小 MTU，从源头避免分片。TCP 据此协商 MSS 属传输层范畴。
-->

# 三模型对照大表

同一套分层思想的不同切法，并非互相矛盾

<v-click>

| OSI 七层 | TCP/IP 四层 | 五层教学 | PDU |
| --- | --- | --- | --- |
| 7 应用 / 6 表示 / 5 会话 | 应用层 | 应用层 | 数据 |
| 4 传输 | 传输层 | 传输层 | 段 |
| 3 网络 | 网际层 | 网络层 | 包 |
| 2 数据链路 | 链路层 | 数据链路层 | 帧 |
| 1 物理 | （并入链路层） | 物理层 | 比特 |

</v-click>

<v-click>

</v-click>

<!--

# 常见协议归层

面试高频题——本表以 TCP/IP 四层为主轴

<v-click>

| 协议 | TCP/IP 层 | 作用速记 |
| --- | --- | --- |
| HTTP / HTTPS | 应用层 | Web 请求响应；HTTPS = HTTP + TLS |
| DNS | 应用层 | 域名 → IP（多 UDP，亦可 TCP） |
| WebSocket | 应用层 | 全双工长连接（由 HTTP 升级） |
| TCP / UDP | 传输层 | 可靠有序 / 不可靠但快 |
| IP / ICMP | 网际层 | 逻辑寻址路由 / 差错控制（ping） |
| Ethernet / Wi-Fi | 链路层 | 有线 / 无线局域网帧传输 |

</v-click>

<v-click>

> 应用层协议无一例外都落到 TCP 或 UDP，再经 IP 跨网段送达。

</v-click>

<!--
OSI 视角下应用层协议被细分到 5~7 层，但 TCP/IP 视角统归应用层。HTTPS=HTTP+TLS、WebSocket 由 HTTP 升级、DNS 多走 UDP，都说明「应用层关心传什么，传输层关心怎么可靠传」。
-->

# ARP 归层为何有争议

把 IP 解析成 MAC，跨接两层

<v-click>

- ARP 的任务是**把网络层的 IP 解析成链路层的 MAC**，因此**同时触及第 2、3 层**
- **OSI 语境**：多数教材归**第 2 层**（报文封装在以太网帧里、不经 IP）
- **TCP/IP 语境**：Wikipedia 明确划归**链路层**

</v-click>

<v-click>

> 实务结论：面试答「**链路层 / OSI 第 2 层**」最稳，再补「它跨接网络层与链路层、为 IP 寻址服务」体现深度。

</v-click>

<!--
也有观点认为 ARP「横跨 2/3」或属「2.5 层」。但 ARP 报文本身封装在以太网帧里、不经过 IP，所以归第 2 层最站得住脚。
-->

# 常见设备归层

设备工作在「它能解析到的最高层」

<v-click>

| 设备 | 工作层 | 依据信息 | 行为 |
| --- | --- | --- | --- |
| 集线器 Hub | 物理层（1） | 无（电信号） | 向所有端口广播（已淘汰） |
| 交换机 Switch | 数据链路层（2） | MAC 地址 | 按 MAC 表精准转发，隔离冲突域 |
| 路由器 Router | 网络层（3） | IP 地址 | 按路由表跨网段转发包 |
| 网关 Gateway | 应用层（7） | 协议 / 载荷 | 在不同协议体系间翻译 |
| 防火墙 Firewall | 3~7 层 | IP / 端口 ~ 载荷 | 包过滤看 3/4，WAF 深入 7 |

</v-click>

<v-click>

> 口诀：层级越高，看到的信息越多——hub 看信号(1)、switch 看 MAC(2)、router 看 IP(3)。

</v-click>

<!--
注意「网关」语境：日常说的「默认网关」通常指路由器（第 3 层）；OSI 语境的「网关」特指第 7 层协议翻译设备。同名不同义，按上下文判断。
-->

---

# 从工作层看「隔离了什么」

设备归层最实用的推论：层级越高隔离范围越大

<v-click>

- **集线器（1 层）**：不隔离——所有端口同属一个**冲突域 + 广播域**
- **交换机（2 层）**：**每个端口是独立冲突域**（消除冲突），但默认同一**广播域**
- **路由器（3 层）**：以 IP 转发、**不转发广播**，天然**分隔广播域**

</v-click>

<v-click>

> 一句话：**hub 什么都不隔离，switch 隔离冲突域，router 隔离广播域**——这就是局域网用交换机取代集线器、用路由器划子网的原因。

</v-click>

<!--
集线器 vs 交换机是经典对照：前者物理层「傻广播」（靠主机自己丢弃），后者链路层「智能转发」（学习 MAC、只发目标口）。这正是物理层 vs 数据链路层差异的最直观体现。
-->

# 回车之后发生了什么

输入 `https://example.com` 回车，先解决三个前置问题

<v-click>

```text
浏览器                                        服务器
  ① DNS 解析   example.com → 93.184.216.34    应用层 UDP/53
  ② TCP 握手   SYN → SYN+ACK → ACK            传输层 1 RTT
  ③ TLS 握手   ClientHello → … → 协商密钥      TLS 1.3 1 RTT
  ④ HTTP 请求  GET / HTTP/1.1（加密承载）  →
               ← 200 OK + HTML
  ⑤ 渲染       解析 HTML/CSS/JS，绘制页面
```

</v-click>

<v-click>

> MDN 给出固定次序：**DNS → TCP 握手 → TLS → HTTP 请求 → 响应**。前三步是铺垫，第④步才是正事。

</v-click>

<!--
DNS、TCP、TLS 解决的都是「正式对话前的准备」：知道往哪发、建好可靠连接、谈妥加密。真正承载业务的是第④步的 HTTP 报文。抓住这条主线就抓住了整条旅程。
-->

---

# 发送方：自顶向下封装

第④步的 `GET /` 要发出去，沿协议栈从上往下打包

<v-click>

| 层 | 加什么头 | 产物 PDU | 关键字段 |
| --- | --- | --- | --- |
| 应用层 | —（生成数据） | HTTP 报文 | 方法、URL、首部 |
| 传输层 | TCP 头 | 段 Segment | **源 / 目的端口**、序列号 |
| 网络层 | IP 头 | 包 Packet | **源 / 目的 IP**、TTL |
| 链路层 | 帧头 + 帧尾 | 帧 Frame | **源 / 目的 MAC**、FCS |
| 物理层 | —（编码信号） | 比特 Bit | 电 / 光 / 电磁波 |

</v-click>

<v-click>

> 三种地址各司其职：**端口**交给哪个进程、**IP** 送到哪台主机、**MAC** 这段链路下一跳给谁。

</v-click>

<!--
每下一层套一个信封。端口（传输层）、IP（网络层）、MAC（链路层）三种地址各管一摊，正是分层思想的直接体现。
-->

---

# 网络中途：交换机与路由器

封装好的帧穿过中间设备到达服务器

<v-click>

```text
主机 ─帧─► [交换机] ─帧─► [路由器] ══包══► [路由器] ─帧─► 服务器
          链路层转发      网络层逐跳转发        网络层逐跳     链路层
          (看目的MAC)     (改MAC·不改IP)      (改MAC·不改IP)
```

</v-click>

<v-click>

- **交换机（二层）**：只看**目的 MAC**在局域网内转发，**不拆 IP 头、不改地址**
- **路由器（三层）**：拆帧读 **IP 头**，按路由表定下一跳，**重新封装新帧**再发

</v-click>

<!--
跨网段就得靠路由器。它拆开帧、读 IP 头决定下一跳出口，然后重新封装一个新帧。这里藏着本章最核心的一条规律，下一页展开。
-->

---

# MAC 逐跳变，IP 端到端不变

本章最该记牢的一句话

<v-click>

- **目的 / 源 IP**：整段旅程**始终不变**（除非 NAT）——标识「最终发送方与接收方」，是**端到端**的
- **目的 / 源 MAC**：**每过一个路由器就被重写一次**——只标识「这一段链路的上一跳与下一跳」，是**逐跳**的

</v-click>

<v-click>

> 一句话：**IP 管「最终去哪」，MAC 管「下一跳给谁」。** 每过一跳 TTL 减 1，归零即丢弃以防环路。

</v-click>

<!--
这是网络层与链路层分工的真实写照。MAC 解决「下一跳给谁」，IP 解决「最终去哪」。理解这条，路由器为什么改 MAC 不改 IP 就豁然开朗。
-->

---

# 接收方：自底向上解封装

帧到达服务器网卡，过程完全反过来

<v-click>

```text
物理层   收到比特流，还原成帧
链路层   剥帧头尾、校验 FCS、核对目的 MAC   → 交网络层
网络层   剥 IP 头、核对目的 IP、TTL        → 交传输层
传输层   剥 TCP 头、按目的端口(443)找进程   → 交应用层
应用层   还原完整 HTTP 报文               → Web 服务器处理
```

</v-click>

<v-click>

> 发送方加的每一层头，都由接收方对应层精确剥掉——**同层对话、逐层封装**正是异构网络能互通的根本。

</v-click>

<!--
链路层坏帧丢弃；网络层按协议号（这里 TCP）上交；传输层按目的端口把数据分用给服务进程，并用序列号/ACK 保证可靠有序；应用层拼回完整请求生成响应。
-->

---

# 响应原路返回：闭环

服务器的响应重复一遍同样的旅程

<v-click>

```text
请求：浏览器 ─封装─► 路由器逐跳 ─► 服务器解封装 ─► 处理
响应：服务器 ─封装─► 路由器逐跳 ─► 浏览器解封装 ─► 渲染呈现
```

</v-click>

<v-click>

- 响应同样**自顶向下封装**（HTTP→TCP→IP→帧→比特）
- 经路由器**逐跳转发**（同样改 MAC 不改 IP）穿回公网
- 到浏览器**自底向上解封装**，最终交给渲染引擎

</v-click>

<!--
一来一回，一次完整的请求—响应闭环就此完成。分层（谁负责什么）、封装/解封装（怎么打包）、各层协议（DNS/TCP/IP/以太网/HTTP）三条线在一次真实请求里彻底贯通。
-->

# 小结

网络分层模型，六块拼图

<v-click>

- **为什么分层**：分而治之，换来模块化 / 解耦 / 互操作 / 可演进
- **OSI 七层 vs TCP/IP 四层**：前者教学排障，后者真正在跑；头尾各合并一处
- **封装 / 解封装**：自顶向下加首部、自底向上剥首部，PDU = 数据→段→包→帧→比特
- **归层与旅程**：协议 / 设备归在「最高可解析层」，**MAC 逐跳变、IP 端到端不变**

</v-click>

<v-click>

> 用 OSI 的语言去理解，用 TCP/IP 的协议去运行——这就是分层模型的总钥匙。

</v-click>

<!--
六块拼图收束成一句话：分层规定「谁负责什么」，封装规定「数据怎么打包」，一次真实请求把它们串成闭环。再往下就是 DNS、TCP、TLS、HTTP 各自的深入了。
-->
