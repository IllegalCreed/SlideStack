# 传输层与端口

IP 送到机器，端口送到进程

---

# 传输层的职责

在「主机到主机」之上补一层「进程到进程」

<v-click>

- **网络层（IP）**：把数据包从**源主机**送到**目的主机**，回答「去哪台机器」
- **传输层**：在 IP 之上，把数据交给主机里**正确的那个进程**，回答「给哪个服务」
- 一台机器上同时跑浏览器、邮件、IM……IP 不知道包到了该交给谁
- 两者常合称 **TCP/IP**：IP 管路由（第 3 层），TCP 管平稳运输（第 4 层）

</v-click>

<v-click>

> 类比：IP 是写字楼地址（送到大楼门口），端口是房间号（决定进哪间办公室）。

</v-click>

<!--
分清网络层和传输层各管什么，是初学者最容易混的一层。IP 定位机器、端口定位进程，缺了端口信只能堆在大堂。
-->

---

# 端口号：16 位逻辑编号

不是物理接口，而是操作系统里的逻辑编号

<v-click>

| 范围 | 名称 | 谁来用 |
| --- | --- | --- |
| 0 ~ 1023 | 知名 / 系统端口 | 核心服务，绑定常需特权 |
| 1024 ~ 49151 | 注册端口 | 应用向 IANA 登记（如 MySQL 3306） |
| 49152 ~ 65535 | 动态 / 私有 / 临时端口 | 客户端发起连接时临时分配 |

</v-click>

<v-click>

- **位宽**：16 位无符号整数 → 取值 `0 ~ 65535`（共 65536 个）
- **服务端**在固定端口守候监听，**客户端**的源端口由系统临时分配、用完回收

</v-click>

<!--
三段分类来自 IANA。客户端访问网站，目的端口固定 443，但源端口是临时区间随手分的一个号，所以多开标签页不冲突。
-->

# 常见知名端口速记

URL 不写端口时浏览器按协议默认补全

<v-click>

| 端口 | 服务 | 传输层 |
| --- | --- | --- |
| 20 / 21 | FTP | TCP |
| 22 | SSH | TCP |
| 25 | SMTP | TCP |
| 53 | DNS | UDP / TCP |
| 80 | HTTP | TCP |
| 443 | HTTPS | TCP（1.1·2）/ UDP（HTTP/3） |

</v-click>

<v-click>

> 记忆要点：**80/443 是 Web 双子星**，**53 是 DNS**，**22 是 SSH**；`https://x.com` ≡ `https://x.com:443`。

</v-click>

<!--
DNS 默认走 UDP，大响应或区域传送才走 TCP。443 在 HTTP/3 时代既是 TCP 也是 UDP（QUIC），别只记一个。
-->

# 复用与分用：端口的根本用途

多进程共享一条网络，靠端口分流回各自进程

<v-click>

- **复用（发送方）**：多个进程共享下层网络，传输层给每段打**源端口**标签后统一交给 IP
- **分用（接收方）**：传输层按数据里的**目的端口**，把每段分发给对应进程

</v-click>

<v-click>

```text
   发送方                              接收方
 浏览器 邮件 IM                      浏览器 邮件 IM
    └─ 打源端口标签 ─┐          ┌─ 按目的端口分发 ─┘
       复用 ↓       │          │      分用 ↑
    传输层(TCP/UDP) │          │  传输层(TCP/UDP)
       ↓ 网络层 IP ─┴── 一条网络 ──┴─ 网络层 IP ↑
```

</v-click>

<!--
正因有端口与分用，操作系统才能把混在同一条网络上的字节流，准确还给该收它的那个进程，浏览器收 Web、邮件收邮件，互不串台。
-->

---

# 复用 ≠ HTTP/2 多路复用

同名不同层，最易混的一个坑

<v-click>

- **传输层复用 / 分用**：多个**进程**共享网络通道，靠端口区分 —— 本页讲的概念
- **HTTP/2 多路复用**：单条 **TCP 连接**上并发多个**请求流** —— 应用层优化
- 两者都叫 multiplexing，但层次完全不同，面试时别答串

</v-click>

<v-click>

> 一句话：传输层复用分「进程」，HTTP/2 复用分「请求流」。

</v-click>

<!--
这是高频混淆点。传输层的复用是通用概念（多进程共享网络），HTTP/2 的多路复用是单 TCP 连接上的并发请求流，属应用层。
-->

---

# socket：通信端点的唯一标识

端口是数字，socket 是「IP + 端口」绑定后的端点

<v-click>

- **socket = IP 地址 + 端口号**（再加协议），如 `203.0.113.5:443`
- **一条连接由四元组唯一确定**：`(源 IP, 源端口, 目的 IP, 目的端口)`
- 四元组任一项不同就是**不同连接** —— 这正是单端口服务海量客户端的原理

</v-click>

<v-click>

> 服务端单个 443 能同时服务千万客户端：每个客户端 `(源 IP, 源端口)` 各异，四元组就各异。

</v-click>

<!--
端口是房间号，socket 是连上这个房间的那条专线。四元组是理解「单端口为何能服务海量连接」的钥匙。
-->

---

# 两大协议概览：TCP vs UDP

都用端口做复用分用，但「性格」截然相反

<v-click>

| 维度 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接，先握手 | 无连接，直接发 |
| 可靠性 | 可靠：不丢、不重、有序 | 尽力而为：可能丢、可能乱 |
| 开销 / 速度 | 机制多、开销大、较慢 | 头小、机制少、更快 |
| 典型场景 | Web、邮件、文件传输 | DNS、音视频、游戏、QUIC |

</v-click>

<v-click>

> 两协议**端口空间互相独立**：TCP 的 80 与 UDP 的 80 是两个互不相干的端口。

</v-click>

<!--
本页只做鸟瞰，握手、可靠传输、拥塞控制的细节留给后面逐页讲透。注意 TCP 与 UDP 端口空间独立，由协议号区分。
-->

# UDP 协议

IP 之上最薄的一层：加端口 + 加校验和

---

# UDP 是什么

无连接的轻量传输协议（RFC 768，IP 协议号 17）

<v-click>

- **比喻**：TCP 像「打电话」（先拨号确认），**UDP 像「寄明信片」**（写地址直接投）
- 在 IP 之上**只做两件事**：
  - **加端口**：用源/目的端口区分进程（复用分用）
  - **加校验和**：发现损坏的包直接丢弃（但不重传）
- IP 不保证的（送达、顺序、去重），UDP **一概不补**

</v-click>

<v-click>

> 丢包不全是 UDP 的锅：底层路由器本就不做排序与确认，UDP 只是「如实暴露」IP 的不可靠。

</v-click>

<!--
UDP 把可靠性的缺口如实暴露，TCP 才是在传输层填补它。理解这点，就懂了可靠性本质是按需付费的开销。
-->

---

# UDP 报文：固定 8 字节首部

四个字段，每个 16 bit，结构极简

<v-click>

```text
 0      7 8     15 16    23 24    31   ← bit
+--------+--------+--------+--------+
|     源端口       |    目的端口      |
+--------+--------+--------+--------+
|      长度        |     校验和       |
+--------+--------+--------+--------+
|              数据 (payload)        |
+-----------------------------------+
```

</v-click>

<v-click>

- **源端口**：可选，无需回包时填 `0` ｜ **目的端口**：分用到正确应用
- **长度** = 首部 + 数据总字节数，最小值 **8** ｜ **校验和**：含伪首部，防误投

</v-click>

<!--
对比 TCP 至少 20 字节、含序号确认窗口标志位的复杂首部，UDP 的 8 字节几乎没有协议开销，这是它轻量快速的根源。
-->

---

# UDP 四不保证

用「不可靠」换「快」

<v-click>

- **不保证送达**：丢了不重传
- **不保证顺序**：可能乱序到达
- **不保证不重复**：可能收到重复包
- **无拥塞控制**：不会因网络拥塞自动降速

</v-click>

<v-click>

> 校验和只能**发现**损坏（坏包丢弃），不能**纠正**；要纠错就该改用 TCP 或在应用层自建。

</v-click>

<!--
四不保证是 UDP 的本性。校验和只发现不纠正，需要可靠就改 TCP，或像 QUIC 那样在应用层自己造一套。
-->

---

# UDP 为什么快

优势全部来自「少做事」

<v-click>

- **无握手，低延迟**：第一个包就是数据，省掉三次握手的一个 RTT
- **无连接状态，开销小**：服务端不为每条「连接」维护 TCB，省内存、易横扩
- **首部仅 8 字节**：TCP 至少 20 字节，UDP 协议开销几乎为零

</v-click>

<v-click>

> 跨洲 RTT 动辄上百毫秒，省掉它对 DNS、游戏指令这类小交互意义巨大。

</v-click>

<!--
三点优势对应三种省：省往返、省状态、省首部。DNS 这种「一台服务器应对海量短查询」的场景，UDP 天然合适。
-->

---

# UDP 适用场景

时效优先、容忍丢包的场合

<v-click>

| 场景 | 为什么选 UDP |
| --- | --- |
| DNS 查询 | 请求小、无需建连，快与省压倒一切 |
| 实时音视频 / 直播 | 时效 > 完整，丢几帧好过卡顿等重传 |
| 在线游戏 / VoIP | 最新一帧才有意义，旧包重传无价值 |
| DHCP | 主机还没 IP 时就要广播获取配置 |
| QUIC / HTTP/3 | 以 UDP 为底座，可靠传输做到应用层 |

</v-click>

<v-click>

> 反过来：网页 HTML、API 数据、文件下载这类「错一字节都不行」的，是 TCP 的主场。

</v-click>

<!--
可靠性可由应用层按需自建，QUIC 即范例，这正是 UDP 在现代协议中复兴的原因。安全副作用是反射放大攻击（伪造源 IP），属 DDoS 范畴。
-->

# TCP 握手与挥手

面向连接：先建连、再传输、最后释放

---

# 为什么要握手与挥手

连接 = 两端各自在内核里维护的一份状态

<v-click>

- **面向连接**：传数据前先建连、传完再释放，两端各分配缓冲区与序列号
- 「建立连接」不是拉物理线，而是双方各记一份**连接控制块（TCB）**
- 握手 = 协商同步这份状态；挥手 = 双方都确认「可以丢弃它了」

</v-click>

<v-click>

| 标志 | 含义 | 场景 |
| --- | --- | --- |
| `SYN` | 请求同步序列号 | 建立连接（握手前两步） |
| `ACK` | 确认收到 | 几乎所有报文都带 |
| `FIN` | 本方向已无数据可发 | 释放连接 |

</v-click>

<!--
TCP 用标志位驱动状态变迁。握手挥手最关键的三个就是 SYN/ACK/FIN，先记住它们各自的语义。
-->

---

# 三次握手

双向同步初始序列号（ISN），确认两个方向都通

<v-click>

```text
客户端                                  服务端
 CLOSED                                 LISTEN
   │ ──── ① SYN, seq=x ──────────────►  │
 SYN_SENT                             SYN_RCVD
   │ ◄─── ② SYN+ACK, seq=y, ack=x+1 ──  │
 ESTABLISHED                             │
   │ ──── ③ ACK, ack=y+1 ────────────►  │
   │                                ESTABLISHED
```

</v-click>

<v-click>

- **① SYN**：客户端选 ISN=x 发出（不带数据但消耗一个序列号）
- **② SYN+ACK**：服务端 `ack=x+1` 确认 + 自己 `seq=y` 同步，合并发出
- **③ ACK**：客户端 `ack=y+1` 确认，双方进入 `ESTABLISHED`

</v-click>

<!--
握手目标只有一个：同步双方 ISN，确认两个方向收发通畅。序列号如何用于可靠传输，是下一节的主题。
-->

---

# 为什么必须三次，两次不够

面试高频题，核心两点

<v-click>

- **① 两次无法让服务端确认「客户端的接收能力」**
  - 前两次只证明「客户端→服务端」通；唯有第三次 ACK 才能证明反向也通
- **② 防止已失效的历史 SYN 凭空建连**
  - 迟到的旧 SYN 到达，两次握手时服务端会单方面建连、白占资源

</v-click>

<v-click>

> 有了第三次：面对迟到旧 SYN，客户端早已不在等待、不回 ACK（甚至回 RST），虚假连接建不起来。

</v-click>

<!--
连接全双工，正常工作必须确认两个方向都通。第三次 ACK 同时堵死了「反向未确认」和「历史 SYN 误建连」两个漏洞。
-->

---

# 四次挥手

全双工：两个方向各自独立关闭

<v-click>

```text
（设客户端主动关闭）
客户端                                  服务端
   │ ──── ① FIN, seq=u ──────────────►  │
 FIN_WAIT_1                          CLOSE_WAIT
   │ ◄─── ② ACK, ack=u+1 ───────────── │
 FIN_WAIT_2                              │ （仍可发数据）
   │ ◄─── ③ FIN, seq=w ─────────────── │
   │                                 LAST_ACK
   │ ──── ④ ACK, ack=w+1 ───────────►  │
 TIME_WAIT                            CLOSED
   │  等待 2MSL → CLOSED
```

</v-click>

<v-click>

- **① FIN** 主动方：我没数据要发了 ｜ **② ACK** 被动方确认（进入半关闭）
- **③ FIN** 被动方数据发完才发 ｜ **④ ACK** 主动方确认，等 2MSL

</v-click>

<!--
第二、三次之间是「半关闭」：客户端方向已关，服务端方向仍可发数据。这段是挥手比握手多一步的根源。
-->

---

# 为什么是四次，能否合并

关键在「半关闭」那段

<v-click>

- **握手能合并**：服务端对 SYN 的 ACK 与自己的 SYN 此刻都能做 → 合成 `SYN+ACK`
- **挥手通常不能合并**：被动方收到 FIN 后**还有数据没发完**
  - 必须先立刻 ACK「我知道你要关了」
  - 自己的 FIN 要等数据发完、应用 `close()` 后才能发 → 时机不同

</v-click>

<v-click>

> 若被动方恰好也没数据要发，ACK 与 FIN 可合并成 `FIN+ACK`，挥手退化成三次。「四次」是逻辑上的最多步数。

</v-click>

<!--
握手两件事时机相同可合并，挥手两件事时机不同（先 ACK 后 FIN）通常不能合并。但被动方无数据时也会退化成三次。
-->

---

# CLOSE_WAIT：应用层信号

大量堆积 = 应用忘记关连接的 Bug

<v-click>

- 收到对端 FIN 后进入 `CLOSE_WAIT`，内核已替你回了 ACK
- 但只有**本端应用调用 `close()`** 才能发出己方 FIN，推进挥手
- 服务器上出现**大量长期不消失的 `CLOSE_WAIT`** → 几乎可断定连接泄漏

</v-click>

<v-click>

> 排查口诀：CLOSE_WAIT 堆积找**自己的应用**（忘了 close）；TIME_WAIT 堆积是**主动关闭方**的正常现象。

</v-click>

<!--
CLOSE_WAIT 意味着对端已关、本端应用还没 close。它是排查连接泄漏的重要线索，内核已 ACK，只差应用主动 close。
-->

---

# TIME_WAIT 为什么等 2MSL

主动关闭方最后那段等待，有两个理由

<v-click>

- **MSL**：报文最大生存时间（RFC 9293 默认约 2 分钟，Linux 常取约 60s）
- **理由①**：确保最后的 ACK 送达 —— ACK 丢了对端会重传 FIN，留在 TIME_WAIT 才能重发 ACK
- **理由②**：让旧连接残余报文消亡 —— 避免污染同四元组的新连接

</v-click>

<v-click>

> 高并发短连接 + 服务端主动关闭 → 海量 TIME_WAIT 占满端口；缓解用长连接 / 连接池 / `tcp_tw_reuse`。

</v-click>

<!--
2MSL 最坏耗时 = 己方 ACK 一个 MSL + 对方重传 FIN 一个 MSL。注意 tcp_tw_recycle 因 NAT 隐患早已被移除，不要再用。
-->

---

# 半连接队列与 SYN 洪水

握手未完成的连接暂存在哪，又如何被攻击

<v-click>

- **半连接队列（SYN queue）**：收到 SYN、回了 SYN+ACK 但**还没收到第三次 ACK** 的连接（half-open）
- **全连接队列（accept queue）**：握手已完成、等应用 `accept()` 取走
- **SYN 洪水**：用**伪造源 IP** 海量发 SYN 却从不回 ACK，塞满半连接队列耗尽资源

</v-click>

<v-click>

> **SYN Cookie**：队列将满时不分配资源，把连接信息编码进 SYN+ACK 的序列号，待合法 ACK 带回再校验重建。

</v-click>

<!--
SYN 洪水是经典 DDoS：伪造源 IP 收不到 SYN+ACK 也回不出正确 ACK。SYN Cookie 让服务端握手完成前不浪费资源，攻击便失去着力点。
-->

# TCP 可靠传输

把不可靠的 IP 改造成有序不丢不重的管道

---

# 可靠性的五件事

一条主线：编号 → 确认 → 重传 → 排序去重

<v-click>

- **序列号**：给字节流里每个字节编号
- **确认号**：累积确认，回告「连续收到哪」
- **重传**：超时重传 + 快速重传，丢了补发
- **校验和**：剔除损坏段（静默丢弃，当丢包处理）
- **排序去重**：接收方重排乱序、丢弃重复

</v-click>

<v-click>

> 比喻：IP 把拼图碎片乱序送达、还可能丢片；TCP 是收件端的「拼图组装工」。

</v-click>

<!--
五件事缺一不可。记住主线「编号→确认→重传→排序去重」，可靠性就是这几步的合奏。
-->

---

# 序列号：给每个字节编号

编号的是字节，不是报文段

<v-click>

```text
段 A: seq=1000, len=500  → 字节 [1000, 1499]
段 B: seq=1500, len=500  → 字节 [1500, 1999]
段 C: seq=2000, len=500  → 字节 [2000, 2499]
```

</v-click>

<v-click>

- 段头「序列号」字段 = 本段**第一个字节**的编号
- 按字节编号的三个好处：① 只重发缺失区间 ② 精确判重 ③ 乱序可重排
- **ISN 不从 0 开始**：握手时随机协商，防旧连接串扰与序号被猜测

</v-click>

<!--
按字节编号是 TCP 的精髓：重传精确到字节区间、判重和重排都靠它。ISN 随机化在握手页讲过，是安全考量。
-->

---

# 确认号 ACK：累积确认

ACK = X 表示「X 之前全收到，下一个期待 X」

<v-click>

```text
发出: seq=1000(500B), seq=1500(500B), seq=2000(500B)
收齐 → ACK=2500   （2500 之前全收到，下一个期待 2500）

若中间 seq=1500 丢失：
  收到 1000 → ACK=1500
  收到 2000（乱序，1500 缺）→ 仍回 ACK=1500（重复 ACK！）
  收到 2500（仍缺 1500）→ 再回 ACK=1500（又一个重复 ACK）
```

</v-click>

<v-click>

> ACK=2500 不是「收到了 2500 号字节」，而是「2499 及之前全收到，请从 2500 开始发」。

</v-click>

<!--
累积确认确认的是「连续无空洞的最高位置」。一旦中间有空洞，后续即便收到也只能反复确认空洞前位置，这就是重复 ACK 的来源。
-->

---

# 超时重传 RTO：兜底安全网

每段发出启动定时器，超时未确认就重传

<v-click>

- **RTO 不能写死**：太短把「慢」误判成「丢」，太长则恢复迟缓
- **RTT 估算（RFC 6298）**：维护 `SRTT`（平滑）+ `RTTVAR`（偏差）

</v-click>

<v-click>

```text
RTTVAR = (1-β)·RTTVAR + β·|SRTT - R'|    // β = 1/4
SRTT   = (1-α)·SRTT   + α·R'             // α = 1/8
RTO    = SRTT + max(G, 4·RTTVAR)         // 初始/最小 1 秒
```

</v-click>

<v-click>

> 加 `4·RTTVAR` 让 RTO 随网络抖动自适应：网络越不稳，留的余量越大。

</v-click>

<!--
RTO 按 RFC 6298 动态估算。SRTT 是平滑往返时间，RTTVAR 是偏差，公式让 RTO 既不过敏也不迟钝。
-->

---

# Karn 算法与指数退避

解决重传带来的 RTT 二义性

<v-click>

- **二义性**：段被重传后收到 ACK，无法分辨它回应**原始段**还是**重传段**
- **Karn 规则**：重传过的段，其 RTT 样本**一律不采用**（避免错误样本）
- **指数退避**：每次超时触发重传 → `RTO = RTO × 2`，避免在拥塞网络火上浇油

</v-click>

<v-click>

> 一句话：算 RTT 时跳过重传段；连续重传时 RTO 翻倍退让。

</v-click>

<!--
Karn 算法专治重传 ACK 的二义性。指数退避则保证在已经拥塞的网络上不会越重传越堵。
-->

---

# 快速重传：3 个重复 ACK

不等 RTO，提前发现丢包

<v-click>

- 某段丢失后，其后到达的乱序段让接收方**反复回送同一个 ACK**（指向空洞）
- 发送方收到**第 3 个重复 ACK** 就立即重传那个「看起来丢了」的段
- 为何是 3 个而非 1 个：单个重复 ACK 可能只是**轻微乱序**，3 个丢包概率才够高

</v-click>

<v-click>

> 触发速度：快速重传约 1 个 RTT 内即可，远快于等一整个 RTO（秒级）。

</v-click>

<!--
快速重传利用重复 ACK 提前发现丢包。阈值定在 3 是为了兼顾灵敏度与误判率，区别于轻微乱序。
-->

---

# 超时重传 vs 快速重传

两条重传路径，分工不同

<v-click>

| 维度 | 超时重传（RTO） | 快速重传 |
| --- | --- | --- |
| 触发 | 定时器超时仍无 ACK | 收到 3 个重复 ACK |
| 速度 | 慢（至少 1 秒级） | 快（约 1 个 RTT） |
| 场景 | 严重丢包、连 ACK 都收不到 | 单段丢失、后续段仍到达 |
| 角色 | 最终兜底安全网 | 常见丢包的快速恢复 |

</v-click>

<v-click>

> 快速重传不止「重传」：还会触发**快恢复**调整拥塞窗口 —— 属拥塞控制，见下一节。

</v-click>

<!--
两者互补：快速重传应对常见的单段丢失，RTO 是连 ACK 都收不到时的最后兜底。
-->

---

# 选择确认 SACK

只补真正丢失的那一段

<v-click>

- **累积确认的浪费**：发 10 段只丢第 3 段，保守实现可能把第 3 段及之后**全部重传**
- **SACK**（RFC 2018）：握手时用 `SACK-Permitted` 协商，ACK 额外携带「已收到的不连续块」

</v-click>

<v-click>

```text
丢了 seq=1500 那段，其余都到：
  ACK = 1500                          // 累积确认停在空洞前
  SACK 块: [2000–2499], [2500–2999]   // 这些不连续块已收到
→ 发送方只重传 [1500–1999]
```

</v-click>

<!--
SACK 把「连续到哪」和「哪些零散块已收」结合，使重传精确到真正丢失的区间，在高时延、多段丢失的链路收益显著。
-->

---

# 滑动窗口：可靠传输的载体

管理「发了什么、确认到哪、还能发多少」的记账本

<v-click>

```text
|== 已发已确认 ==|== 已发未确认 ==|== 可发未发 ==|== 不可发 ==|
                 ↑               ↑              ↑
              SND.UNA         SND.NXT        窗口右边界
```

</v-click>

<v-click>

- **已发未确认**（飞行中数据）：放重传队列，超时或 3 个重复 ACK 就从这里取出重传
- **可发未发**：窗口内仍有空间，可继续发
- 收到 ACK → `SND.UNA` 右移、窗口「滑动」，腾出空间发新数据

</v-click>

<v-click>

> 窗口**大小** = `min(rwnd, cwnd)`（流量控制 + 拥塞控制），下一节展开。

</v-click>

<!--
滑动窗口是序列号、确认、重传的承载结构。「已发未确认」区是可靠性机制作用的核心区域。
-->

# 流量控制与拥塞控制

两道闸门：护接收方 vs 护网络

---

# 一句话分清两个「控制」

都限速，但理由完全不同

<v-click>

- **流量控制**：防发送方**压垮接收方** —— 保护**对端这一台机器**的缓冲区
- **拥塞控制**：防发送方**压垮网络** —— 保护**整条链路**这个公共资源

</v-click>

<v-click>

| 维度 | 流量控制 | 拥塞控制 |
| --- | --- | --- |
| 保护对象 | 接收方缓冲区 | 网络中间链路 |
| 核心变量 | rwnd（接收窗口） | cwnd（拥塞窗口） |
| 谁维护 | 接收方通告，发送方遵守 | 发送方自己估算 |
| 在报文里传 | 是（16 位 Window 字段） | 否（本地状态） |

</v-click>

<!--
比喻：流量控制像「水杯接水」（杯满喊慢点），拥塞控制像「高速公路限流」（堵车全体减速）。实际车速取两者更慢的。
-->

---

# 流量控制：接收窗口 rwnd

接收方在每个 ACK 里通告「我还能收多少」

<v-click>

- TCP 头里有 16 位 **`Window` 字段** = 接收方愿意再收的字节数（rwnd）
- 随接收方应用读取数据的快慢**动态变化**
- 发送方据此实现滑动窗口：窗口越大，在途数据越多，吞吐越高

</v-click>

<v-click>

> 16 位最大仅 65535 字节，长肥管道不够用 → **窗口缩放选项**（RFC 7323）握手时协商，扩到 1 GB 量级。

</v-click>

<!--
rwnd 是接收方对发送方的明确承诺，必须显式传达。窗口缩放是流量控制跟上现代高速网络的前提。
-->

---

# 零窗口与窗口探测

缓冲区满了怎么办，又如何避免死锁

<v-click>

- 接收方缓冲区满 → 通告 **`Window = 0`**，发送方立即停发
- **死锁风险**：接收方腾出空间后的「窗口已打开」通告**可能丢失** → 双方互等
- **解法**：发送方启动**持续计时器**，超时发**零窗口探测报文**逼对端回最新窗口

</v-click>

<v-click>

> 只要接收方腾出空间，探测就能问出新的 rwnd，连接得以继续 —— 避免永久挂起。

</v-click>

<!--
零窗口本身正常，怕的是「窗口打开」通告丢失导致死锁。持续计时器 + 零窗口探测就是为了打破这种死锁。
-->

---

# 拥塞控制：拥塞窗口 cwnd

发送方自己估算的「网络能吃多少」

<v-click>

- 流量控制只看接收方，**看不见中间网络的拥堵**
- **cwnd** 不出现在任何报文里，纯靠网络反馈（丢包、ACK 节奏、RTT）估算

</v-click>

<v-click>

```text
发送窗口 = min(rwnd, cwnd)
            ↑          ↑
     接收方还能收的   网络还能扛的
```

</v-click>

<v-click>

> 谁更紧听谁的：接收方慢被 rwnd 卡，网络堵被 cwnd 卡，两道闸门同时工作。

</v-click>

<!--
cwnd 是发送方对网络的主观猜测，网络不会主动说「我堵了」，只能靠丢包间接推断，所以它是本地状态，无法由对端通告。
-->

---

# 拥塞控制四阶段

核心：没堵就试探着加速，一堵就果断退让

<v-click>

- **① 慢启动**：从小窗口起步，每 RTT 大致**翻倍**（指数增长）
- **② 拥塞避免**：到达 `ssthresh` 后改为每 RTT 只 **+1 MSS**（线性增长）
- **③ 快重传**：收到 3 个重复 ACK 立即重传，不等超时
- **④ 快恢复**：不回慢启动，cwnd 减半后**直接进入线性增长**

</v-click>

<v-click>

> 「慢启动」一点都不慢：起点低，但增长是**指数级**，几个 RTT 就能撑得很大。

</v-click>

<!--
四阶段是框架。慢启动的「慢」指谨慎起步而非增速慢。快恢复避免了「一丢包就重走慢启动」的浪费。
-->

---

# AIMD：为什么这套规则公平

加性增、乘性减，四阶段的数学内核

<v-click>

- **加性增（AI）**：不拥塞时每 RTT 线性 `+1 MSS` —— 温和试探可用带宽
- **乘性减（MD）**：一拥塞就成比例砍掉 cwnd（Reno 砍一半）—— 剧烈退让让路

</v-click>

<v-click>

```text
cwnd
  │      ╱|        ╱|     ← 拥塞避免：线性 +1/RTT（AI）
ssthresh╱─┼──────╱──┼──
  │   ╱   |×丢包╱   |×丢包 ← 丢包：cwnd 砍半（MD）
  │  ╱ 慢启动  ╱            形成经典锯齿
  └─┴────────┴──────────→ 时间(RTT)
```

</v-click>

<v-click>

> 「缓慢加、剧烈减」的不对称设计，让多条 TCP 连接收敛到公平分配，避免拥塞崩溃。

</v-click>

<!--
AIMD 是 TCP 公平共享带宽的核心。抢得过头的连接更易丢包、被砍得更狠，最终大家趋于均衡，这是互联网无中央调度仍不崩的基石。
-->

---

# 拥塞算法演进

Reno → CUBIC → BBR

<v-click>

| 算法 | 拥塞判据 | 窗口增长 | 丢包后收缩 |
| --- | --- | --- | --- |
| Reno / NewReno | 丢包（被动） | 线性 +1/RTT | 砍半 |
| CUBIC（Linux 默认） | 丢包（被动） | 三次函数 | 减 30% |
| BBR（Google） | 带宽 + RTT 建模 | 按管道容量发 | 不靠丢包 |

</v-click>

<v-click>

- **CUBIC**：只改拥塞避免（慢启动同 Reno），三次函数离峰值远时猛追、近时缓增
- **BBR**：不把丢包当拥塞信号，持续测带宽与最小 RTT —— 弱网 / 长肥管道优势明显

</v-click>

<!--
四阶段是框架，怎么增怎么减各算法有取舍。CUBIC 收缩更温和（减 30%），BBR 模型驱动不被随机丢包欺骗，QUIC 也在用。
-->

# 选型与队头阻塞

可靠不是免费的，它换来的延迟有时致命

---

# TCP vs UDP 全面对比

设计哲学截然相反

<v-click>

| 维度 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接，先握手 | 无连接，直接发 |
| 可靠 / 顺序 | 确认重传 + 序号重排 | 不保证送达 / 顺序 |
| 速度 / 延迟 | 较慢（握手+重传+退让） | 快、低延迟 |
| 首部开销 | 20~60 字节 | 固定 8 字节 |
| 流控 / 拥控 | 有 | 无 |
| 传输单位 | 字节流（无边界） | 数据报（保留边界） |

</v-click>

<v-click>

> 抓三个关键差异：**连接、可靠、有序** —— TCP 全要、UDP 全不要，其余都是连带结果。

</v-click>

<!--
TCP 用复杂度换可靠，UDP 用不可靠换简单与速度。其余差异（速度、首部、流控拥控）都是「连接/可靠/有序」三点的连带结果。
-->

---

# 选型决策：可靠优先还是延迟优先

只问业务：数据能不能丢？迟到的还有没有价值？

<v-click>

```text
        数据必须完整、不能丢？
       是 │              │ 否（可容忍丢包）
          ▼              ▼
       选 TCP        对延迟极敏感？
     文件/网页/      是 │       │ 否
     邮件/数据库       ▼       ▼
                    选 UDP   通常仍选 TCP
                  直播/游戏/  （省心，可靠性
                  VoIP/DNS    由内核兜底）
```

</v-click>

<v-click>

> 实时场景偏爱 UDP：**「丢」比「等」代价小** —— 丢一帧顶多卡一瞬，等重传则后面全被拖住。

</v-click>

<!--
选型不是「谁更先进」，而是问业务对「丢」与「等」的容忍度。UDP 不等于不可靠到不能用，而是可靠性由你按需自建（QUIC 即例证）。
-->

---

# TCP 队头阻塞：可靠有序的代价

严格有序交付的固有副作用

<v-click>

```text
应用层期望:  段1 段2 段3 段4 段5
                      ↓ 段3 在网络中丢失
接收端到达:  [段1][段2][ ✗ ][段4][段5]  ← 4、5 已到却被卡
                      ↓
向应用交付:  段1 段2 …（阻塞，等段3重传）… 然后 3 4 5 一起放行
```

</v-click>

<v-click>

- 段 4、5 明明已收到，却因段 3 没补齐而**无法交付**
- 因为 TCP 只认字节序号，**无法「跳过缺口先交付后面的」**

</v-click>

<!--
队头阻塞（HOL Blocking）是 TCP「按发送顺序、无缺漏交付」承诺的副作用，而非实现缺陷。比喻：TCP 像一条链条，断一环后面全得等。
-->

---

# HTTP/1.1 与 HTTP/2 都困于 TCP 层 HOL

「HTTP/2 解决了队头阻塞」只对了一半

<v-click>

- **HTTP/1.1 应用层 HOL**：一条连接上请求**串行**排队，前一个没回后面干等
- **HTTP/2 多路复用**：一条 TCP 上并发交错传输，**消除了应用层 HOL**
- **但**：所有流塞进**同一条 TCP**，阻塞从应用层**下沉到传输层**

</v-click>

<v-click>

> 只要流 A 的一个 TCP 包丢了，B、C 的数据明明已到也被一起卡住 —— 因为 TCP 不知道「流」的存在。

</v-click>

<!--

# HTTP/2 弱网下反不如 HTTP/1.1

单连接无处分摊丢包

<v-click>

- **HTTP/1.1**：通常开**最多 6 条并行 TCP**，丢包分摊到多条，一条卡了还有五条
- **HTTP/2**：全压在**一条 TCP**，一个包丢，全连接所有流齐卡
- 在约 **2% 丢包**下，HTTP/1 用户体验有时反而更好

</v-click>

<v-click>

> 根治之道：换掉传输层，让它「认识流」 —— 这正是 QUIC 的诞生动机。

</v-click>

<!--
这是 HTTP/2 的隐痛：高丢包下单连接无处可逃。要根治必须换传输层，让它认识「流」，于是有了 QUIC。
-->

---

# QUIC 如何用 UDP 绕开

另起炉灶：在 UDP 上重建传输层

<v-click>

- TCP 改不动（内核实现、被全球中间设备硬编码）→ QUIC 构建在 **UDP** 之上
- 在用户态自实现连接、可靠传输、丢包重传、拥塞控制、流量控制
- **关键**：QUIC 让**每个流相互独立** —— 一个流丢包只阻塞那一个流，其余照常交付

</v-click>

<v-click>

> 因果链：TCP 有序交付 → TCP 层 HOL → HTTP/2 仍受困 → QUIC（UDP + 独立流）→ HTTP/3。

</v-click>

<!--
QUIC 在 UDP 上重建传输层，让每个流独立处理丢包，从根上绕开 TCP 层 HOL。这是从 TCP/UDP 通往 QUIC/HTTP3 的演进主线。
-->

# 小结

IP 送到机器，传输层送到进程，六块拼图

<v-click>

- **端口 / 复用分用**：端口定位进程，socket 四元组唯一确定连接
- **UDP**：8 字节首部、四不保证，用不可靠换无握手低延迟
- **TCP 握手挥手**：三次握手同步 ISN、四次挥手分方向关、TIME_WAIT 等 2MSL
- **可靠 + 拥塞**：编号→确认→重传→排序去重；rwnd 护接收方、cwnd 护网络（AIMD）
- **选型 / HOL**：能丢选 UDP、要稳选 TCP；TCP 层队头阻塞催生 QUIC

</v-click>

<v-click>

> 理解「TCP 可靠却会卡、为何把传输层搬到 UDP 重写」，就握住了通往 QUIC / HTTP/3 的钥匙。

</v-click>
