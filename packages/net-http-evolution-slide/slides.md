# 这一章讲什么

一条主线：**每次升级都在解决「更快搬运资源」**

<v-click>

- **版本演进史**：0.9 → 1.0 → 1.1 → 2 → 3，变的是线格式，不变的是语义
- **1.1 瓶颈**：应用层队头阻塞、6 连接上限、头部冗余与被迫的 hack
- **HTTP/2**：二进制分帧 + 多路复用，消灭应用层队头阻塞
- **HPACK 与推送**：头部压缩抗 CRIME，服务器推送已废弃 → 103 Early Hints
- **HTTP/3 与 QUIC**：消灭 TCP 层队头阻塞、连接迁移、0-RTT
- **版本对比与实践**：三代横向对比，过时的旧优化要主动拆掉

</v-click>

<v-click>

> 抓住「队头阻塞」这条暗线，三代协议的取舍就全串起来了。

</v-click>

<!--
全程围绕「队头阻塞」与「连接/头部模型」两条线展开，每一项性能特性都能落到前端能感知的点上。
-->

# 版本演进史

内核几乎不变，外层不断为性能加码

---

# HTTP 的三条设计哲学

理解一切的前提：什么从未改变

<v-click>

- **文本协议（人类可读）**：1.1 之前用 `telnet` 就能手敲请求；HTTP/2 起改二进制是性能取舍
- **请求—响应模型**：客户端先发，服务端被动应答，一来一回
- **无状态（stateless）**：每个请求独立可理解，状态靠 Cookie/Token 额外补

</v-click>

<v-click>

> 2022 年 RFC 把**语义**（方法/状态码/头部含义）和各版本**线格式**彻底拆开 —— 从 1.1 到 3，变的只是字节怎么排，`GET`/`200` 始终一致。

</v-click>

<!--
语义 vs 语法是贯穿全史的暗线：RFC 9110 管语义，9112/9113/9114 各管一个版本的报文格式。前端几乎感知不到版本切换，正源于此。
-->

---

# HTTP/0.9 与 1.0

从「单行协议」到「可扩展」

<v-click>

- **0.9（1991）**：一行 `GET /page.html`，响应直接是 HTML；无头部、无状态码、无版本号，只能传 HTML
- **1.0（1996, RFC 1945）**一次性补上四大件：
  - **版本号** `HTTP/1.0` + **状态码**（`404`/`500`，程序能判断成败）
  - **HTTP 头部**（`User-Agent`/`Server`）+ **`Content-Type`**（能传图片/视频/JSON）

</v-click>

<v-click>

> 1.0 **致命短板**：默认每个资源新建一条 TCP 连接、用完即关 —— 反复三次握手 + 慢启动成了性能黑洞，这正是 1.1 要扶正的红利。

</v-click>

<!--
0.9 是「能跑就行」的最小内核；1.0 补上头部、状态码、内容类型让协议可扩展，但默认非持久连接埋下性能黑洞。1.0 时期 Connection: keep-alive 是非标准实验，1.1 才扶正。
-->

---

# HTTP/1.1：标准化与持久化

1997 至今，生命周期最长的版本

<v-click>

| 特性 | HTTP/1.0 | HTTP/1.1 |
| --- | --- | --- |
| 连接 | 默认非持久 | **默认持久** |
| `Host` 头 | 无 | **必填**（虚拟主机） |
| 管线化 | 无 | 支持（浏览器默认关） |
| 分块传输 | 无 | `Transfer-Encoding: chunked` |
| 缓存控制 | `Expires` | `Cache-Control`/`ETag` |
| 范围请求 | 无 | `Range` / `206` |

</v-click>

<v-click>

> **`Host` 必填**是虚拟主机的基石（一个 IP 跑成百上千站）；**分块传输** `chunked` 让服务端边生成边发，不必预知正文总长。

</v-click>

<!--
1.1 解决的核心是「连接效率」与「一机多站」。Host 头撑起整个共享主机与 CDN 架构，分块传输让动态内容流式输出。2022 年由 RFC 9110-9112 重组升格为 Internet Standard（STD 97）。
-->

---

# 版本时间线

贯穿始终的驱动力只有一个词：复杂度

<v-click>

| 版本 | 年份 | 关键 RFC | 一句话特征 |
| --- | --- | --- | --- |
| 0.9 | 1991 | 无 | 单行 GET，无头部无状态码 |
| 1.0 | 1996 | RFC 1945 | 头部/状态码/Content-Type |
| 1.1 | 1997~ | 9110~9112 | 持久连接、Host、分块、缓存 |
| 2 | 2015 | RFC 9113 | 二进制分帧、多路复用、HPACK |
| 3 | 2022 | RFC 9114 | 基于 QUIC/UDP，消除 TCP 队头阻塞 |

</v-click>

<v-click>

> 网页从「一篇文档」长成「HTML + 几十个资源」，每次升级都在回答：如何在一组连接上更快、更并发地搬资源。

</v-click>

<!--
SPDY 是 HTTP/2 的前身。三大版本并存不互相废弃，共享同一套语义。理解这条脉络，是看懂后续每项优化「为何存在」的前提。
-->

---

# 版本如何协商

客户端与服务端怎么「商量」用哪个版本

<v-click>

- **`Upgrade` 头（明文 h2c）**：实践中几乎不用，浏览器只在加密连接上启用 HTTP/2
- **ALPN over TLS（主流）**：在 TLS 握手的 `ClientHello` 里列出 `h2`/`http/1.1`，**握手内一次性协商、零额外往返**
- **`Alt-Svc` 头**：服务端响应里广播 `Alt-Svc: h3=":443"`，客户端**后续请求**才升级到 HTTP/3

</v-click>

<v-click>

> HTTP/3 跑在 UDP 上、无法靠 TLS-ALPN 直接协商，`Alt-Svc`（及 HTTPS DNS 记录）是其主要发现路径。

</v-click>

<!--
版本号首位代表报文线格式，次位代表兼容的最高次版本。所以 HTTP/2、HTTP/3 严格说是新主版本（新线格式），但共享 RFC 9110 的语义。
-->

持久连接没解决的「排队」问题

# 持久连接只省了「握手」

keep-alive 解决了什么、没解决什么

<v-click>

- **1.1 默认持久连接**：一条 TCP 建立后保持打开，可被多个请求复用，省去反复握手
- 服务端用 `Keep-Alive: timeout=5, max=100` 告知保活策略

</v-click>

<v-click>

- **但请求依然串行**：在一条连接上，客户端必须等上一个响应**完整回来**，才能发下一个
- 下载第 N 个资源前要苦等前 N-1 个 —— 这正是队头阻塞的雏形

</v-click>

<!--
持久连接的代价：连接即使空闲也占用服务器文件描述符与内存，所以浏览器不能对单域名无限开连接。它只省了握手，没改变「串行」本质。
-->

---

# 管线化：一个失败的修补

不等响应就连发，但必须按序返回

<v-click>

```text
客户端 →  请求1  请求2  请求3   （一口气发出，不等响应）
服务端 ←  响应1  响应2  响应3   （但必须按这个顺序返回！）
```

</v-click>

<v-click>

- **响应必须按请求顺序返回**：队首「请求 1」慢，响应 2/3 早就绪也只能干等
- 叠加坏代理普遍有 bug、难正确实现、收益微薄
- 结论：**浏览器默认关闭**，被 HTTP/2 多路复用取代

</v-click>

<!--
理论上能省请求间的等待，但从设计上就埋了雷。队首慢响应卡死后面所有响应，这就是应用层队头阻塞。
-->

---

# 应用层 HOL ≠ TCP 层 HOL

最高频考点：两种队头阻塞的边界

<v-click>

- **应用层 HOL**（本节主角）：HTTP 协议要求**响应按序返回**造成的阻塞 → **HTTP/2 多路复用即可解决**
- **TCP 层 HOL**（后面 QUIC 详述）：即使多路复用，TCP 丢一个包，该连接上**所有流**都得停等重传 → 要靠 **HTTP/3 的 QUIC** 才能根治

</v-click>

<v-click>

> 一句话区分：应用层 HOL 是「协议规定排队」，TCP 层 HOL 是「字节流丢包连坐」。

</v-click>

<!--
这是整章最容易混的两个概念，务必分清楚。HTTP/2 只解决了前者，后者是 TCP 字节流的本质限制，HTTP/2 无能为力。
-->

---

# 并发连接数限制：每域名约 6 条

单连接串行 + 管线化不可用，只能多开连接凑并发

<v-click>

| 时期 | 每域名并发连接数 |
| --- | --- |
| 早期浏览器 | 2 ~ 3 |
| 现代浏览器（主流） | **6** |

</v-click>

<v-click>

- **为何限制**：保护服务器（防连接爆炸 + DoS 误判），保护网络与客户端自身
- **后果**：现代页面上百个资源，**6 条根本不够分**，多出来的只能排队 → 逼出各种 hack

</v-click>

<!--
每条连接都有内存、握手、拥塞控制状态的成本，盲目增多反而劣化吞吐。6 条连接不够分，是前端被迫发明 hack 的直接原因。
-->

---

# 头部冗余：重复且无压缩

1.1 的第二大瓶颈在 Headers

<v-click>

- **每个请求都重复携带**几乎相同的头部：`User-Agent`、`Accept`、`Referer`
- **Cookie 是重灾区**：一旦写入，每个同域请求（哪怕取一张几百字节小图）都原样重发整串
- **完全没有压缩**：正文可以 gzip，头部不行（纯文本明文传输）

</v-click>

<v-click>

> 对几十上百个小资源来说，**重复头部的体积甚至可能超过正文本身**，移动网络上尤其浪费。

</v-click>

<!--
头部冗余是 HTTP/2 用 HPACK 重点要解决的问题。Cookie 较大时，每个请求都把整坨 Cookie 再发一遍，纯属浪费上行带宽。
-->

---

# 前端被迫发明的 hack

减少请求数 / 绕开连接上限的四套路

<v-click>

| Hack | 做法 | 目的 |
| --- | --- | --- |
| 域名分片 | 资源拆到 `img1/img2.x.com` | 突破 6 连接上限 |
| 雪碧图 | 小图拼成大图 + 裁切 | N 个图片请求合 1 |
| 文件合并 | 多个 JS/CSS 打成大包 | 减少请求数 |
| 资源内联 | `data:` URI 嵌进 HTML | 干脆省掉请求 |

</v-click>

<v-click>

> 本质都是「用打包/合并对冲请求数与连接数限制」—— 在当年有效，但都带副作用。

</v-click>

<!--
这四套路在 HTTP/1.1 时代确实有效，是当年前端性能优化的标配。但每一个都以牺牲缓存粒度、强制提前加载为代价。
-->

---

# hack 的代价与 HTTP/2 反噬

当年的优化，升级后反成负担

<v-click>

- **牺牲缓存效率**：合并/内联让「各自独立缓存」变成「一损俱损」，改一处全量重下
- **强制提前加载**：把用不到的资源塞进首屏包，浪费带宽
- **增加复杂度**：雪碧图坐标、分片域名都要工具链维护

</v-click>

<v-click>

> **HTTP/2 下这些 hack 反噬**：域名分片被「连接合并」抵消甚至有害；合并/内联反而拖慢缓存命中。升级时要主动**拆掉**它们。

</v-click>

<!--
HTTP/2 用一条连接的多路复用就能高效并发，请求数不再是瓶颈。此时旧 hack 失去前提，反成累赘。
-->

# HTTP/2 分帧与多路复用

换传输方式，不换语义

# 设计目标：换传输，不换语义

HTTP/2 的出发点是纯粹的性能

<v-click>

- `GET` 还是 `GET`，`404` 还是 `404`，`Content-Type` 还是 `Content-Type`
- 改变的只是这些信息在网络上**如何被编码与传输**
- 瞄准三件事：**多路复用**、**HPACK 头部压缩**、请求优先级与推送

</v-click>

<v-click>

> 对前端意味着：你写的 `fetch()` / `<img src>` 一行不用改，浏览器与服务器协商出 HTTP/2 后自动启用。

</v-click>

<!--
RFC 反复强调：HTTP/2 不以任何方式修改 HTTP 的应用语义。所有核心概念原样保留，这也是版本切换对前端透明的根本原因。
-->

---

# 三层模型：帧、消息、流

理解 HTTP/2 的钥匙

<v-click>

| 概念 | 定义 | 类比 |
| --- | --- | --- |
| 帧 frame | 最小通信单位，含帧头 | 一节车厢 |
| 消息 message | 一个请求/响应的完整帧序列 | 一列列车 |
| 流 stream | 连接内一条双向字节流 | 一条铁轨 |

</v-click>

<v-click>

- 一条 TCP 连接里可有任意多条流，每条流跑消息，消息由帧组成
- **关键**：不同流的帧可在同一连接上**交错传输**，到对端按 Stream ID 重新归位

</v-click>

<!--
帧、消息、流层层包含。多条轨道（流）上跑着列车（消息），列车由车厢（帧）组成，车厢可以在一条连接上交错运输。
-->

---

# 二进制分帧层

HTTP/2 性能红利的根基

<v-click>

```text
单条 TCP 连接
├─ stream 1 ── HEADERS ── DATA ── DATA
├─ stream 3 ── HEADERS ── DATA
└─ stream 5 ── HEADERS ── DATA ── DATA

连接上的实际字节（帧被交错）：
… [s1 HEADERS] [s3 HEADERS] [s1 DATA] [s5 HEADERS] [s3 DATA] …
```

</v-click>

<v-click>

- 文本好读但难高效解析、易歧义；**二进制定长帧头**便于高速、无歧义地切分重组
- 这一层夹在 socket 与 HTTP API 之间，对上层应用**完全透明**

</v-click>

<!--
HTTP/1.1 逐行解析文本；HTTP/2 把通信切成二进制帧。分帧与重组由协议栈代劳，应用层无感知。
-->

---

# 帧头与帧类型

固定 9 字节帧头 + 变长负载

<v-click>

| 帧类型 | 作用 | 受流控 |
| --- | --- | --- |
| `HEADERS` | 开启流、携带头部块 | 否 |
| `DATA` | 携带报文体 | **是** |
| `SETTINGS` | 协商连接参数 | 否 |
| `WINDOW_UPDATE` | 增大流量窗口 | 否 |
| `RST_STREAM` / `GOAWAY` | 终止流 / 优雅关连接 | 否 |
| `PUSH_PROMISE` / `PRIORITY` | 推送预告 / 优先级(已废弃) | 否 |

</v-click>

<v-click>

> 帧头：Length(24) + Type(8) + Flags(8) + Stream ID(31)。伪头部 `:method`/`:status` 取代了文本请求行。

</v-click>

<!--
HTTP/2 共定义 10 种帧，前端最常打交道的是 HEADERS 和 DATA。伪头部以冒号开头，和普通头一起被 HPACK 压进 HEADERS 帧。
-->

---

# 多路复用：消灭应用层队头阻塞

HTTP/2 最重要的能力

<v-click>

```text
HTTP/1.1（6 连接，各自串行）   HTTP/2（1 连接，帧级并发）
conn1: [req1]→[resp1]          stream1 ┐
conn2: [req2]→[resp2]          stream3 ├─ 帧交错于单条连接
conn3: [req3]→[resp3]          stream5 ┘   同时在途、互不阻塞
```

</v-click>

<v-click>

- 流 9 不必等流 7 完成，二者的帧交错收发
- 一个慢响应不再卡住后面的快响应 → **应用层队头阻塞被彻底消除**
- 不再需要域名分片、雪碧图等绕过并发限制的老技巧

</v-click>

<!--
把一条 HTTP 消息拆成独立帧、交错发送、再到对端重组。并发流数可达数百，远超 HTTP/1.1 的 6 连接。
-->

---

# 流的标识、并发与流控

多流共享一条连接的秩序

<v-click>

- **标识符**：客户端发起用**奇数** ID，服务端用**偶数**，`0` 号保留给连接级控制帧
- **并发上限**：由 `SETTINGS_MAX_CONCURRENT_STREAMS` 声明，超限触发 `REFUSED_STREAM`
- **流量控制**：基于窗口的信用机制，**仅 `DATA` 帧受限**，初始 65,535 字节，靠 `WINDOW_UPDATE` 增窗，分流级与连接级两层、双向、不可关闭

</v-click>

<v-click>

> 流量控制是**应用层、逐跳、非端到端**的，解决「连接内多条流之间」的带宽分配，不替代 TCP 拥塞控制。

</v-click>

<!--
流量控制防止某条流（如大文件下载）独占带宽饿死其他流。它和 TCP 拥塞控制分工不同，二者并存。
-->

---

# 流优先级：实践中支持参差

一个「看起来很美」但不可靠的特性

<v-click>

- **原始方案**：流声明对另一条流的**依赖** + 1~256 的**权重**（如权重 12:4 按 3:1 分配资源）
- **已废弃**：RFC 9113 废弃该方案，改推 RFC 9218 的扩展优先级（`urgency` + `incremental`）
- 规范明确：优先级始终是**「传输偏好」非强制**，各端实现差异很大

</v-click>

<v-click>

> 结论：**不能依赖优先级做正确性保证**，只能视作性能提示。

</v-click>

<!--
即便在旧方案下，规范也说流依赖与权重只是传输偏好，不保证任何处理顺序。现实中各浏览器、CDN 实现参差，别当真。
-->

# 单连接 vs 多连接

一个 origin 一条长连接的收益

<v-click>

- **更少握手**：TCP 三次握手与 TLS 握手只做一次，省下大量首字节延迟
- **更好的复用**：HPACK 压缩状态、TLS 会话在同一连接内持续受益
- **更省资源**：服务端维护的连接数、内存占用大幅下降
- **更合理的拥塞控制**：单连接的拥塞窗口探测优于多连接互相争抢

</v-click>

<v-click>

> 官方结论：丢包场景下队头阻塞的负面影响，被压缩和优先级带来的收益**抵消** —— 整体仍划算。

</v-click>

<!--
HTTP/2 对每个 origin 只用一条持久连接。相比 HTTP/1.1 的 6 连接，握手、内存、拥塞控制全面占优。
-->

---

# 尚未解决：TCP 层队头阻塞

HTTP/2 埋下的新瓶颈

<v-click>

- TCP 提供**有序、可靠的字节流**：一个段丢失，后续已到达的段也必须在内核缓冲区**等重传补齐**
- HTTP/2 把所有流复用在**同一条** TCP 连接上
- 结果：**单个丢包阻塞该连接上所有流**，哪怕这些流的数据已完整到达

</v-click>

<v-click>

> 连接数越少，单次丢包的「连坐」范围反而越大 —— 高丢包/弱网下，HTTP/2 甚至可能不如多连接的 HTTP/1.1。

</v-click>

<!--
TCP 不知道「流」的存在，它只认字节序。这正是 HTTP/3 放弃 TCP、改用基于 UDP 的 QUIC 的根本动机。
-->

# 为什么头部需要压缩

HTTP/2 的多路复用把头部冗余进一步放大

<v-click>

- 同站点每个请求都重复 `User-Agent`、`Accept`、`Referer` 等几乎一致的字段
- **Cookie 重灾区**：写入后每个请求（图片/字体/API）都原样重发整串
- 一个页面几十上百个子资源，头部开销**线性累加**

</v-click>

<v-click>

> 经验数据：每次传头部约 **500-800 字节**开销，**带 Cookie 时常多出数 KB** —— 这就是 HPACK 要解决的。

</v-click>

<!--
请求体可以 gzip，但 HTTP/1.1 头部始终明文不压缩。多路复用让一条连接并发跑很多请求，头部冗余被放大，必须有专门方案。
-->

---

# HPACK 三件套

RFC 7541：静态表 + 动态表 + Huffman

<v-click>

- **静态表**：双端硬编码的 **61 条**常见头（`:method`、`:path`、`content-type`），命中直接用小整数索引代替整行
- **动态表**：连接内从空开始增量填充，**FIFO 淘汰**，默认上限 **4096 字节**；重复头第二次起只发一个索引号
- **Huffman 编码**：对头部字符串用针对 HTTP 频率优化的静态码表压缩

</v-click>

<v-click>

> 三者叠加削减约 **85%-88%** 头部体积 —— 动态表正是 Cookie 重发问题的根治手段。

</v-click>

<!--
静态表消掉「人人都发的通用头」，动态表消掉「本连接重复发的头」，Huffman 再压剩余字面量字节。慢速链路单页加载可因此快几十到上千毫秒。
-->

---

# 为何不直接用 gzip 压头部

这是安全问题，不是性能问题

<v-click>

- SPDY 起初正是用 gzip 压头部，但很快被推翻
- **CRIME / BREACH 压缩侧信道攻击**：攻击者注入可控字符串，观察压缩后**长度变化**，逐字符猜出同上下文里的机密（如会话 Cookie）
- 可把**指数级暴力破解降为线性级**

</v-click>

<v-click>

> CRIME 针对请求头压缩，BREACH 针对响应体压缩 —— 这就是「头部不能简单套通用压缩」的根本原因。

</v-click>

<!--
通用压缩的本质是「重复内容压得更短」，恰好被攻击者利用。HPACK 的设计针对性地削弱了这条攻击路径。
-->

---

# HPACK 如何抗 CRIME

整值匹配 + 永不索引字面量

<v-click>

- **整值匹配，而非逐字符**：动态表以**整条头部字段值**为索引单位，攻击者必须一次猜中整个值才能观察长度变化，成本剧增
- **永不索引字面量**：对 `cookie`、`authorization` 等敏感头标记为「绝不存入动态表」，彻底关闭压缩探测面

</v-click>

<v-click>

> HPACK 是「缓解」而非「根除」：低熵值仍可能被攻破，高熵值基本无法还原 —— 故规范建议敏感头用永不索引。

</v-click>

<!--
整值匹配让逐字符猜测失效；永不索引则对最敏感的头彻底关闭压缩。这是 HPACK 相对通用压缩的核心安全设计。
-->

# 服务器推送：一个被废弃的特性

高频考点：务必记住结论是「已废弃」

<v-click>

```text
PUSH_PROMISE 帧：声明「我要推这个 URL」（须在父响应数据前到达）
       ↓
服务器发起流上推送资源内容
       ↓
客户端不需要（如本地已缓存）→ 用 RST_STREAM 拒收
```

</v-click>

<v-click>

- **设想**：请求 `index.html` 时顺手把 `app.css`/`app.js` 一并推下，省去二次请求往返
- **现实**：`PUSH_PROMISE` 帧 + 服务器发起流，听起来很美

</v-click>

<!--
服务器推送允许在客户端仅请求一个 HTML 时，主动把后续大概率要用的资源连同推给浏览器。下一页讲它为何被淘汰。
-->

# 服务器推送为何被废弃

设想收益几乎无法兑现

<v-click>

- **采用率极低**：仅约 **1.25%** 站点用，后续降到 **0.7%**
- **收益低且常致性能回退**：往往推了浏览器**已缓存**或**当前页用不上**的资源（**推送浪费**）
- **缓存难处理**：推送在浏览器查缓存**之前**就发生，极易重复推送
- **HTTP/3 未跟进**：很多 H3 端根本没实现，等于在新协议上已淘汰

</v-click>

<v-click>

> 结论：**Chrome 106（2022）默认移除**，其它 Chromium 系跟进。别当现行最佳实践。

</v-click>

<!--
正确配置推送（推什么、推多少、何时停）门槛高，收益却不稳定。这是它被淘汰的综合原因，高频考点。
-->

# 替代方案：103 Early Hints

RFC 8297：提示而非硬塞

<v-click>

```http
HTTP/1.1 103 Early Hints
Link: </style.css>; rel=preload; as=style
Link: <https://cdn.example.com>; rel=preconnect

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
```

</v-click>

<v-click>

- 服务端**不主动塞资源**，而是先发信息性响应，用 `Link` 头**提示**可 `preload`/`preconnect`
- **由浏览器自主决定**是否抓取（自然会查缓存、不浪费）
- 浏览器**只处理第一个** 103；建议仅在 HTTP/2+ 发送

</v-click>

<!--
本质区别：提示（hint）让浏览器去拉，推送（push）是服务器硬塞。前者保留了缓存感知与客户端控制权。
-->

# HTTP/3 与 QUIC

把多路复用下沉到传输层

---

# 为什么 HTTP/2 还不够

TCP 层队头阻塞再回顾

<v-click>

- HTTP/2 消除了**应用层**队头阻塞，但把所有流塞进**同一条 TCP 连接**
- TCP 是严格有序的字节流：一个段丢失，后到的段即使已在内核缓冲区，也得**等重传补齐**
- 流 A 丢一个包，**B、C 已到达却被卡住** —— TCP 不认「流」，只认字节序

</v-click>

<v-click>

> 要根治，必须让传输层「认识流」，对每个流独立处理丢包 —— 而 TCP 做不到。于是 QUIC 登场。

</v-click>

<!--
这正是 HTTP/2 在弱网/高丢包下有时不如多连接 HTTP/1.1 的原因。一次丢包，全连接所有流一起阻塞。
-->

---

# QUIC 是什么

跑在 UDP 上的新传输层

<v-click>

```text
   HTTP/2 栈            HTTP/3 栈
┌──────────────┐    ┌──────────────────────┐
│    HTTP/2    │    │       HTTP/3         │
│     TLS      │    │  QUIC(含TLS1.3、多路  │
│     TCP      │    │  复用、可靠传输、流控)│
│     IP       │    │       UDP / IP       │
└──────────────┘    └──────────────────────┘
```

</v-click>

<v-click>

- QUIC（RFC 9000）在 UDP 之上**自己实现**连接管理、可靠传输、丢包检测、拥塞控制 —— 把 TCP 能力在**用户态**重造一遍
- **为何不改造 TCP**：TCP 在内核里、海量中间设备有硬编码假设，加新特性等内核升级可能要十年；QUIC 在用户态随浏览器/服务端发版，快速迭代

</v-click>

<v-click>

> UDP 几乎在所有网络里都能通行，是承载新协议的理想「管道」。QUIC 内置 TLS 1.3，连接建立即加密，没有「明文 QUIC」。

</v-click>

<!--
HTTP/3 = HTTP over QUIC，语义与前两代完全一致，只换底层传输（RFC 9114）。选 UDP 的核心理由是可演进性：用户态协议栈可像普通软件快速发版。
-->

---

# QUIC 如何解决队头阻塞

流是一等公民

<v-click>

| 场景 | HTTP/2 over TCP | HTTP/3 over QUIC |
| --- | --- | --- |
| 流 A 一个包丢失 | B、C 也被卡住 | 只有 A 等重传 |
| 传输层认识「流」吗 | 否，只认字节序 | 是，流是一等公民 |
| 弱网/高丢包表现 | 丢包放大 | 影响隔离在单流内 |

</v-click>

<v-click>

> QUIC 把队头阻塞从「连接级」降到「**流级**」。注意：单个流内部仍有序可靠，丢包仍阻塞「这个流自己」，只是不再殃及其他流。

</v-click>

<!--
QUIC 把「流」做成传输层原生概念，每个流独立维护丢包检测与重传状态。这是 QUIC 相对 HTTP/2 最核心的传输层改进。
-->

---

# 连接建立：1-RTT 与 0-RTT

握手合并带来的提速

<v-click>

| 阶段 | TCP + TLS1.3 | QUIC |
| --- | --- | --- |
| 传输层握手 | TCP 三次握手 1 RTT | 与 TLS 合并 |
| 加密握手 | TLS 1.3 1 RTT | 合并进 QUIC 握手 |
| 首次连接合计 | 约 **2 RTT** | **1 RTT** |
| 恢复已知源 | 传输层仍需 TCP 握手 | **0-RTT** 捎带请求 |

</v-click>

<v-click>

> 对回头客，QUIC 缓存握手参数，第一个数据包里就把请求一起发出 —— **零往返即开始传业务数据**。

</v-click>

<!--
QUIC 把 TLS 1.3 握手融进自己的握手，加密与连接建立一次完成。移动端高延迟场景，省下的往返直接转化为更快首屏。
-->

---

# 0-RTT 的安全代价

重放攻击的硬约束

<v-click>

- 0-RTT 数据在握手完成前发出，理论上可被**重放（replay）**
- 规范要求：**只有幂等且无副作用的请求**（如 `GET`）才适合走 0-RTT
- **绝不能**把「下单」「转账」这类有副作用的写操作放进 0-RTT 包

</v-click>

<v-click>

> 这是协议层的硬约束 —— 前端发起预连接/预请求时需留意，别把写操作塞进 0-RTT。

</v-click>

<!--
0-RTT 是性能与安全的权衡。可重放是它的固有风险，所以协议层强制限定只用于幂等无副作用的请求。
-->

---

# 连接迁移：换网不断连

移动端杀手锏

<v-click>

- **TCP 由「四元组」标识**：源 IP/端口 + 目的 IP/端口。手机 Wi-Fi 切蜂窝，源 IP 一变，四元组失效，**TCP 直接断开**
- **QUIC 用 Connection ID** 标识连接，与网络路径无关
- IP/端口变了，只要 Connection ID 不变，服务端就知道「还是同一连接」，**无缝继续**

</v-click>

<v-click>

> 进电梯、出地铁、Wi-Fi 回落蜂窝……视频流、长轮询、实时推送在切网时**不中断**，体验连续性大幅提升。

</v-click>

<!--
移动用户网络切换是常态，每次切换在 TCP 时代都意味着断连重连。连接迁移让长连接在切网时无缝继续，对移动端意义重大。
-->

---

# QPACK：乱序世界里的 HPACK

HPACK 在 QUIC 上的适配版（RFC 9204）

<v-click>

- **问题**：HPACK 依赖 TCP 有序交付，动态表增删必须按序应用；而 QUIC 的流**乱序到达**
- **方案**：头部数据仍随各自请求流走，但**动态表更新指令走两条专用单向流**（编码器流 / 解码器流）
- **阻塞流**：引用尚未送达的表项时，该流短暂等待，**而不阻塞其他流**

</v-click>

<v-click>

> 用 `SETTINGS_QPACK_BLOCKED_STREAMS` 在**压缩率**与**队头阻塞风险**间权衡：越激进用动态表，压缩越好但等待风险越高。

</v-click>

<!--
HPACK 假设「头部按序到」，QPACK 承认「头部可能乱序到」。它把易引发依赖的动态表更新拆到独立流，保住压缩收益又不破坏流间互不阻塞。
-->

# HTTP/3 怎么协商

先 HTTP/2，再靠 Alt-Svc 升级

<v-click>

```http
HTTP/2 200 OK
content-type: text/html
alt-svc: h3=":443"; ma=3600, h2=":443"; ma=3600
```

</v-click>

<v-click>

1. 浏览器照常用 HTTPS（HTTP/2 over TCP）连接源站
2. 服务端响应头带 `Alt-Svc`，声明「我在某端口也提供 HTTP/3」
3. 浏览器记下，**后续请求**才异步尝试 QUIC；连不通自动回退到 HTTP/2

</v-click>

<v-click>

> DevTools → Network → Protocol 列：`h3` 即 HTTP/3。冷启动首次往往仍是 `h2`，刷新后才升级 —— 正常现象。

</v-click>

<!--
浏览器不会对新域名直接发 HTTP/3，因为不知道对方支不支持，且 UDP 443 可能被封。`h3=":443"` 中 ma 是有效期，早期草案曾用 h3-29 等带版本号标识，现已统一为 h3。
-->

# 过时甚至有害的旧优化

升级 HTTP/2+ 后，前提消失，技巧反成负担

<v-click>

- **域名分片**（直接有害）：多域名 = 多条连接各自重走 DNS + 握手；**破坏单连接多路复用与统一调度**，反而更慢
- **雪碧图 / 文件合并**（破坏缓存粒度）：任一小改动让整个大文件缓存失效，用户被迫全量重下
- **资源内联**（无法独立缓存）：内联资源随 HTML 变化重传，多页内联 = 重复传输

</v-click>

<v-click>

> 这些技巧初衷都是绕开 HTTP/1.1 的「单连接串行 + 限并发 + 每请求开销大」—— 升级后应**移除**。

</v-click>

<!--
域名分片会被连接合并抵消甚至有害；合并/内联破坏细粒度缓存。极少量首屏关键 critical CSS 仍可酌情内联，是权衡而非默认。
-->

---

# HTTP/2+ 现代最佳实践

信任多路复用，回归细粒度

<v-click>

- **细粒度拆包**：按路由/特性拆 chunk，「改一处只失效一块」，配合内容哈希 + `immutable`
- **信任浏览器多路复用**：同源资源放一条连接并行下载，**别再手动分片或盲目合并**
- **`preconnect`**：对跨源关键来源（字体、关键 API）提前完成 DNS+TCP+TLS
- **`preload`**：对当前页确定要用、但发现得晚的关键资源提前拉，但**别滥用**

</v-click>

<v-click>

> 慎用 Server Push（主流浏览器已弃用），优先 `preload` / `103 Early Hints`。

</v-click>

「细粒度」不等于「碎到几百个文件」——目标是按变更频率分层（稳定第三方库一块、业务代码一块、按路由再拆），而非无脑拆最细。

# 如何检测站点的 HTTP 版本

三种实用手段

<v-click>

```bash
# 强制 HTTP/2，只看响应头 → 首行类似 HTTP/2 200
curl --http2 -I https://example.com

# 尝试 HTTP/3（需 curl 带 QUIC 支持）→ HTTP/3 200
curl --http3 -I https://example.com

# nghttp2 工具集，-n 丢响应体，-v 打印协商过程
nghttp -nv https://example.com
```

</v-click>

<v-click>

- **Chrome DevTools**：Network 面板 → 表头右键勾选 **Protocol** 列
- 值为 `http/1.1` / `h2` / `h3`（QUIC），一眼看清实际协商的协议

</v-click>

# 升级前提与务实路线

HTTPS + HTTP/2 是基线，HTTP/3 交给 CDN

<v-click>

- **两个硬前提**：HTTP/2 浏览器**只在 HTTPS 上启用**；HTTP/3 要求 **UDP 443 可达**（企业防火墙可能拦）
- **基线**：开 **HTTPS + HTTP/2** —— 多路复用 + 头部压缩，几乎零业务改造成本
- **进阶**：**HTTP/3 交给 CDN/反向代理一键开启**（Cloudflare、Nginx/quiche、Caddy）

</v-click>

<v-click>

> 升级后必做：**回收 HTTP/1.1 时代的旧优化** —— 撤域名分片、放宽过度合并、减内联，改走细粒度拆包，才能真正吃到红利。

</v-click>

<!--
务实路线：HTTPS + HTTP/2 是当下所有站点的默认起点；HTTP/3 由 CDN 一键开启，前端基本无感，弱网与移动场景收益明显。
-->

# 小结

三代协议，一条「队头阻塞」暗线

<v-click>

- **演进史**：内核（请求-响应/无状态/统一语义）不变，外层为性能加码
- **1.1 瓶颈**：应用层队头阻塞 + 6 连接 + 头部冗余 → 逼出 hack
- **HTTP/2**：分帧多路复用灭应用层 HOL，但留 TCP 层 HOL；HPACK 压头抗 CRIME，推送已废弃
- **HTTP/3**：QUIC 灭 TCP 层 HOL，连接迁移 + 0-RTT；旧优化要拆掉

</v-click>

<v-click>

> 抓住「应用层 HOL（H2 解）vs TCP 层 HOL（H3 解）」这条线，三代取舍就在手里了。

</v-click>

-->
