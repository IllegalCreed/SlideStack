# 微前端一句话

martinfowler.com 的定义至今是最通用口径

<v-click>

> 一种把**可独立交付**的前端应用**组合**成一个更大整体的架构风格。

</v-click>

<v-click>

| 关键词 | 含义 | 不满足时的形态 |
| --- | --- | --- |
| **独立交付** | 每块有自己的仓库/构建/CI/CD，可单独发布到生产 | 拆了目录但同一次构建发布 = 模块化单体 |
| **组合** | 多块在运行时拼成一个产品，用户感知为整体 | 各自独立域名互跳 = 多个网站 |
| **架构风格** | 由属性与收益定义，不绑定实现技术 | 「用了 qiankun」不等于做对了微前端 |

</v-click>

<v-click>

> 定义没有点名任何技术：iframe、SSI、Module Federation 拼的都是微前端。

</v-click>

<!--
三个关键词各管一件事。最容易被忽略的是「架构风格」：它由属性与收益定义，不绑定技术。所以先用定义对齐目标——「我们要买的到底是独立部署，还是技术栈混跑」——选型讨论会短一半。术语 2016 年底进入 ThoughtWorks 技术雷达。
-->

# 浏览器里的微服务

single-spa 的工程化镜像：「存在于浏览器里的微服务」

<v-click>

| 微服务世界 | 微前端里的对应物 | 关键差异（别照搬） |
| --- | --- | --- |
| 一个服务 | 一个微应用 / 子应用 | 服务有进程边界，子应用共享一个页面 |
| API 网关 | 容器 / 主应用的路由分发 | 网关转发请求，容器要管挂载与卸载 |
| 服务注册与发现 | 注册表 / import map | 前端「发现」发生在浏览器里 |
| 进程隔离 | JS 沙箱 / iframe / Shadow DOM | 天然隔离 vs 人工模拟，强度差一个量级 |
| 服务间 RPC / 消息 | 自定义事件 / props / URL | 通信越多，拆分越可疑 |

</v-click>

<v-click>

> 浏览器里只有一个 DOM、一个 `window`、一张级联样式表 —— 隔离条件远比服务器苛刻，这正是各框架差异的来源。

</v-click>

<!--
微服务的组织学被原样搬进前端：独立仓库、独立 package.json、独立构建、独立 CI/CD。但右列是必须补上的刹车——浏览器只有一个 DOM、一个 window，隔离靠人工模拟而非进程边界，强度差一个量级。这正是核心机制叶要处理的难题。
-->

# 巨石应用之痛

微前端的全部动机，藏在它要替换的东西里

<v-click>

| 症状 | 根因 | 代价 |
| --- | --- | --- |
| 构建与 CI 全量跑 | 单一构建产物 | 改一行等二十分钟，反馈回路断裂 |
| **锁步发布** | 单一部署单元 | 发布火车 + code freeze，最慢团队定义节奏 |
| 框架升级 stop-the-world | 单一技术栈与版本 | 升级 = 全员停下业务，永远排不上期 |
| 代码腐化 | 没有物理边界 | 「谁都不敢动的模块」逐年增多 |
| 团队互相踩脚 | 多团队共用一库 | 合并冲突、公共代码所有权不清 |

</v-click>

<v-click>

> Geers 点出病根在组织：**按技术层横切**（前端组/后端组）时，任何特性都要跨团队排队交接。

</v-click>

<!--
前端巨石是企业级 Web 应用的普遍宿命。qiankun 文档的概括：单体随时间与人手增长，演变为难以维护的 Frontend Monolith。核心词是锁步发布——任何一块的变更都迫使整体重新构建、测试、发布，最慢的团队决定所有人的节奏。
-->

---

# 痛感可量化：三指标超线才上桌

Conway 定律：想拆系统，先看组织能不能拆

<v-click>

| 指标 | 健康线 | 巨石晚期典型值 |
| --- | --- | --- |
| 全量构建时长 | 分钟级以内 | 15 分钟起步，CI 队列常态拥堵 |
| 发布节奏 | 各团队按需随时发 | 固定发布火车 + code freeze |
| 框架升级排期 | 一个迭代内消化 | 以「季度」为单位、常年排不上 |

</v-click>

<v-click>

- 拆分粒度第一原则：**块数对齐团队数，而不是页面数**——Fowler 示例只拆三块
- 一个团队维护八个「微」前端，得到的不是自治，而是八倍的部署与集成开销

</v-click>

<v-click>

> 没有这些痛，微前端的每分收益都用不上，成本却照付 —— 反判据的种子。

</v-click>

<!--
三个指标超线，才值得把微前端摆上桌面。技术上的「拆」，本质是给组织上的「分」提供物理边界——组织拆不动的地方，技术强拆出来的边界会被日常协作重新踏平。粒度对齐团队边界，不对齐组织的拆分只产生通信成本。
-->

---

# Fowler 四收益：全部长在「独立」上

四收益逐条计价，不是打包赠送

<v-click>

| 收益 | 核心 | 验收问题 | 拿不到的典型 |
| --- | --- | --- | --- |
| **增量升级** | 绞杀者路径，各块择时换代 | 三年内有框架换代 / 遗留改造吗？ | 绿地新项目、栈已统一 |
| **解耦代码库** | bounded context 让坏决策变难 | 业务上下文边界画得出吗？ | 按页面机械切块 |
| **独立部署** | 每块自己的 CD 管道直达生产 | 每块能配一条直达管道吗？ | 发布仍需统一窗口审批 |
| **团队自治** | 按业务纵切、完整拥有一段 | 每块有长期负责的团队吗？ | 一团队维护全部「微」前端 |

</v-click>

<v-click>

> 四收益全部长在「独立」二字上 —— 任何重新引入共享运行时/状态/领域模型的做法，都在直接偿还收益。

</v-click>

<!--
立项时逐条验收，答不上验收问题的收益就从预期清单里划掉。增量升级是遗留改造的头号理由；独立部署让部署单元=风险单元=回滚单元；团队自治完全依赖组织配套——公司只有一个前端团队却拆八块，自治收益为零、成本全额照付。
-->

---

# Geers：端到端团队与五核心理念

跨职能团队从数据库到 UI，端到端负责一块业务

<v-click>

| 理念 | 要点 | 违反的代价 |
| --- | --- | --- |
| **技术无关** | 各队自选自升技术栈 | 组合层绑死某框架，升级变回 stop-the-world |
| **隔离团队代码** | 同框架也不共享运行时 | 共享单例 = 一处升级全员陪跑 |
| **团队前缀** | CSS/事件/存储/Cookie 命名空间化 | 冲突从运行时 bug 提前到 review 可见 |
| **原生特性优先** | 用浏览器事件而非自建 PubSub | 自建通信层 = 新耦合点与单点故障 |
| **韧性站点** | JS 失败仍可用、渐进增强 | 一个子应用挂掉就白屏整站 |

</v-click>

<v-click>

> 前缀是「隔离还做不到时的退路」—— 理念没过时，只是覆盖面随沙箱 / Shadow DOM 收缩。

</v-click>

<!--
Geers 把组织切法说得最直白：按业务使命纵切，取代按技术层横切。所有权在 DOM 里都看得见——自定义元素名带团队前缀。第二条最关键：就算所有团队用同一框架也不共享运行时，「反正都是 Vue 就共一个实例」正是最常见的偷懒式回耦。
-->

---

# qiankun 四核心价值：三家口径互印证

架构（Fowler）· 组织（Geers）· 工程（qiankun）

<v-click>

| qiankun 核心价值 | 表述 | 对应 Fowler / Geers |
| --- | --- | --- |
| **技术栈无关** | 主框架不限制接入应用技术栈 | Geers「技术无关」 |
| **独立开发独立部署** | 微应用仓库独立，部署后主框架自动同步 | Fowler「独立部署」 |
| **增量升级** | 微前端是渐进式重构的手段与策略 | Fowler「增量升级」 |
| **独立运行时** | 微应用之间状态隔离、运行时不共享 | Geers「隔离团队代码」 |

</v-click>

<v-click>

> 「为什么做微前端」是业界稳固共识，没有流派之争 —— 分歧全在「该不该用」与「怎么组合」。

</v-click>

<!--
qiankun 是蚂蚁基于 single-spa 的生产级实现，四核心价值是国内最常引用的口径。把三家并排看，架构、组织、工程三个视角的价值几乎一一对应。真正的分歧集中在后两个问题：该不该用、怎么组合代价多大。
-->

# 什么时候该用：三类正判据

Vercel 官方口径（它本身卖微前端托管）

<v-click>

| 判据 | 成立的信号 | 硬扛巨石的代价 |
| --- | --- | --- |
| **多团队独立协作** | ≥3 个特性团队常态并行、发布互相排队 | 发布火车、code freeze、合并地狱 |
| **开发与构建提速** | 全量构建 15 分钟起步、CI 队列拥堵 | 反馈回路断裂，工程师在等待中失去心流 |
| **渐进迁移** | 存量老系统跑核心业务、新特性想上新栈 | 全量重写 = 长期双轨 + 业务停摆，大概率烂尾 |

</v-click>

<v-click>

> 叠加 Fowler 的时间维度：**系统生命周期越长，「增量升级」这条收益越值钱** —— 五年期中后台和一个活动页，判断完全不同。

</v-click>

<!--
Vercel 文档给出的适用场景恰好三条。反面信号同样明确：单团队十人以下、单一产品、技术栈统一、没有遗留包袱——三条正判据一条不沾时上微前端，收益全部为零、三大 Downside 全额照付。
-->

# Fowler Downsides：三笔必付的账

全网被引用最多、也最诚实的成本清单

<v-click>

| Downside | 内容 | 对冲 / 缓解 |
| --- | --- | --- |
| **① payload 重复** | 各带一份 React，「逼用户下载 n 遍」 | 独立编译即隐式代码分割，单页可能反而更快 |
| **② 环境漂移** | 开发容器与生产容器行为不一致 | 频繁集成部署到接近生产的环境做测试 |
| **③ 运营治理复杂度** | 更多仓库 / 工具 / 管道 / 服务器 / 域名 | 需 IaC、模板化 CI、平台工程能力 |

</v-click>

<v-click>

> 成熟度判语：「你在选择**创造许多小东西而非一个大东西** —— 否则就是制造混乱。」性能结论以生产实测为准。

</v-click>

<!--
payload 争论在会议室里没有赢家：把依赖共享当治理问题，把性能结论交给生产 RUM 数据。环境漂移的解药是一条集成回归流水线，省掉它等于把集成测试外包给生产用户。治理复杂度把复杂度从「代码内」搬到「代码间」——后者靠平台工程解决。
-->

---

# Vercel 官方反判据：先考虑替代方案

卖微前端托管的平台，在文档里公开劝你先试别的

<v-click>

| 你的真实痛点 | 更便宜的药 | 何时才升级到微前端 |
| --- | --- | --- |
| 构建慢、HMR 慢 | 更快的编译器（Turbopack / Rspack）、构建缓存 | 提速后瓶颈仍是「人等人」 |
| 发布节奏互相拖累 | **feature flags** + 主干开发 | flag 失控，或连合进同一主干都做不到 |
| 想共享代码、统一规范 | **monorepo + Turborepo**：原子重构、增量构建 | 你要的其实是部署解耦而非代码共享 |

</v-click>

<v-click>

> 三条替代与微前端**并不互斥** —— 正确读法是「顺序」：先用便宜的药，无效再上贵的。

</v-click>

<!--
平台方劝退，恰恰说明复杂度真实、售后纠纷常见。Vercel 自己明确支持「monorepo 里跑微前端」。跳过这一步的团队，常在半年后发现用微前端解决了一个 feature flag 就能解决的问题，而沙箱、通信、依赖治理的成本已经沉没。
-->

# single-spa 性能反论：巨石未必更快

「微前端 = 更慢」是最常见的直觉反对

<v-click>

- **懒加载是内建的**：应用按路由注册，不活跃就不加载——巨石里要专门做代码分割
- **巨石会隐藏性能问题**：全站一个 bundle 无从追责；拆开后每块独立可测量、有「责任团队」

</v-click>

<v-click>

| 论者 | 结论 | 成立条件 |
| --- | --- | --- |
| Fowler | 独立构建可能让 payload 变重 | 不做依赖共享治理 |
| Fowler（对冲） | 单页加载可能反而更快 | 隐式代码分割生效 |
| single-spa | 常常比巨石快 | 懒加载 + **大库共享单例** |

</v-click>

<v-click>

> 性能是**治理变量**，不是命题常量 —— 既不否决、也不构成采用理由，最终裁判是生产实测。

</v-click>

<!--
single-spa 正面回击：微前端常常比它们脱胎的巨石更快。但反论有个不可省略的前提——大型库共享单一实例是被强烈建议的，不共享就落回 Fowler 的 payload 批评。依赖共享方案的选择与执行质量，决定你落在表里的哪一行。
-->

# 决策清单：评审会直接用

左列多数成立再谈选型，右列命中两条以上建议先撤

<v-click>

| 该用的信号 | 别用的信号 |
| --- | --- |
| ≥3 个特性团队常态并行，发布互相排队 | 单团队维护整个前端 |
| 存量遗留系统 + 渐进迁移诉求 | 绿地新项目、无历史包袱 |
| 生命周期以 5 年计，必经框架换代 | 活动页 / 短生命周期产品 |
| 技术栈诉求真实分裂（收购、多 BU） | 全员同栈且无分裂计划 |
| 有平台工程能力（IaC、模板化 CI、监控） | 一条 CI 都常年没人修 |
| 已试过 monorepo / flags 仍卡协作边界 | 还没试过任何替代方案 |

</v-click>

<v-click>

> 口诀：**微前端解决的是组织协作边界问题** —— 构建慢、发布节奏、代码共享各有更便宜的药。

</v-click>

<!--
这张表可直接搬进评审会。核心口诀记牢：痛点是构建慢就换更快编译器，是发布节奏就 feature flags，是代码共享就 monorepo——只有痛点是组织协作边界时，微前端才是对症药。
-->

# 组合模式三分法

碎片在哪里、什么时候拼成整体

<!--

# 三分法：按组合的时机与位置分

技术会过时，这个分类轴二十年有效

<v-click>

| 模式 | 组合时机 | 组合位置 | 独立部署 | 典型技术 |
| --- | --- | --- | --- | --- |
| 构建时组合 | `npm install` + 编译 | CI 产物 | **✗** | npm 包集成、submodule |
| 服务端组合 | HTTP 请求时 | Web 服务器 / 边缘 | ✓ | SSI/ESI、模板、平台路由 |
| 客户端运行时组合 | 页面运行时 | 浏览器 | ✓ | iframe、JS 渲染函数、WC、MF |

</v-click>

<v-click>

> 表里唯一的 ✗ 就是分水岭：真正的问题不是「用什么技术」，而是**独立部署这条及格线过不过得去**。

</v-click>

<!--
三分法真正的分水岭不是用什么技术，而是独立部署这条及格线过不过得去。注意 Module Federation 虽由打包器实现，但 remote 在运行时加载、各应用构建独立，归运行时组合而非构建时。
-->

# 构建时组合：Fowler 点名的反模式

微前端发成 npm 包，容器当依赖装、打成一个 bundle

<v-click>

```json
{
  "dependencies": {
    "@feed-me/browse": "^1.2.3",
    "@feed-me/order-food": "^4.5.6",
    "@feed-me/user-profile": "^7.8.9"
  }
}
```

</v-click>

<v-click>

- Fowler 批评：**改产品任何一部分，都必须重新编译发布每一个微前端** —— 锁步发布原样回归，「强烈建议不要采用」
- 症状自查：「升级公共库要开 30 个 MR」「上线要按依赖顺序发 5 个包」
- **例外**：只要代码复用、不要独立部署 → npm 包 + monorepo 是正解，那叫模块化单体
- **MF 不在此列**：remote 运行时加载、各应用构建独立 —— 属运行时组合

</v-click>

<!--
做法很自然，所以坑过的团队最多。你付出了微前端的全部治理成本，却收不到头号收益——风险单元没变小，发布队列没消失，只是包管理器多了几行。把 MF 误归为构建时集成，是 2026 年还很常见的过时印象。
-->

---

# 服务端组合：从 SSI 到边缘路由

Web 服务器按 URL 组装页面，插入各团队独立渲染的 HTML 片段

<v-click>

```html
<body>
  <h1>Feed me</h1>
  <!--# include file="$PAGE.html" -->
</body>
```

</v-click>

<v-click>

- **Geers 同构配方**：首屏走服务端 include（快、SEO 友好、韧性好），到浏览器后自定义元素接管交互
- **2026 形态上移边缘**：Vercel Microfrontends 按 path 路由到不同项目，`microfrontends.json` 一份声明；Cloudflare Web Fragments 同路
- 代价：跨微前端导航是**整页跳转**；**页面内多团队并存做不到** —— 页内组合仍要回客户端

</v-click>

<v-click>

> 组合粒度 = 页面/路径时，把拼接整个交给平台，应用层几乎零侵入——没有沙箱、没有运行时容器。

</v-click>

<!--
最老也最耐用的一路。Geers 的同构配方兼得首屏与动态性：服务端负责第一屏，客户端自定义元素接管后续交互。2026 年主流形态已上移到 CDN/边缘层。这是内容型/电商站点做微前端的默认答案；纯中后台通常不值得为它维护两条渲染通路。
-->

---

# 客户端运行时组合：三条路径

页内多团队并存、跨栈混跑的主场

<v-click>

| 路径 | 隔离 | 集成 | Fowler / 生态定位 |
| --- | --- | --- | --- |
| **iframe** | 最强（原生） | 最僵（路由/深链/响应式都难） | 隔离好但体验割裂；wujie 底层 |
| **JS 渲染函数** | 靠沙箱补 | **最灵活** | Fowler 的默认选择、野外最常见 |
| **Web Components** | Shadow DOM 中等 | 较灵活 | 标签名/属性/事件即团队间契约 |

</v-click>

<v-click>

- **框架化演进都长在 JS 渲染函数上**：single-spa 规范成 `bootstrap/mount/unmount`，qiankun 的 **HTML Entry** 让子应用直接交出整个 HTML
- micro-app 沿 Web Components 路做组件化接入；WC 的标签名/属性/事件就是给其他团队的契约

</v-click>

<!--
JS 渲染函数是 Fowler 的默认方案：每个微前端在全局暴露一个渲染函数，容器按路由调用并传入容器节点。这条路径的灵活性加上独立部署能力使它成为野外最常见的选择。后续框架化演进都长在它上面。
-->

# 三分法取舍表

围绕两根轴：隔离 vs 融合、首屏 vs 无缝

<v-click>

| 维度 | 构建时 | SSI/模板 | 平台路由 | iframe | JS 函数 | Web Comp |
| --- | --- | --- | --- | --- | --- | --- |
| 独立部署 | **✗** | ✓ | ✓ | ✓ | ✓ | ✓ |
| 隔离强度 | 无 | 页面级 | 页面级 | **最强** | 靠沙箱 | Shadow DOM |
| 首屏/SEO | 好 | **最好** | 好 | 差 | 一般 | 一般 |
| 页内多应用 | —— | ✓ 片段 | ✗ | ✓ 笨重 | ✓ | ✓ |
| 典型代表 | 反模式 | 电商门户 | Vercel MFE | wujie 底层 | single-spa | micro-app |

</v-click>

<v-click>

- 路径粒度能满足 → 服务端/平台路由，复杂度最低，甚至不需要「微前端框架」
- 页内多团队并存、跨栈混跑 → 客户端运行时，再在 iframe（隔离）与 JS/WC（融合）间按信任度取舍
- **验收铁律**：凡是要「重新构建整体」才能上线的组合，都退化为模块化单体

</v-click>

<!--
两根取舍主轴：隔离强度与集成灵活度成反比，首屏/SEO 与交互无缝成反比。无论选哪路，都把「能否独立部署」当验收标准回测一遍——过不了，你搭的是模块化单体。
-->

碎片拼起来之后，下一个问题是 URL 归谁管、容器该多厚。容器是整个系统里唯一常驻的一方，它的厚度是微前端架构最诚实的健康指标。

# 容器：整个系统唯一常驻的一方

container / shell / root config / 主应用·基座——同一角色的不同称谓

<v-click>

| 职责 | 内容 | 归容器的理由 |
| --- | --- | --- |
| **路由分发** | 维护 URL ↔ 微前端活性映射 | 一个地址栏只能有一个裁判 |
| **公共 chrome** | header/nav/footer、主题、Layout | 「整体性」由常驻方保证 |
| **鉴权与会话** | 登录流程、token 获取与注入 | 用户只该登录一次（Fowler） |
| **生命周期编排** | 加载/挂载/卸载、失败降级 UI | 一个子应用挂掉不能拖垮整站 |
| **依赖底座** | import maps / 共享库版本裁决 | 全局只能有一个版本仲裁点 |

</v-click>

<v-click>

> 这份清单**只有平台性职责、没有业务** —— 这不是巧合，是「最薄哲学」的伏笔。

</v-click>

<!--
各家生态对同一角色叫法不同：Fowler 的 container、single-spa 的 root config、国内的主应用/基座、MF 的 host。职责恰好五件，全是平台性的。这份「没有业务」的清单，正是最后一节最薄哲学的伏笔。
-->

# 路由分发：两种心智

URL 决定谁活着，还是挂载点决定谁加载

<v-click>

```js
registerApplication({
  name: 'orders',
  app: () => import('https://cdn.example.com/orders/entry.js'),
  activeWhen: (location) => location.pathname.startsWith('/orders'),
});
```

</v-click>

<v-click>

| 对比 | 声明式活性映射 | 手动挂载（组件式） |
| --- | --- | --- |
| 事实源 | 中心注册表 + URL | 宿主组件树 |
| 原语 | `activity function` `(loc)=>boolean` | `<micro-app>` 标签 / 组件 / parcel |
| 典型 | single-spa、qiankun（activeRule） | micro-app、wujie |

</v-click>

<v-click>

> 可混用，但**同一块业务只能有一个路由裁判** —— 双裁判是挂载竞态与循环跳转的高发源。

</v-click>

<!--
声明式活性映射让 URL 成为唯一事实源：activity function 是纯函数，框架在每次导航事件统一调用、对比前后真值、驱动 mount/unmount。手动挂载把微前端当组件用，路由裁判是宿主自己的 router。排这类 bug 先画「谁在决定这块 DOM 的生死」。
-->

# 应用间跳转与深链：URL 是第一通信总线

Fowler 把路由跳转列为跨应用通信的推荐手段

<v-click>

| 跳转手段 | 体验 | 代价 |
| --- | --- | --- |
| 容器统一导航 API / 共享 history | SPA 级无刷新 | 各应用对容器产生 API 依赖 |
| 直接改 URL（`pushState` 约定） | SPA 级无刷新 | 约定散落，须文档化 base 前缀 |
| 原生 `<a href>` 整页跳转 | 有白屏，但**最解耦** | 状态全丢，靠 URL 与存储重建 |

</v-click>

<v-click>

- **deep link 铁律**：每个业务页面 URL 必须支持**直接刷新进入** —— 验收方式：把全站路由贴进无痕窗口
- 给每个子应用分配**互不重叠的 base 前缀**（`/orders/*`、`/profile/*`），越界即抢别人的路由

</v-click>

<!--
地址栏是全局状态里最稳、耦合最低的一份。能用路由表达的状态迁移，就不要再发明自定义消息。deep link 铁律的验收方式简单粗暴：把全站每个路由贴进新开的无痕窗口，冷启动能凭 URL 独立走通才算过。
-->

# 横切职责：鉴权与公共 chrome

Fowler 写得最斩钉截铁的一段：用户只应认证一次

<v-click>

```text
容器登录、持有 token
      │ 注入（如 customProps）
      ▼
微应用 A ─带 token─► 自家后端 / BFF
微应用 B ─带 token─► 自家后端 / BFF
```

</v-click>

<v-click>

- **鉴权坚定归容器**：认证是横切关注点，容器取 token、初始化注入，微应用带 token 调自家 API
- **公共 chrome** 默认归容器；改得勤的导航可拆成独立「导航微前端」——**谁改得勤，谁就该独立部署**
- **错误兜底**：子应用加载失败时容器给降级 UI —— Geers「韧性站点」在容器层的落点

</v-click>

<!--
鉴权是横切关注点，坚定归容器所有：容器登录持有 token，挂载时注入各微应用，微应用带 token 请求自家后端。错误兜底是容易漏掉的第四件事——加载函数返回的 Promise 被 reject 时渲染什么，应当是容器的显式设计而不是空白。
-->

# 最薄哲学：容器越薄，自治越真

single-spa 官方定性只有一句，分量很重

<v-click>

> root config 的存在只为启动各应用（exists only to start up the applications）。

</v-click>

<v-click>

- 容器是**全体团队的公共依赖，唯一无法独立于他人发布的东西**：
- 容器发版 → 所有微应用被动回归，代码越多全员陪跑越频繁
- 业务逻辑进容器 → 「改我的功能却要等平台排期」，自治名存实亡
- 共享工具函数会自然膨胀成事实框架 —— 新巨石的胚胎

</v-click>

<v-click>

> 变胖症状：子应用上线要等容器发版 / 容器里长出业务组件 / `common/utils` 逐月增长。

</v-click>

<!--
为什么执拗地薄？因为容器是全体团队的公共依赖，是整个架构里唯一无法独立于他人发布的东西。纪律很简单：业务逻辑一律下放子应用，公共 UI 抽成带版本的共享库或独立导航微前端。root config 常年几十行说明边界立住了；逼近万行说明你在用微前端的成本运营一个分布式巨石。
-->

# vs iframe：老对手的经典结论

Fowler 的双面评价至今是标准答案

<v-click>

| 维度 | 裸 iframe | 微前端框架（qiankun 系） |
| --- | --- | --- |
| JS/样式隔离 | 浏览器原生，**最强** | 沙箱/作用域样式模拟，有逃逸面 |
| URL 与深链 | 内外两套 history，要自行同步 | 容器统一路由 |
| 布局与弹窗 | 遮罩/下拉难跨边界，响应式难 | 同一文档流，天然自由 |
| 通信 | `postMessage` 序列化边界 | props/事件/共享内存均可 |
| 加载成本 | 每帧一套完整浏览上下文 | 可共享依赖与运行时 |

</v-click>

<v-click>

> 一句话：**iframe 用隔离换融合** —— 对信任度低、融合要求低的场景（第三方页、遗留系统「先挂上」），它依旧是性价比之王。

</v-click>

<!--
iframe 是最早也最省事的运行时组合。三种场景下裸 iframe 依旧是正解：低信任第三方内容、合规硬隔离、一次性挂载遗留系统。判断句式：融合要求越低、信任度越低，裸 iframe 越占优。
-->

---

# vs iframe：wujie 把它拆开用

体验割裂的根源不是 iframe 本身，而是「把渲染也关进了 iframe」

<v-click>

```text
wujie 路线：执行与渲染分离
  JS   ──► 放进 iframe 执行（浏览器原生全局隔离，不必自研 Proxy 沙箱）
  DOM  ──► 渲染到主文档的 Web Component 容器（布局/弹窗/响应式回同一文档流）
```

</v-click>

<v-click>

- 隔离与融合的经典互斥，被「执行与渲染分离」拆掉一角；**不改写脚本，ESM/Vite 天然兼容**
- 2026-06 v2.0 全新 iframe 沙箱完成复活（一个月连发 4 版）
- **「iframe = 落后」在 2026 作废**：正确问法是「iframe 用在哪一层」——渲染容器 / JS 沙箱 / 安全边界

</v-click>

<!--
腾讯 wujie 把 iframe 拆开用：JS 放进 iframe 拿原生隔离，DOM 渲染到主文档 Web Component 容器。隔离与融合的经典互斥被执行与渲染分离拆掉一角。2026 年的正确问法不再是「用不用 iframe」，而是「iframe 用在哪一层」。
-->

---

# vs monorepo：正交的两个维度

最常见的混淆：「上微前端 = 拆仓库」。Vercel 把这件事说死了

<v-click>

|  | 单体部署 | 微前端（独立部署） |
| --- | --- | --- |
| **monorepo** | 最常见默认形态 | 2026 主流之一：同仓多项目，共享配置 |
| **polyrepo** | 传统多站点 | Fowler 默认想象：每团队一仓一管道 |

</v-click>

<v-click>

- **仓库形态（mono/poly）与部署单元（单体/微）正交** —— Vercel：mono/polyrepo 下微前端行为完全一致
- monorepo 解决**代码共享与原子重构**；微前端解决**部署与运行时解耦**
- 只要前者 → monorepo 就够（官方反判据）；两者也常叠用（monorepo + 微前端是 2026 常见形态）

</v-click>

<!--
仓库形态与部署单元是两个正交维度，四个象限都有名有姓。决定上微前端后，仓库形态按团队边界与权限敏感度选：跨 BU、权限硬隔离选 polyrepo，其余默认 monorepo——既拿原子重构，又不牺牲独立部署。
-->

---

# vs BFF：纵切一路切到后端

纵切不该停在浏览器——理想团队从视觉一路拥有到 API、数据库

<v-click>

```text
┌── 团队 A ─────────┐   ┌── 团队 B ─────────┐
│ 微前端 A          │   │ 微前端 B          │
│   ↓              │   │   ↓              │
│ BFF A（仅服务 A）  │   │ BFF B（仅服务 B）  │
└───┬──────────────┘   └───┬──────────────┘
    └──► 下游共享微服务 / 领域服务 ◄──┘
```

</v-click>

<v-click>

- BFF =「唯一目的是服务这一个前端」；判断原则：**别让团队等其他团队给它做东西**
- 不动点：**登录鉴权不下放** —— token 由容器注入，BFF 只校验与消费
- 代价：服务数量翻倍，计入 Downside③ 治理复杂度一起核算

</v-click>

<!--
微前端把团队按业务纵切到 UI 层，Fowler 追问数据从哪来——纵切不应停在浏览器。判断要不要配 BFF 的原则很朴素：如果每次新需求都要跨团队提接口排期，那就是需要 BFF 的信号。边界上有个不动点：登录鉴权不下放。
-->

---

# vs 模块化单体与共享组件库

一切「这算不算微前端」的争论，到一条公式为止

<v-click>

- **模块化单体**：构建时组合的归宿——有边界、有复用，是**好架构**，但不满足独立部署，**不是微前端**
- 用错标签的代价：按微前端立项（多管道治理）、按单体交付（锁步发布），两头亏一起吃
- **共享组件库与微前端正交**：design system 照建，但只放 **UI 逻辑**——领域逻辑进共享库会制造高度耦合

</v-click>

<v-click>

> **微前端 = 独立部署 ∧ 运行时组合。** 只有代码拆分 → 模块化单体；只有运行时加载 → 懒加载；两条同时成立才是微前端。

</v-click>

<!--

# 判别速查表

六个近邻，一栏看清与微前端的关系

<v-click>

| 方案 | 解决什么 | 与微前端的关系 |
| --- | --- | --- |
| iframe | 页面级强隔离嵌入 | 组合手段之一；wujie 重用为 JS 沙箱 |
| monorepo | 代码共享、原子重构 | 正交，常叠用；也是官方前置替代方案 |
| BFF | 前端专属后端、纵切闭环 | 服务端的自然延伸；鉴权仍归容器 |
| 模块化单体 | 代码边界与复用 | 构建时组合的归宿；好架构但非微前端 |
| 组件库 / design system | 视觉与交互一致性 | 正交；只放 UI 逻辑，领域逻辑禁入 |
| 懒加载 / 代码分割 | 首屏性能 | 只有运行时加载、无独立部署，不是微前端 |

</v-click>

# 格局总览：三主流 + Module Federation

从「qiankun 一超」到多强并立；**Vite/ESM 兼容是第一分水岭**

<v-click>

| 方案 | 出品 | 沙箱路线 | Vite/ESM | 状态 |
| --- | --- | --- | --- | --- |
| **qiankun** | 蚂蚁 | with + Proxy 模拟 | 弱，需插件绕行 | 存量王者，恢复活动 |
| **wujie** | 腾讯 | iframe 原生隔离 | 好 | 复活，上升 |
| **micro-app** | 京东 | with / iframe 双沙箱 | 较好 | 活跃，常青 RC |
| **single-spa** | 社区 | 无沙箱，纯编排 | 好（import maps） | 稳定，演进停滞 |
| **Module Federation** | 字节 | 无沙箱，模块共享 | 官方 @mf/vite | **事实主线** |

</v-click>

<v-click>

> 沙箱收敛为两派：**Proxy/with 模拟**（拦不住 ESM）与 **iframe 原生隔离**（天然兼容 ESM）。

</v-click>

<!--
两个横向观察：Vite/ESM 是第一分水岭，把候选表切成两半，这正是 qiankun 焦虑与 wujie 复活共同的背景板。沙箱路线两派收敛——模拟派兼容老产物但拦不住 ESM，浏览器派天然兼容 ESM 但要解决通信与渲染桥接。

# 逐家点评：国内三主流

版本核于 2026-07

<v-click>

- **qiankun（存量最大，换代最难）**：稳定版停在 v2.10.16（2023-11），**3.0 三年难产**（rc.21）；2026 恢复活动。根因：JS 沙箱靠 `with + Proxy` 改写脚本，**原生 ESM 不经这层** → 「Vite 之痛」

- **wujie（iframe 沙箱路线复活）**：2026-06 v2.0 一个月连发 4 版；不改写脚本 → ESM/Vite 天然兼容，渲染走主文档不割裂。风险：新沙箱受生产检验时间还短

- **micro-app（组件化接入，常青 RC）**：接入最轻（`<micro-app>` 标签），月度活跃，但 1.0 长期停在 RC（rc.32）——按「实质稳定、名义 RC」评估并锁小版本

</v-click>

<!--
webpack 存量 + 已有 qiankun 资产继续用完全合理，2.x 被大规模生产验证过；全新 Vite 项目起手选 qiankun 是逆着生态阻力走。wujie 是 Vite 栈 + 强隔离的首选评估对象，但刚复活要自己压测内存回收。micro-app 心智成本最低，适合宿主是业务应用顺手嵌几个子应用。
-->

---

# 逐家点评：single-spa 与 Module Federation

一个停滞的鼻祖，一条运行时化的主线

<v-click>

- **single-spa（鼻祖的停滞与遗产）**：定义了生命周期协议 / activity function / root config，qiankun 建其上。**v6 稳定、v7 卡 beta**（2025-09 后停）；官方推荐架构已是**原生 ESM + import maps**；无沙箱 → 隔离全靠团队纪律

- **Module Federation（事实主线）**：2.0 从「webpack 编译期特性」变为**独立于打包器的运行时体系**，字节 + Zack Jackson 维护，core 高频发版。解决**模块级共享**、不带沙箱 → 与应用级容器常是**叠用**而非竞品

</v-click>

<v-click>

> 生态换代细节：早年 originjs 的 Vite 插件已停滞，**官方 @module-federation/vite 是现行方案** —— 搜到旧教程先看日期。

</v-click>

<!--
single-spa 停滞不等于死亡——它要做的事本来就少，v6 + 原生 ESM 今天依然能打，但要接受不会有大演进。MF 解决模块级共享，不带沙箱，与 qiankun 类应用级容器不是同类竞品，更常是叠用：容器管编排隔离，MF 管依赖共享。同栈同大版本的多团队，直接用 MF 是链路最短的微前端。
-->

# 选型决策树

决策树只做第一轮收敛，终选靠双候选 POC

<v-click>

```text
Q0 要不要？判据不满足 → monorepo + feature flags，到此为止
Q1 同栈同大版本、只要模块共享 → Module Federation 2.0
Q2 要应用级容器 → 看构建器：
     webpack 存量 → qiankun 2.x（资产复用）/ micro-app
     Vite/ESM   → 强隔离选 wujie；轻接入选 micro-app（iframe 模式）
     纯编排无沙箱 → single-spa v6 + 原生 ESM + import maps
Q3 页面粒度 + 接受平台绑定 → Vercel Microfrontends
终选：双候选 POC（沙箱逃逸 / 样式冲突 / 构建兼容 / 内存回收）
```

</v-click>

<v-click>

> 选型三问：先**「要不要」**，再**构建器与产物形态**（Vite/ESM 分水岭），最后才是框架特性对比。

</v-click>

<!--

# 小结

一条判别公式，六段架构决策

<v-click>

- **是什么**：可独立交付 + 运行时组合；四收益全长在「独立」上，Geers 补五理念
- **该不该用**：多团队 / 渐进迁移 / 长生命周期才付账；先试 monorepo + feature flags
- **怎么组合**：构建时（反模式）/ 服务端（首屏最强）/ 客户端运行时（页内共存）
- **容器**：唯一常驻方，只管路由/chrome/鉴权/编排/依赖；**root config 越薄越好**
- **划边界**：vs iframe（wujie 重用为沙箱）/ monorepo（正交）/ BFF / 模块化单体
- **2026 选谁**：三主流 + MF 事实主线；Vite/ESM 分水岭；双候选 POC 定生死

</v-click>

<v-click>

> **微前端 = 独立部署 ∧ 运行时组合** —— 解决的是组织协作边界问题，不是技术炫技。

</v-click>
